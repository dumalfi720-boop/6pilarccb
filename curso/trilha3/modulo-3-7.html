MODULE 3.7

👾 Create Pull Request

Complete workflow to create professional Pull Requests using Claude Code's /pr command.

1

📦 Prepare Commits

Before PR, ensure yourcommits are clean and organized. A PR with messy commits makes review difficult.

💡Main Concept

Prepare the branch before creating the PR:

  • •"Verifique se tem mudancas nao commitadas e faca commit de tudo"
  • •"Rode os testes uma ultima vez antes do PR"
  • •Make sure the branch is updated:"Faca rebase com main se necessario"

⚡Practical Tip

Pre-PR checklist:"Antes do PR: 1) rode os testes 2) rode o linter 3) verifique se nao tem console.log perdido 4) faca /review"

2

⚡ The /pr Command

The command/prcreates a complete Pull Request on GitHub with automatically generated title, description and labels.

💡Main Concept

How /pr works:

  • •Enter/prat Claude Code
  • •Claude analyzes all commits in the branch, understands the changes and creates the PR
  • •He usesgh pr createwith generated title and description

⚡Practical Tip

You need to have thegh(GitHub CLI) installed and authenticated. Rotategh auth statusto check.

3

📝 What Claude Generates

Claude generates afull description of the PRincluding summary, changes, testing and notes for reviewers.

💡Main Concept

Typical structure of the generated PR:

  • •Title:Concise and descriptive following Conventional Commits
  • •Summary:What was done and why
  • •Changes:Detailed list of changes per file/module
  • •Test plan:How to test the changes

⚡Practical Tip

Customize the template in CLAUDE.md:"Ao criar PRs, sempre inclua: screenshots se tiver UI, link do ticket Jira, e breaking changes"

4

👁️ Review Description

Alwaysreview the descriptionbefore confirming. Claude may miss details or omit important context that only you know.

💡Main Concept

What to check in the generated description:

  • •Does the title reflect the main change? Is it clear to those who don't have context?
  • •Is the "why" explained? (motivation, ticket, bug report)
  • •Do you have sensitive information that should not be in the PR?

⚡Practical Tip

Ask for adjustments:"Adicione o link do ticket JIRA-1234 e mencione que isso e um hotfix para producao"

5

🔧 Adjust and Customize

Ask Claude toadjust the PRaccording to your team's conventions - labels, reviewers, milestones, etc.

💡Main Concept

Common customizations for PRs:

  • •"Adicione os labels 'bug' e 'high-priority' ao PR"
  • •"Assigne @joao e @maria como reviewers"
  • •"Crie como draft PR para review interno primeiro"

⚡Practical Tip

Automate in CLAUDE.md:"Ao criar PRs para bugfix, sempre adicione label 'bug' e assigne ao tech lead como reviewer"

6

📚 Good PR Practices

Follow thesegood practicesto create PRs that are easy to quickly review and approve.

💡Main Concept

Excellent PRs follow these rules:

  • •Small PRs (less than 400 lines) are reviewed 3x faster
  • •Each PR must do one thing - don't mix feature + refactor + bugfix
  • •Include screenshots for UI changes
  • •Use the workflow:/review→ fix →/commit → /pr

⚡Practical Tip

For large features, use stacked PRs:"Crie 3 PRs separados: 1) modelo + migration, 2) API endpoints, 3) frontend"
