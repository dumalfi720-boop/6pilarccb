MODULE 3.3

👾 Refactor Code

Secure workflow to refactor existing code without breaking functionality, using Claude as a refactoring partner.

1

🏗️ Understand Architecture

Before refactoring, Claude needs tounderstand the current architectureso as not to break dependencies or change behaviors.

💡Main Concept

Ask Claude to map the code before changing anything:

  • •"Analise src/services/payment.js. Quais funcoes ele exporta e quem as importa?"
  • •"Mapeie todas as dependencias do modulo de autenticacao"
  • •Claude uses grep and file reading to create a mind map of the code

⚡Practical Tip

Get a diagnosis first:"Antes de refatorar, identifique os code smells e problemas de arquitetura neste modulo"

2

🎯 Set the Goal

Clearly definewhat do you want to improveand what restrictions exist. Refactoring without a clear objective becomes unnecessary rewriting.

💡Main Concept

Be specific in the objective of the refactoring:

  • •"Extraia a logica de validacao para um modulo separado mantendo a mesma API publica"
  • •"Converta os callbacks para async/await sem mudar o comportamento externo"
  • •Specify restrictions: "Do not change the public interface" or "Maintain compatibility with v2"

⚡Practical Tip

Use Plan Mode:"[Plan Mode] Quero refatorar o PaymentService. Proponha um plano que melhore testabilidade sem mudar a API"

3

👁️ Review Diff

After Claude made the changes,review each diff carefully. In refactoring, a wrong detail can cause subtle bugs.

💡Main Concept

Claude shows diffs before applying. Pay attention to:

  • •Unintentional changes in behavior (e.g. changed execution order)
  • •Removed imports that are still needed elsewhere
  • •Part:"Me mostre o git diff completo antes de eu aceitar"

⚡Practical Tip

If the diff is large, ask for a summary:"Resuma as mudancas que voce fez e quais arquivos foram afetados"

4

🧪 Run Tests

Tests are therefactoring safety net. Run all tests after each change to ensure nothing broke.

💡Main Concept

Tests before and after refactoring:

  • •"Rode todos os testes e me mostre quais passam e quais falham"
  • •If you don't have tests, ask:"Escreva testes para o comportamento atual antes de refatorar"
  • •After refactoring:"Rode os testes novamente. Os mesmos testes devem continuar passando"

⚡Practical Tip

Rule of thumb:"Sem testes = sem refatoracao. Primeiro escreva testes, depois refatore". This is valid for both humans and AI.

5

📦 Incremental Commit

Doincremental commitsduring refactoring to have safe return points in case something goes wrong.

💡Main Concept

Small commits make it easier to revert if necessary:

  • •/commitafter each stage of refactoring
  • •Ex:refactor: extract validation logic to separate module
  • •If something breaks, it's easy to fixgit revertfrom the last commit

⚡Practical Tip

Ask Claude:"Faca commit apos cada etapa da refatoracao com mensagens descritivas". So you have a clean record.

6

📚 Good Practices

Follow thesegood refactoring practicesto maximize quality and minimize risks.

💡Main Concept

Principles for efficient refactoring with Claude Code:

  • •Never refactor and add features at the same time - they are separate commits
  • •Use"Refatore para melhorar X, mas mantenha o comportamento identico"
  • •Ask for code review after refactoring:/reviewto validate the changes

⚡Practical Tip

For large refactorings, use the "strangler fig" strategy:"Crie o novo modulo em paralelo, migre os callers um por um, e depois remova o antigo"
