🐛 Fix a Bug

The complete workflow for finding, diagnosing, and fixing bugs using Claude Code. From initial context to final commit.

What is it:

The first step of the bug fix workflow: give Claude Code all the necessary context. Include the error message, expected vs current behavior, and relevant files.

Why learn:

Without adequate context, Claude will come up with generic solutions. With rich context - stack trace, logs, file where the bug occurs - it investigates surgically and gets it right the first time.

Key concepts:

Paste the complete error (stack trace, message, status code). Describe: “I expected X, I received Y.” Mention relevant files. Example: "The /api/users endpoint returns 500. Stack trace: [paste here]. The handler is in src/routes/users.ts."

What is it:

Explicitly ask Claude to investigate the root cause before proposing any solutions. Use phrases like "investigate first, don't fix yet" or "explain to me the root cause before changing code".

Why learn:

Claude's instinct is to resolve it quickly. But complex bugs need correct diagnosis first. Forcing investigation prevents him from applying band-aids that mask the real problem.

Key concepts:

Prompt: "Investigate this bug without applying any fixes. Read the relevant files, trace the data flow, and tell me what the root cause is." Claude will read code, follow imports and map the problem.

What is it:

Evaluate the analysis that Claude presented before proceeding with the correction. Confirm that the identified root cause makes sense and that he understands the flow correctly.

Why learn:

Claude could make a mistake in the diagnosis. Reviewing before applying the fix prevents you from introducing new regressions. It is the security checkpoint between investigation and action.

Key concepts:

Ask: "Are you sure the cause is X and not Y?" Validate that it read the right files. If the diagnosis is not convincing, ask to investigate further before continuing.

What is it:

Now, ask Claude to apply the correction based on the validated diagnosis. It will edit the necessary files and make the minimum changes to resolve the issue.

Why learn:

After investigation and validation, the correction tends to be surgical. You avoid unnecessary changes and keep the diff small, facilitating code review and reducing the risk of regression.

Key concepts:

Prompt: "Now apply the fix. Make the minimum change necessary." Review the diff with git diff. Run the tests. If something doesn't work, go back to the investigation step.

What is it:

Use Claude to generate the correction commit with a descriptive message. /commit analyzes the diff and creates a message that explains what was fixed and why.

Why learn:

A good bug fix commit documents the problem and the solution. When someone does git blame months later, the commit message explains the full context of the fix.

Key concepts:

Use /commit or ask: "commit this fix with descriptive message". Claude generates something like "fix: resolves null pointer in user auth by adding null check before token validation".

What is it:

Advanced techniques for bug fixing with Claude: use git bisect with Claude, request regression tests along with the fix, and patterns for recurring bugs.

Why learn:

Experienced operators don't just fix bugs - they prevent recurrences. Requesting regression testing along with the fix guarantees that the same bug never returns.

Key concepts:

Always ask: "in addition to the fix, write a test that fails without the fix and passes with it". Use git bisect + Claude to track when the bug was introduced. Document recurring bug patterns in CLAUDE.md.

🏗️ Build New Feature

The complete workflow to plan, implement and deliver new functionality from scratch using plan mode and incremental execution.

What is it:

Start developing a feature using Shift+Tab to activate plan mode. Claude analyzes the codebase and creates a detailed plan without writing any code yet.

Why learn:

Features built without planning result in disorganized code and constant refactorings. Plan mode forces an architecture thought out before implementation.

Key concepts:

Activate plan mode with Shift+Tab. Describe the feature: requirements, expected behavior, restrictions. Claude will map affected files, required interfaces and implementation order.

What is it:

Critically review the plan generated by Claude before implementing. Check if it considered edge cases, if the architecture makes sense and if no components are missing.

Why learn:

The plan is your last chance to make cheap course corrections. Changing a plan costs zero. Changing implemented code costs hours. Review now, save later.

Key concepts:

Ask: "What if the user does X?", "Have you considered authentication/validation?", "What is the impact on performance?". Adjust the plan until you are satisfied before proceeding.

What is it:

Execute the plan step by step, asking Claude to implement one component at a time. Don't try to implement everything at once - go incrementally.

Why learn:

Incremental implementation allows you to validate each part before moving on. If something goes wrong, you know exactly where the problem is. The diff is small and easy to review.

Key concepts:

Prompt: "Implement plan step 1: create the data model." Review. "Now step 2: create the endpoint." Review. Each step is a validated checkpoint before the next.

What is it:

Ask Claude to run all existing tests and write new tests for the implemented feature. Check that nothing is broken and that the feature works as expected.

Why learn:

Feature no test and time bomb. Claude can write unit tests, integration tests and edge cases in seconds. There is no excuse for delivering code without coverage.

Key concepts:

Prompt: "Run existing tests and write tests for the new feature covering happy path and edge cases." Claude runs npm test / pytest, analyzes failures and creates complementary tests.

What is it:

Ask Claude to review all feature code, clean TODOs, improve variable names, add comments where necessary and ensure it follows project standards.

Why learn:

Functional code is not enough. Clean, documented code that is consistent with the rest of the project is what differentiates professional delivery from a prototype. The polish and the finish.

Key concepts:

Use /review for automatic review. Ask: "review all new code, check naming, documentation, unhandled edge cases and compliance with project standards."

What is it:

Use /commit to generate atomic and descriptive commits for the feature. If the feature is large, divide it into logical commits: model, endpoint, tests, etc.

Why learn:

Atomic commits facilitate code review, rollback and historical understanding. A giant commit with the entire feature is much more difficult to review and partially revert.

Key concepts:

For large features, make incremental commits: "feat: add user model", "feat: add user API endpoint", "test: add user endpoint tests". Use /commit at each step or ask Claude to organize the commits.

♻️ Refactor Code

The workflow to improve the existing code structure without changing behavior. Secure refactoring with Claude as a programming peer.

What is it:

Before refactoring, ask Claude to explain the current architecture of the code that will be modified. Understand dependencies, couplings and reasons why the code is the way it is.

Why learn:

Refactoring without understanding the current state is a recipe for disaster. Code can have complexities for good reasons (edge ​​cases, performance, compatibility). Understand before changing.

Key concepts:

Prompt: "Analyze the current architecture of [module]. Explain the dependencies, data flow, and any patterns used. Identify code smells and technical debt." Claude maps everything out before playing.

What is it:

Establish a clear goal for the refactoring: extract function, rename, simplify logic, remove duplication, or improve testability. Refactoring without purpose becomes free change.

Why learn:

Clear objectives keep scope controlled. "Refactor everything" is impossible to revise. "Extract the validation logic into a separate module" is concrete and verifiable.

Key concepts:

Be specific: "Refactor to extract the parsing logic from file X into a separate module, keeping the same public interface." This gives Claude a closed scope and measurable objective.

What is it:

After Claude applies the refactoring, review the diff line by line. Check that the behavior was preserved and that only structural changes were made.

Why learn:

Refactoring by definition does not change behavior. If the diff shows new logic or changed conditions, something has gone out of scope. The diff review is your security guardian.

Key concepts:

Run git diff after refactoring. Ask Claude: "review this diff and confirm that no behavior has changed." Any semantic and red flag changes must be reversed or justified.

What is it:

Run the entire test suite after refactoring. If all tests pass, you have high confidence that the behavior has been preserved. If one fails, the refactoring introduced regression.

Why learn:

Tests are the safety net of refactoring. Without passing tests, you have no way to guarantee that the refactoring was safe. This step is non-negotiable.

Key concepts:

Ask: "run all the tests and tell me if any failed." If a test fails, it's regression: roll back or fix it. If there are no tests, ask Claude to write tests BEFORE refactoring.

What is it:

Commit the refactoring with a clear message indicating that it is a structural change, without changing behavior. Convention: "refactor: [change description]".

Why learn:

Refactoring commits must be separated from feature commits. This allows you to revert refactorings independently and makes it easier to understand the project history.

Key concepts:

Use /commit. Claude generates something like "refactor: extract validation logic into separate module". Never mix refactoring with features in the same commit - they are changes with different purposes.

What is it:

Guiding principles for effective refactoring: the boy scout rule (leave the code better than you found it), opportunistic refactoring, and when NOT to refactor.

Why learn:

Not every refactoring is worth it. Code that works, has tests and no one touches does not need to be refactored. Knowing when to stop is just as important as knowing when to start.

Key concepts:

Refactor when: you are going to change the code in any way, duplicate code causes bugs, or testability is compromised. DO NOT refactor: stable legacy code that no one touches, or before a tight deadline.

🗺️ Understand a New Codebase

The workflow for navigating and understanding an unfamiliar project using Claude as a guide. From overview to specific details.

What is it:

Ask Claude to analyze the general structure of the project: directories, technologies used, architecture, entry points and main dependencies. The high level map.

Why learn:

Without an overview, you will get lost in the details. Start at the top: understand what the project does, how it is organized and what technologies it uses. Then zoom in on the details that matter.

Key concepts:

Prompt: "Analyze this project. Give me an overview: what is its purpose, what technologies does it use, how is it organized, what are the main entry points and key dependencies." Claude sweeps the entire project.

What is it:

After the overview, drill-down into specific areas: a module, a service, a data flow. Ask about specific parts you need to understand for your assignment.

Why learn:

You don't need to understand the entire project on the first day. You need to understand enough to do your task. Zoom in on what matters now, expand as needed.

Key concepts:

Prompt: "Now explain to me in detail how the authentication module works. What files are involved, what is the data flow and how tokens are managed?" Ask what you need for your task.

What is it:

Ask Claude to trace the complete flow of an operation: from user input to the final response. Follow the chain of calls between files and modules.

Why learn:

Understanding the flow reveals how the modules connect. It's the difference between knowing that a file exists and understanding how it is used in the real context of the application.

Key concepts:

Prompt: "Trace the complete flow of a POST /api/orders request: from middleware to database and back to response. Show me each file and function along the way." Claude follows the entire chain.

What is it:

Use /init to generate a CLAUDE.md that documents the project's architecture, conventions, and standards. This creates a "persistent memory" that every future session will automatically load.

Why learn:

Without /init, each new session starts from scratch. With CLAUDE.md generated, Claude already knows the project conventions, directory structure and code patterns from the first message.

Key concepts:

Run /init after exploring the project. Claude generates a CLAUDE.md with: architecture, stack, code conventions, build/test commands and naming standards. Review and adjust as necessary.

What is it:

Ask Claude "why" architectural decisions were made, not just "what" the code does. Understanding the intent behind the design is just as important as understanding the implementation.

Why learn:

Knowing that a cache exists is not enough. Knowing WHY a cache exists (bank performance, API limit, etc.) and what prevents you from removing something essential out of ignorance.

Key concepts:

Prompt: "Why does this module use event sourcing instead of direct CRUD?", "Why is there this caching layer here?", "What problem does this workaround solve?" Claude analyzes the context and infers reasons.

What is it:

The systematic approach to entering a new project: overview first, then critical flows, then details as needed. Don't try to understand everything at once.

Why learn:

Efficient onboarding reduces time to first contribution from weeks to hours. With Claude, you can explore the codebase interactively instead of reading static documentation.

Key concepts:

Recommended flow: 1) Big picture with Claude 2) /init to generate CLAUDE.md 3) Trace critical flows 4) First small task to validate understanding 5) Expand knowledge as needed.

🧪 Write Tests

The complete workflow to identify coverage gaps, generate tests and adopt TDD practices using Claude Code as a test co-pilot.

What is it:

Ask Claude to analyze the code and identify areas without test coverage: untested functions, uncovered branches, ignored edge cases and untested error scenarios.

Why learn:

You can't test what you don't know is missing. Claude analyzes the code and existing tests to map out exactly where the gaps are. This prioritizes the testing effort.

Key concepts:

Prompt: "Analyze existing tests and code in [directory]. Identify untested functions, uncovered branches, and untested edge cases. Prioritize by risk." Claude generates a gap map.

What is it:

Ask Claude to generate tests for the identified gaps. It creates tests that follow the patterns and framework already used in the project (jest, pytest, vitest, etc.).

Why learn:

Writing tests manually is time-consuming and repetitive. Claude generates tests in seconds, covering happy paths, edge cases and error scenarios. You review and adjust instead of writing from scratch.

Key concepts:

Prompt: "Write tests for [function/module] covering: happy path, invalid input, edge cases and error scenarios. Follow the same framework and patterns as existing tests." Claude analyzes existing tests and replicates the style.

What is it:

Run the tests generated by Claude and check that they all pass. Tests that fail need to be fixed - Claude may have generated incorrect assertions or incomplete mocks.

Why learn:

Tests generated by AI can have errors: incorrect mocks, wrong assertions, incomplete setup. Running and checking is essential before considering the task complete.

Key concepts:

Ask: "run the tests you created and fix any errors." Claude executes, analyzes errors and corrects. Repeat until all tests pass. Check that existing tests also continue to pass.

What is it:

Run the coverage report after adding the new tests and check if coverage has improved. Identify areas that are still uncovered and decide if they need further testing.

Why learn:

Coverage quantifies progress. Without metrics, you don't know if the new tests really covered what was missing. The coverage report and your proof of work.

Key concepts:

Ask: "run the tests with coverage and show me the report." Claude runs npm test --coverage / pytest --cov and analyzes the result. If the coverage is not enough, he suggests further testing.

What is it:

Use Claude Code for TDD: first write the test that fails, then implement the code that makes the test pass, then refactor. Red-Green-Refactor with AI.

Why learn:

TDD with Claude is extremely efficient. You describe the desired behavior, Claude writes the test, you validate it, and then Claude implements the code. Testing guides implementation.

Key concepts:

Prompt: "Write a test for [feature]. The test should fail because the function does not yet exist." Then: "Now implement the function to pass the test." Then: "Refactor while keeping the tests passing."

What is it:

Principles for quality testing: deterministic testing, isolation, descriptive names, arrange-act-assert and when to use mocks vs real data.

Why learn:

Bad tests are worse than no tests at all: they give false confidence, they break for the wrong reasons, and they are difficult to maintain. Good tests document behavior and protect against regression.

Key concepts:

Ask Claude: "Review these tests for good practice." It checks: deterministic tests, without order dependence, clear names, specific (non-generic) assertions, and adequate isolation between tests.

🔍 Code Review

The workflow for using Claude Code as a reviewer. From asking for review to addressing feedback and preparing code for merge.

What is it:

Ask Claude to review your changes before creating a PR. You can be specific: "security review", "performance review", or ask for a complete general review.

Why learn:

All code benefits from a second pair of eyes. Claude identifies problematic patterns, vulnerabilities, and inconsistencies that you don't notice when you're immersed in the code.

Key concepts:

Prompt: "Review my changes since the last commit. Focus on: potential bugs, security, performance and readability." The more specific the focus, the better the quality of the review.

What is it:

The slash command /review automatically reviews the current diff. Analyzes all staged and unstaged changes looking for security, bugs, performance and style issues.

Why learn:

/review is faster than writing a review prompt. A single command and you have a structured review. Ideal to use before every /commit and every /pr.

Key concepts:

Type /review and Claude analyzes the full diff. It returns: issues found by category (bug, security, performance, style), severity and correction suggestions. Use as pre-commit check.

What is it:

Solve the problems highlighted in the review. You can ask Claude to automatically correct or decide on a case-by-case basis which suggestions to accept.

Why learn:

Not all feedback needs to be accepted. But each point must be evaluated. Learning to triage - accepting critical corrections, discussing style suggestions, discarding false positives - is a review skill.

Key concepts:

Prompt: "Fix security issues and bugs. For style issues, apply only those that make sense for this project's standard." You direct which changes to apply.

What is it:

Run /review again after applying the fixes. Check that the problems have been resolved and that the corrections have not introduced new issues.

Why learn:

Corrections can introduce new problems. Re-review is the feedback loop that ensures you're not playing whack-a-mole with bugs. Only proceed when the review is clear.

Key concepts:

Run /review again. If clear, proceed. If new issues, fix and repeat. The review > fix > re-review cycle ensures quality before commit and eliminates problems iteratively.

What is it:

Request reviews focused on specific areas: security (SQL injection, XSS, authentication), performance (N+1 queries, memory leaks, algorithms), or compliance (coding standards, patterns).

Why learn:

Generic reviews catch superficial problems. Focused reviews catch deep problems. A focused security review finds vulnerabilities that a general review misses.

Key concepts:

Prompts: "Review this code ONLY for security vulnerabilities. Look for: injection, XSS, bypassed authentication, sensitive data exposed." Or: "Review for performance: N+1 queries, unnecessary loops, excessive allocations."

What is it:

A mental checklist (and Claude's) to validate before opening a PR: tests passing, review clean, no forgotten console.logs, no pending TODOs, no secrets committed.

Why learn:

PRs who come back with comments about basic errors waste everyone's time. The pre-PR checklist eliminates the most common mistakes before asking for human review.

Key concepts:

Prompt: "Do a pre-PR checklist: tests pass? review clean? console.logs removed? ALL resolved? secrets in the code? imports not used? correct types?" Claude checks each item automatically.

🚀 Create Pull Request

The workflow to prepare commits, use /pr and generate complete and professional pull requests with description generated by Claude.

What is it:

Before creating the PR, organize the commits: squash if necessary, clear messages, atomic commits. The commit history is part of the PR communication.

Why learn:

A PR with 47 "wip" and "fix typo" commits makes review difficult. Well-organized commits tell the history of the change and allow review by commit, not by file.

Key concepts:

Ask Claude: "review my commit history and suggest how to organize it for the PR." It may suggest squashing WIP commits, logical reordering, or keeping it as is if it's already good.

What is it:

The slash command /pr automatically creates a pull request: analyzes all commits, generates title, detailed description, and opens the PR in GitHub/GitLab directly from the terminal.

Why learn:

Creating PRs manually requires writing descriptions, listing changes, adding context. /pr does all this automatically based on commits and diff, saving minutes on each PR.

Key concepts:

Type /pr and Claude: 1) analyzes all commits in the branch, 2) generates descriptive title, 3) creates description with context, changes and impact, 4) opens the PR via gh cli. All in one command.

What is it:

The content that Claude generates for the PR: concise title, summary of changes, motivation/context, list of modified files, test instructions and review notes.

Why learn:

Understanding what Claude generates allows you to adjust and improve. It creates an excellent foundation, but you may want to add screenshots, links to issues or context that only you know.

Key concepts:

Claude generates: title in the repo pattern, description with "What" and "Why", list of changes per file, breaking changes if any, and instructions on how to test. And an automatic professional PR description.

What is it:

Review the PR description generated by Claude before submitting. Check that the description is accurate, that the context is correct and that there is no missing important information.

Why learn:

Claude doesn't know about Slack conversations, meeting decisions or business context. You need to add that human context that makes the PR description really useful for reviewers.

Key concepts:

Check: does the title reflect the change? Does the description explain the “why”? Are links to issues correct? Are breaking changes documented? Are testing instructions clear? Add what's missing.

What is it:

Make final adjustments to the PR: add labels, assignees, reviewers, milestone and any metadata that /pr did not add automatically.

Why learn:

PR metadata facilitates sorting and organization. Labels categorize (bug, feature, refactor), reviewers speed up the process, and milestones connect to planning. /pr creates the PR, you customize it.

Key concepts:

After /pr creates the PR, add: labels (type of change, priority), reviewers (who needs to approve), milestone (sprint/release) and linked issues (closes #123). Claude can help via gh cli.

What is it:

Principles for quality PRs: ideal size (200-400 lines), a concern per PR, description that tells a story, and quick response to review comments.

Why learn:

Giant PRs are not reviewed carefully. Small, focused PRs receive quick, quality reviews. The size of the PR directly affects the quality of the feedback you receive.

Key concepts:

Rules: PRs < 400 lines, one purpose per PR, description with context and test instructions. If the feature is large, divide it into smaller chained PRs. Reply review comments in < 24h.

🪵 Debug with Logs and Errors

The workflow for debugging using logs, stack traces and error messages. Paste the error, pipe the logs and let Claude trace the root cause.

What is it:

The most direct way to debug: copy the complete error (stack trace, message, context) and paste directly into Claude Code. It analyzes the error in the context of your codebase.

Why learn:

Claude has access to your codebase. When you paste an error, it not only interprets the stack trace but navigates to the mentioned files and understands the real context of the error.

Key concepts:

Paste the complete error, not just the message. Include: stack trace, error line, type of exception, and context of when it happened. Prompt: "This error happens when [action]. Here is the stack trace: [paste everything]."

What is it:

Use pipe to send logs directly to Claude: tail -f app.log | claude -p "parse these logs". Ideal for long logs that don't fit in a manual folder.

Why learn:

Production logs can be thousands of lines long. Pipe allows you to filter and send only the relevant one: tail -100 app.log | grep ERROR | claude -p "what is causing these errors?".

Key concepts:

Examples: cat error.log | claude -p "analysis". tail -500 app.log | grep -A5 "Exception" | claude -p "identify error pattern". Combine grep to filter before sending to Claude.

What is it:

Ask Claude to trace the root cause starting from the error: follow the stack trace, analyze the data flow, identify where the state became inconsistent and why.

Why learn:

Claude traces automatically: reads the error line file, follows the chain of calls, checks inputs and outputs of each function. And how to have a debugger that understands semantics.

Key concepts:

Prompt: "Trace the root cause of this error. Start with the stack trace line and follow the chain of calls until you find where the data became invalid." Claude browses the files and presents the causal chain.

What is it:

Apply the correction based on the identified root cause. Claude edits the necessary files with minimal changes to resolve the issue.

Why learn:

After understanding the root cause, the correction must be surgical. Minimal changes are easier to review, test, and revert if necessary. Claude applies exactly what is necessary.

Key concepts:

Prompt: "Apply the minimum fix to resolve the identified root cause." Review the diff. If the fix is ​​more complex than expected, question whether the root cause is actually correct.

What is it:

Check that the fix really resolved the problem: reproduce the scenario that caused the error and confirm that it now works. Run related tests.

Why learn:

A fix that is not verified is not a fix. Reproduce the original error and confirm that it no longer happens. Run the test suite to ensure nothing else broke.

Key concepts:

Ask: "run the scenario that caused the error and confirm that it is resolved. Then run all the tests." Claude runs the verification test and completes the suite, reporting the results.

What is it:

Advanced techniques: add temporary logging with Claude, use strategic breakpoints, debug race conditions and timing problems.

Why learn:

Not every bug has a clear stack trace. Intermittent bugs, race conditions and state issues require specialized approaches. Claude can suggest and implement advanced debugging strategies.

Key concepts:

Prompt: "Add strategic temporary logging to track the value of [variable] throughout the flow." Or: "This bug is intermittent - suggest a debugging strategy for race conditions." Remove temporary logs after resolution.

🔀 Multi-file Refactoring

The workflow for large refactorings that affect many files. Planning, executing in batches and using /compact during long migrations.

What is it:

Use plan mode (Shift+Tab) to map ALL files affected by the refactoring before touching any code. Claude creates a complete inventory of needed changes.

Why learn:

Multi-file refactoring without a plan and chaos. You change one file and break 15 others. The plan shows all dependencies, order of execution and risks before starting.

Key concepts:

Prompt in plan mode: "Plan the refactoring from [X] to [Y]. List ALL affected files, the order of change and dependencies between them." Claude generates a complete migration plan.

What is it:

Critically review the scope of the refactoring: how many files, which are critical, what is the order of dependency, and whether the scope has not grown beyond what is necessary.

Why learn:

Scope creep is the enemy of large refactorings. Claude may include "nice to have" changes that are not necessary now. Review and cut the scope to the minimum viable.

Key concepts:

Ask: "Does this file really need to change at this stage?" and "Can we break this refactoring into smaller phases?" Prioritize critical changes and postpone optional ones to another PR.

What is it:

Divide the refactoring into batches of 3-5 files at a time. Each batch is a complete unit that can be tested independently before proceeding.

Why learn:

Changing 50 files at once is impossible to debug if something breaks. In batches of 5, you know exactly which batch introduced the problem. It's safer and more controllable.

Key concepts:

Prompt: "Run batch 1 of plan: files [list]. Apply the changes and let me know when you're done." Test. Commit. Next batch. Each batch is a security checkpoint.

What is it:

Run all tests after each batch of changes. If tests fail, correct BEFORE moving on to the next batch. Never accumulate batches without validation.

Why learn:

Testing after each batch keeps the project functional throughout the migration. If you accumulate 5 batches without testing and something breaks, finding the cause in 25 files is a nightmare.

Key concepts:

Flow: batch > test > fix if necessary > commit > next batch. Ask: "run all tests after this batch." If it fails: "correct the failures by keeping the batch changes." Never skip tests.

What is it:

After all the batches, do a final scan: check that there are no old references, broken imports, or inconsistencies between the migrated files.

Why learn:

Batches can leave inconsistencies between them: one file uses the new interface, another still references the old one. The final scan catches these gaps that individual tests do not capture.

Key concepts:

Prompt: "Do a final scan: look for old references to [old name], broken imports, inconsistent types, and any artifacts from the refactoring left behind." Claude sweeps the entire project.

What is it:

Use /compact strategically during long migrations to free up context without losing the migration plan. Compact while preserving the status: which batches were made, which are missing.

Why learn:

Long migrations consume a lot of context. Without /compact, you reach the limit and Claude starts to forget the plan. With strategic /compact, you keep the session functional throughout the migration.

Key concepts:

Prompt for /compact: "Compact preserving: the migration plan, which batches have already been completed (1-3), which are missing (4-7), and any important decisions made." Claude compresses while maintaining the critical context.
