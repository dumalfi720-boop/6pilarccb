MODULE 3.6

👾 Code Review

Use Claude Code to automatically code review your changes before submitting, catching bugs and improvements.

1

🔍 Ask Claude for a review

Before opening a PR, ask Claude toreview your changes. It finds bugs, vulnerabilities and suggestions for improvement.

💡Main Concept

Different ways to request a review:

  • •"Revise minhas mudancas. Foque em bugs, seguranca e performance"
  • •"Analise o git diff e me diga se tem algum problema antes de eu fazer PR"
  • •Claude reads the diff, understands the context and points out specific problems with correction suggestions

⚡Practical Tip

Via pipe:git diff | claude -p "Faca code review deste diff. Aponte bugs, security issues e melhorias"

2

⚡ The /review Command

The command/reviewand the shortcut for automatic code review. It analyzes all pending changes at once.

💡Main Concept

How /review works:

  • •Enter/reviewat Claude Code
  • •It analyzes the full diff (staged + unstaged) against the base branch
  • •Returns a categorized list: bugs, security, performance, style, suggestions

⚡Practical Tip

Use/reviewas your last step before any PR. And how to have a senior dev review your code instantly.

3

🔧 Resolve Feedback

After the review,resolve each feedback itemdirectly asking Claude to apply the suggested corrections.

💡Main Concept

Feedback and correction cycle:

  • •"Corrija todos os itens que voce apontou no review"
  • •In other words, be selective:"Corrija os bugs e issues de seguranca. Ignore as sugestoes de estilo"
  • •Claude applies the corrections while maintaining the context of the original review

⚡Practical Tip

Disagree when necessary:"O item 3 nao se aplica porque X. Corrija apenas os itens 1, 2 e 4"

4

🔁 Re-Review

After the corrections, run/review againto confirm that all problems have been resolved and no new ones have been introduced.

💡Main Concept

The review-fix-review cycle guarantees quality:

  • •1. /review→ finds problems
  • •2. "Fix the problems" → apply fixes
  • •3. /review→ confirms that everything is OK

⚡Practical Tip

A "clean review" (no issues) and the green light to make the PR. If Claude doesn't find anything, you can go with confidence.

5

🎯 Focus Areas

Direct the review tospecific areasdepending on the type of change you made.

💡Main Concept

Customize the focus of the review by type of change:

  • •Auth/security:"Revise focando em: SQL injection, XSS, CSRF, validacao de input"
  • •Performance:"Revise focando em: N+1 queries, memory leaks, loops desnecessarios"
  • •API:"Revise focando em: breaking changes, validacao de request, error handling"

⚡Practical Tip

For database changes:"Revise as migrations. Sao reversiveis? Tem risco de data loss? Precisa de backfill?"

6

📋 Review Checklist

Create onepersonalized checklistfor Claude to use in all reviews of his project.

💡Main Concept

Add a checklist to CLAUDE.md for consistent reviews:

  • •Tests included for new features
  • •Error handling at all points of failure
  • •No secrets or hardcoded credentials
  • •Logs suitable for debugging in production
  • •Documentation updated if the API has changed

⚡Practical Tip

Add to CLAUDE.md:"Ao fazer /review, sempre verifique: [sua checklist aqui]". Claude had followed automatically.
