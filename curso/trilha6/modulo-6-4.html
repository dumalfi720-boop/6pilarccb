🐙 What GitHub MCP Does

Main Concept

🎯Direct Access to GitHub

GitHub MCP connects Claude to the GitHub API via the MCP protocol. It can list and read issues, create and manage pull requests, check CI/CD status, search repositories and read content from remote files. All without leaving the terminal.

  • • list_issues/get_issue:Access issues with filters (label, state, assignee)
  • • create_pull_request:Create PRs with title, body and branch
  • • get_pull_request_reviews:Read feedback from reviewers
  • • search_repositories:Search repos by name, topic or language
Why It Matters

📊Claude As Team Member

  • With GitHub MCP, Claude understands the CONTEXT of the project: who requested what, which issues are open
  • Complete workflow: read issue -> implement -> create PR -> without copy-paste
  • Automated reviews with real PR context, not assumptions
Practical Tip

💡Practical Tip

The most powerful workflow with GitHub MCP: 'read issue #42, implement what was requested and create a PR linking the issue'. Claude searches for the issue, understands the requirements, implements it in the code and creates the formatted PR. A single prompt replaces 15 minutes of manual work.

Do vs Avoid

✓ What TO DO

  • ✓ Use to integrate issues with implementation
  • ✓ Request specific queries with filters
  • ✓ Combine with Review Skills for complete PRs

✗ What NOT to do

  • ✗ List all issues without filter (fills the context)
  • ✗ Create PRs without reviewing what Claude generated
  • ✗ Leave connected during pure coding sessions

🎯 Why It’s Worth It

Main Concept

🎯Full Project Context

Without GitHub MCP, you need to copy and paste issues, PR comments, and reviewer feedback manually. With GitHub MCP, Claude searches everything himself and works with the full context of what is happening in the repository.

  • • Without MCP: copy-paste from issue -> prompt -> code -> manual PR = 15-20 min
  • • With MCP: 'implement issue #42 and create PR' = 2-5 min
  • • Savings: 75%+ of time in issue-to-PR workflows
Why It Matters

📊Impact on Productivity

  • Issues as direct input: Claude reads the issue and translates it into code
  • Reviews as feedback: read comments and adjust code automatically
  • CI/CD as validation: checks if checks passed and corrects if not
Practical Tip

💡Practical Tip

To maximize value from GitHub MCP, structure your issues clearly. A well-written issue with acceptance criteria becomes a perfect prompt for Claude to implement. Clear issue + GitHub MCP = almost automatic implementation.

Do vs Avoid

✓ What TO DO

  • ✓ Write clear issues with acceptance criteria
  • ✓ Use GitHub MCP to close the issue->code->PR loop
  • ✓ Check CI/CD status after creating PRs

✗ What NOT to do

  • ✗ Waiting for Claude to implement vague issues correctly
  • ✗ Ignore reviewer feedback accessible via MCP
  • ✗ Use for tasks that don't involve GitHub

⚙️ Installation + Token

Main Concept

🎯Setup with Authentication

Installation requires two steps: configuring GitHub's Personal Access Token (PAT) as an environment variable and running the MCP add command. The token defines what permissions Claude will have on GitHub.

  • • Step 1:Create a PAT at github.com → Settings → Developer settings → Personal access tokens → Fine-grained
  • • Step 2: export GITHUB_TOKEN=github_pat_xxxxx
  • • Step 3: claude mcp add github -- npx -y @anthropic/mcp-github
  • • Step 4:Check with/status
Why It Matters

📊Token Security

  • Use Fine-grained PAT (not classic) for precise control
  • Minimum scopes: Contents (read), Issues (read/write), PRs (read/write)
  • Add GITHUB_TOKEN to your .bashrc or .zshrc for persistence
Practical Tip

💡Practical Tip

Never use a token with admin permission on GitHub MCP. Create a Fine-grained token with minimal scopes: access only to the repos you work on, read permissions for most, and write permissions only for issues and PRs. Review and rotate the token every 90 days.

Do vs Avoid

✓ What TO DO

  • ✓ Use Fine-grained PAT with minimal scopes
  • ✓ Add GITHUB_TOKEN to shell profile
  • ✓ Rotate token every 90 days

✗ What NOT to do

  • ✗ Use classic token with full access
  • ✗ Share token with others
  • ✗ Commit token to project files

💼 Use Cases

Main Concept

🎯Practical Workflows

GitHub MCP enables workflows that integrate project management (issues, PRs) with implementation (code). The most valuable cases are those that eliminate switching between browser and terminal.

  • • Issue-driven:'read issue #42 and implement' → Claude searches, implements and can create PR
  • • Review-driven:'read reviews of PR #15 and adjust the code' → Claude reads feedback and applies corrections
  • • Triage:'list issues with label bug created this week' → Claude filters and summarizes
  • • CI-fix:'check why CI failed in the last commit and fix it' → Claude diagnoses and fixes
Why It Matters

📊ROI Per Use Case

  • Issue-driven: saves 10-15 min per issue (no copy-paste)
  • Review-driven: saves 5-10 min per review cycle
  • Triage: saves 15-30 min per screening session
Practical Tip

💡Practical Tip

The use case with the best ROI and issue-driven development: keep your issues well structured and ask Claude to 'read issue #X, implement and create PR'. For complex issues, break the issue into sub-tasks and ask Claude to implement them one by one.

Do vs Avoid

✓ What TO DO

  • ✓ Use issue-driven development as a standard workflow
  • ✓ Combine with Review Skills for more complete PRs
  • ✓ Filter queries to minimize returned data

✗ What NOT to do

  • ✗ Request huge lists without filter
  • ✗ Create PRs without reviewing the generated diff
  • ✗ Use for repos that you do not have permission to

🟡 Token Impact: Moderate (~2,000 tokens)

Main Concept

🎯Balanced Cost-Benefit

GitHub MCP has moderate overhead: ~2,000 tokens per message. This puts it in the middle ground - more expensive than Filesystem but much cheaper than Browser. The cost is justifiable when you are actively working on issues and PRs.

  • • ~2,000 tokens/msg = 1% of 200k window
  • • In 50 messages: 100k tokens overhead (significant)
  • • Justifiable when: actively working with issues/PRs/reviews
Why It Matters

📊When to Connect vs Disconnect

  • Connect: sprint start, issue triage, PR cycle
  • Disconnect: pure coding without reference to issues, internal refactoring
  • Rule: if you are not going to interact with GitHub in the next 10 msgs, disconnect
Practical Tip

💡Practical Tip

An efficient strategy: connect GitHub MCP at the beginning of the session to read issues and understand what to implement. Then disconnect and code with just Filesystem. At the end, reconnect to create the PR. This saves ~2000 tokens x pure coding messages.

Do vs Avoid

✓ What TO DO

  • ✓ Connect when working with issues/PRs
  • ✓ Disconnect during pure coding
  • ✓ Use the connect-disconnect-reconnect strategy

✗ What NOT to do

  • ✗ Always keep it on 'for convenience'
  • ✗ Ignore the 2,000 token overhead in long sessions
  • ✗ Connect for tasks that don't involve GitHub
TOPIC 6
6

🎯 Specific Queries

Main Concept

🎯Optimize Your Orders

The key to using GitHub MCP efficiently is to be specific in requests. Broad queries return a lot of data that fills the context. Focused queries return only what is necessary.

  • • Bad:'list all issues in the repo' → can return hundreds of issues
  • • Good:'list issues with label bug created this week' → returns 3-5 issues
  • • Bad:'show PR #42 complete with diff' → huge diff in context
  • • Good:'show title, description and comments of PR #42' → focused data
Why It Matters

📊Impact on Context

  • Wide query: can inject 10k-50k data tokens into context
  • Focused Query: Injects 500-2000 relevant data tokens
  • The difference could be 10-25x fewer tokens spent
Practical Tip

💡Practical Tip

Always filter by: label, state (open/closed), assignee, created date. Ask for specific fields instead of 'all'. For PRs, ask for 'title and description' before asking for 'full diff'. Evaluate whether you really need the diff in the context or whether the local code is already sufficient.

Do vs Avoid

✓ What TO DO

  • ✓ Filter by label, state, date, assignee
  • ✓ Request specific fields instead of complete objects
  • ✓ Assess need before asking for large diffs

✗ What NOT to do

  • ✗ List 'all issues' without filter
  • ✗ Request full diffs of large PRs
  • ✗ Search entire repos without specifying filters
