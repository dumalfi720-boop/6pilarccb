===== MODULE 6.1 =====
6.1 ~12 min

💰 Token Tax and Context Cost

Understand what token tax is, how MCPs consume context, and why monitoring /cost is essential for operators connecting external services.

What is it:

Token tax is the cost in tokens that each connected MCP adds to EVERY interaction, even when you are not using that MCP. The tools schema is injected into the context automatically.

Why learn:

Connecting 5 MCPs at the same time can consume 30-50% of your context with schemas alone. Without understanding token tax, you lose response quality without knowing why.

Key concepts:

Each MCP exposes tools with descriptions and parameters. These definitions are sent as input tokens in each message. More MCPs = more sticky tokens = less space for your code.

What is it:

Filesystem: ~500 tokens (Low). GitHub: ~2,000 tokens (Moderate). Browser: ~5,000+ tokens (HIGH). Supabase: ~1,500 tokens (Moderate). Google Sheets: ~800 tokens (Low-Moderate).

Why learn:

With the overhead table, you plan which MCPs to keep connected and which to activate on demand. It is the basis of the active rotation strategy (module 6.9).

Key concepts:

MCPs with many tools (Browser has 10+) cost more. Simple MCPs (Filesystem has 3-4 tools) cost little. The overhead is FIXED per message, not per use.

What is it:

Think of context as oxygen inside an astronaut helmet. The more MCPs you connect, the more tubes share the same tank. With less oxygen, Claude thinks worse.

Why learn:

Metaphor makes something abstract tangible. Operators who understand this naturally avoid connecting unnecessary MCPs and prioritize space for code and instructions.

Key concepts:

Context window = oxygen tank (200k tokens). MCPs = equipment that consumes oxygen. Code + conversation = what needs oxygen to function well.

What is it:

Rule of thumb: keep a maximum of 2-3 MCPs connected simultaneously. More than that and you start to sacrifice quality of response for convenience.

Why learn:

Having a mental limit prevents the gradual accumulation of MCPs that erode performance without you realizing it. It is the basis of operational discipline with MCPs.

Key concepts:

2-3 MCPs = sweet spot. 4-5 MCPs = noticeable degradation. 6+ MCPs = significant waste. Always unplug what you are not actively using.

What is it:

Use /cost before and after connecting MCPs to see the real impact on token consumption. The difference between sessions with and without MCPs reveals the token tax in practice.

Why learn:

Theory without medication and guesswork. /cost transforms token tax from an abstract concept into a concrete number that you can optimize.

Key concepts:

Compare: session without MCPs vs with MCPs doing the same task. The difference in /cost is the actual token tax. Use this to justify which MCPs are worth it.

What is it:

With many MCPs, Claude is given less space to "think" about his code. Answers become shorter, less detailed and contain more errors. The competitive context degrades effective intelligence.

Why learn:

Many operators blame the model when quality drops, but the real problem is saturated context. Understanding this avoids frustration and optimizes results.

Key concepts:

Context < 50% used = great answers. 50-70% = good answers. 70-90% = visible degradation. 90%+ = frequent errors and truncated answers. MCPs accelerate this curve.

🧭 Decision Framework: Skill vs MCP

Learn when to use a Skill and when to use an MCP. Practical decision table to never make the wrong choice again.

What is it:

If the task follows a predictable and repetitive pattern (code review, create component, generate tests), it is a perfect candidate for Skill. Skills are Markdown instructions, zero token overhead.

Why learn:

Using MCP for something that a Skill solves and wasting tokens. Skills cost zero in token tax, MCPs cost hundreds to thousands per message.

Key concepts:

Skill = reusable prompt (zero infra). Examples: /review, /create-component, /generate-tests. If the task does not need live external data, it is Skill.

What is it:

If the task needs live data from an external service (query database, read GitHub issues, access spreadsheet), it requires MCP. Skills do not access APIs.

Why learn:

Trying to do with Skill what requires live data results in inventions and hallucinations. Claude would fabricate data instead of admitting that he doesn't have access.

Key concepts:

MCP = bridge to living services. Examples: querying DB production, reading open issues, accessing spreadsheet data. If you need fresh data, and MCP.

What is it:

Formatting tasks, code standardization, boilerplate generation, template creation - all of this is the domain of Skills. These are instructions that shape output, not fetch data.

Why learn:

Many operators create complex MCPs for tasks that a 20-line Skill solves. The result is unnecessary overhead and more points of failure.

Key concepts:

Skill for: code style, commit messages, PR templates, component boilerplate, documentation format. If and about HOW to format, and Skill.

What is it:

Collecting data from websites, accessing pages, extracting information from URLs - all of this requires a browser MCP or fetch. Claude does not have native internet access.

Why learn:

Web scraping is one of the most powerful use cases of MCPs. With Browser MCP, Claude reads real pages and extracts structured data.

Key concepts:

MCP Browser for: reading web pages, extracting prices, collecting leads from LinkedIn, checking competitor websites. High cost of tokens but high output value.

What is it:

When token efficiency is a priority (long sessions, plans with limits), prefer Skills whenever possible. Skills consume zero fixed overhead.

Why learn:

In long sessions with frequent compressions, each token saved multiplies. A Skill that replaces an MCP can save thousands of tokens per session.

Key concepts:

Skills = 0 token tax. MCPs = 500-5000+ fixed tokens. In 50 messages, an MCP of 2000 tokens costs an extra 100k tokens. Prefer Skill when possible.

What is it:

Table: Code review -> Skill. Consult DB -> MCP. Generate component -> Skill. Read GitHub issues -> MCP. Format commit -> Skill. Browse website -> MCP.

Why learn:

A quick reference eliminates hesitation. Stick it on the wall or on CLAUDE.md and consult it whenever you have any questions about Skill vs MCP.

Key concepts:

Rule of thumb: If the answer to "need live external data?" for YES = MCP, NO = Skill. Simple, effective, and covers 90% of cases.

📂 MCP #1: Filesystem

The lightest and highest value MCP. Access files outside the project directory, manage multiple repositories, and navigate the entire file system.

What is it:

Filesystem MCP expands Claude's access beyond the current directory. With it, Claude reads and writes files in any path on your system that you authorize.

Why learn:

Without this MCP, Claude only accesses the directory where it was started. With it, you can copy configs between projects, access global templates and manage multiple repos.

Key concepts:

Expoe tools: read_file, write_file, list_directory, search_files. It works via absolute paths with permissions that you define at installation.

What is it:

Filesystem is the most cost-effective MCP: ~500 tokens overhead for full file system access. It is the only MCP that many operators always keep connected.

Why learn:

Knowing that the Filesystem is "always on" simplifies decisions. While other MCPs are rotatable, this one is fixed in its setup.

Key concepts:

~500 tokens overhead (less than a short message). Enables cross-repo workflows. Very high ROI for any type of project.

What is it:

Install with: claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem /path/permissive. Replace /path/allowed with the directory you want to expose.

Why learn:

The command is simple but path configuration is critical for security. Exposing / (root) is dangerous. Prefer specific paths like ~/projects.

Key concepts:

claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem ~/projetos. You can add multiple space-separated paths. Paths are whitelisted, not blacklisted.

What is it:

Cases: copying components between projects, accessing global configs (~/.config), reading templates from another repo, comparing implementations between different projects.

Why learn:

Advanced operators work on multiple projects simultaneously. The MCP Filesystem is what enables this fluid multi-repo workflow.

Key concepts:

Cross-repo: copy patterns from one project to another. Global configs: read .bashrc, .gitconfig, etc. Templates: access boilerplate from anywhere.

What is it:

Filesystem MCP has the lowest token rate among popular MCPs: ~500 tokens per message. That's less than a paragraph of text and practically insignificant.

Why learn:

Knowing that it is Low impact justifies keeping it always connected. It doesn't make sense to disconnect something that costs so little and delivers so much value.

Key concepts:

~500 tokens = 0.25% of 200k context. You can keep it on all the time. The only MCP where "always on" is recommended for everyone.

What is it:

Best practices: never expose / (root). Use specific paths. Do not expose ~/.ssh or ~/.aws. Prefer ~/projects or ~/dev. Review permissions periodically.

Why learn:

Security with Filesystem MCP and criticism. A misconfigured path can expose SSH keys, AWS credentials, and other secrets to the Claude context.

Key concepts:

Whitelist > blacklist. Least privilege: expose only what is necessary. Never expose directories with credentials. Review the list periodically with claude mcp.

🐙 MCP #2: GitHub

Connect Claude directly to GitHub to manage issues, PRs, reviews and repositories without leaving the terminal.

What is it:

GitHub MCP connects Claude to the GitHub API. Claude can list issues, create PRs, read reviews, verify CI/CD checks and manage repositories programmatically.

Why learn:

GitHub is the central development hub. With this MCP, Claude understands the complete context of his project: open issues, pending PRs, feedback from reviewers.

Key concepts:

Tools: list_issues, create_pull_request, get_pull_request_reviews, search_repositories, get_file_contents. Direct access via GitHub API with token authentication.

What is it:

With GitHub MCP, Claude knows which issues are open, who requested what, and what the reviewers' feedback is. This transforms Claude from "code editor" to "team member".

Why learn:

Without GitHub MCP, you need to manually copy/paste issues and comments. With it, Claude searches and processes everything himself, saving you time.

Key concepts:

Workflow: "read issue #42 and implement" - Claude searches for the issue, understands the requirements and implements it. Zero copy-paste, 100% automatic context.

What is it:

Install with: claude mcp add github -- npx -y @anthropic/mcp-github. Configure GITHUB_TOKEN as an environment variable with a GitHub Personal Access Token (PAT).

Why learn:

Installation is simple but token configuration requires attention: define minimum scopes (repo, read:org) and never use tokens with admin permissions.

Key concepts:

1. Create a PAT at github.com/settings/tokens. 2. export GITHUB_TOKEN=ghp_xxx. 3. claude mcp add github -- npx -y @anthropic/mcp-github. 4. Check with /status.

What is it:

Cases: "read issue #42 and implement", "create a PR with these changes", "list issues with label bug", "check if the CI passed the last commit".

Why learn:

Each use case represents hours saved. Claude can triage issues, create formatted PRs and even respond to reviews - all via MCP.

Key concepts:

Issue-driven: read issue -> implement -> create PR -> linked issue. Review-driven: read comments -> adjust code -> respond to review. CI-driven: check for faults -> fix.

What is it:

GitHub MCP has moderate impact: ~2,000 tokens per message. Justifiable when you are actively working on issues and PRs, but unnecessary in purely code tasks.

Why learn:

Knowing that it is Moderate helps you decide when to connect/disconnect. Use it when you need to interact with GitHub, disconnect when you're just coding.

Key concepts:

~2,000 tokens = 1% of context. Connect when: working with issues/PRs. Disconnect when: coding features without reference to issues. Smart rotation.

What is it:

Instead of "list all issues", ask "list issues with label bug created this week". Specific queries return less data and consume fewer response tokens.

Why learn:

Broad queries return hundreds of issues that fill the context. Focused queries return only what is necessary, keeping the context clean for code.

Key concepts:

Filter by: label, assignee, state, created date. Ask for specific fields. Avoid listing entire repos. The more precise the order, the lower the cost in tokens.

===== MODULE 6.5 =====
6.5 ~12 min

🌐 MCP #3: Browser/Chrome

The most powerful and most expensive MCP. Give Claude eyes to browse the web, extract data and check UI in real time.

What is it:

MCP Browser gives Claude the ability to open web pages, click on elements, fill out forms, take screenshots and extract content from any website.

Why learn:

With Browser MCP, Claude becomes a complete web agent. It can check if your deployment worked, scrape data from competitors, or test UI automatically.

Key concepts:

Tools: navigate, click, fill, screenshot, get_text, evaluate_js. Uses Puppeteer/Playwright behind. Claude sees and interacts with the web like a real user.

What is it:

Browser is the only MCP that gives Claude "eyes". No other MCP allows you to visually check UI, scrape live data, or test user flows. It is irreplaceable for certain tasks.

Why learn:

Operators who master Browser MCP can automate workflows that previously required manual intervention: visual QA, lead scraping, competitive analysis.

Key concepts:

Elite because: unique with visual capabilities. Expensive because: screenshots consume a lot of tokens. Strategy: use punctually, never "always on".

What is it:

Install with: claude mcp add browser -- npx -y @anthropic/mcp-puppeteer. Requires Chrome/Chromium installed on the system. Works in headless mode by default.

Why learn:

The installation may fail if Chrome is not available. Knowing the prerequisites avoids unnecessary troubleshooting.

Key concepts:

Prereq: Chrome/Chromium. Command: claude mcp add browser -- npx -y @anthropic/mcp-puppeteer. Test: ask Claude to navigate to a website and take a screenshot.

What is it:

Cases: visually verifying whether a deployment is correct, extracting data from web pages, collecting leads from LinkedIn, testing checkout flows, monitoring competitor prices.

Why learn:

Browser MCP turns Claude into a visual QA assistant and an intelligent web scraper. Tasks that used to take hours manually now take minutes.

Key concepts:

QA: "navigate to localhost:3000 and verify that the header is correct." Scraping: "extract prices from product.com". Leads: "collect names and emails from directorio.com".

What is it:

Browser and most expensive MCP: ~5,000+ tokens of fixed overhead per message, and screenshots add thousands of extra tokens. A single screenshot can cost 10,000+ tokens.

Why learn:

Without this knowledge, operators leave Browser connected during pure coding sessions, wasting 5,000+ tokens per message for no benefit.

Key concepts:

~5,000 tokens fixed overhead + tokens per screenshot/page. In 20 messages = 100k+ tokens wasted if you are not using them. ALWAYS disconnect when you don't need to.

What is it:

MCP Browser rule of thumb: connect ONLY when you need visual access. Pure backend? Disconnect. Refactoring API? Disconnect. So do you need UI check? Connect, use, disconnect.

Why learn:

The Browser connect/disconnect discipline is what separates efficient operators from wasteful operators. And rule #1 of active rotation.

Key concepts:

Workflow: claude mcp add browser (when needed) -> use -> claude mcp remove browser (when finished). Never leave it on “for convenience”.

🗄️ MCP #4: Database/Supabase

Connect Claude directly to your PostgreSQL or Supabase database. Queries, migrations and data analysis without leaving the terminal.

What is it:

Database MCP connects Claude to PostgreSQL databases. Claude can execute SQL queries, explore schemas, analyze data and even suggest migrations based on the current state of the database.

Why learn:

With direct access to the database, Claude can debug data issues, validate migrations before applying, and write optimized queries based on the real schema.

Key concepts:

Tools: execute_query, list_tables, describe_table, get_schema. Claude sees the real structure of the database and writes SQL based on it, not on assumptions.

What is it:

For generic PostgreSQL: claude mcp add postgres -- npx -y @anthropic/mcp-postgres "postgresql://user:pass@host:5432/dbname". The connection string is passed as an argument.

Why learn:

PostgreSQL is the most popular database for modern apps. Knowing how to connect directly allows for fast debugging and data exploration without external tools.

Key concepts:

Connection string: postgresql://user:pass@host:5432/db. For local: localhost. For cloud: use the provider's hostname. NEVER commit the connection string.

What is it:

For Supabase: claude mcp add supabase -- npx -y @supabase/mcp-server-supabase --supabase-url https://xxx.supabase.co --supabase-key your-service-role-key.

Why learn:

Supabase adds features beyond pure SQL: auth, storage, realtime. The Supabase-specific MCP exposes these APIs directly to Claude.

Key concepts:

Use service_role key (not anon key) for full access. Find URL and key at supabase.com/dashboard. Supabase's MCP includes tools for auth, storage and database.

What is it:

Cases: "show the last 10 orders with failed status", "suggest an index for this slow query", "create a migration to add email_verified column", "analyze distribution of users per plan".

Why learn:

Debugging live data on the terminal is extremely productive. Instead of opening pgAdmin or Supabase Dashboard, ask Claude directly.

Key concepts:

Debug: queries with WHERE and LIMIT. Migrations: based on the real schema. Analyze: GROUP BY, COUNT, AVG. Performance: EXPLAIN ANALYZE via Claude.

What is it:

Database MCP has moderate impact: ~1,500 tokens overhead. Large query results can add a lot of extra tokens, so use LIMIT and be specific.

Why learn:

Fixed and moderate overhead, but queries without LIMIT can explode the context. A SELECT * without LIMIT on a large table can consume the entire context window.

Key concepts:

Fixed overhead: ~1,500 tokens. Be careful with: queries without LIMIT, large tables, multiple JOINs. Always use LIMIT and select specific columns.

What is it:

NEVER connect to the production database with the admin user. Create a read-only user for exploration. Use staging/dev for mutations. Production = read only, ALWAYS.

Why learn:

A DELETE without WHERE executed by Claude in production can destroy customer data. Security with the bank via MCP is non-negotiable.

Key concepts:

Production: read-only user. Staging: user with write. Dev: can be admin. Never expose credentials in committed files. Use environment variables.

===== MODULE 6.7 =====
6.7 ~12 min

📊 MCP #5: Google Sheets

Connect Claude to Google Sheets to read, write, and analyze spreadsheet data directly from the terminal.

What is it:

Google Sheets MCP allows Claude to read and write in Google Sheets. It can query ranges, update cells, create new tabs and process tabular data.

Why learn:

Many teams use Google Sheets as a source of truth for non-technical data: lead lists, project plans, inventory. Connecting to Claude automates the interaction.

Key concepts:

Tools: read_spreadsheet, write_spreadsheet, get_sheets, create_sheet. Access via Google Sheets API with service account credentials.

What is it:

Install with: claude mcp add google-sheets -- npx -y @anthropic/mcp-google-sheets. Requires configuration of Google credentials (service account or OAuth).

Why learn:

Installing MCP is simple, but configuring Google credentials is the most laborious part. Prepare to create a project in the Google Cloud Console.

Key concepts:

1. Create project in Google Cloud Console. 2. Enable Sheets API. 3. Create service account. 4. Download credentials.json. 5. Configure path in the env var.

What is it:

You need a service account with access to spreadsheets. Share the spreadsheet with the service account email. export GOOGLE_APPLICATION_CREDENTIALS=/path/to/credentials.json.

Why learn:

Without correct credentials, the MCP connects but does not access anything. The most common mistake is forgetting to share the spreadsheet with the service account.

Key concepts:

Service account email: xxx@projeto.iam.gserviceaccount.com. Share spreadsheet with this email. Permission: Editor for read/write, Viewer for read-only.

What is it:

Cases: "read the Leads tab and filter companies in SP", "update the status of completed tasks", "create a weekly summary from the spreadsheet data", "populate the spreadsheet with bank data".

Why learn:

Spreadsheets are the universal data interface. Connecting Claude to Sheets allows a bridge between technical data (bank, API) and business data (spreadsheets, reports).

Key concepts:

Read: Extract and analyze existing data. Write: populate with processing results. Bridge: move data between database and spreadsheet via Claude.

What is it:

Google Sheets MCP has low-moderate overhead: ~800 tokens. The real cost comes from the data returned - large spreadsheets can inject thousands of tokens into context.

Why learn:

The fixed overhead is acceptable, but remember to use specific ranges rather than reading entire spreadsheets. "A1:D50" is much better than "all tab".

Key concepts:

~800 tokens fixed overhead. Data returned varies. Use ranges: Sheet1!A1:D50 instead of Sheet1. Limit columns and rows to the minimum necessary.

What is it:

Always specify ranges when querying spreadsheets. Instead of "read the entire spreadsheet", it says "read Leads!A1:E100". This controls how many tokens the data consumes.

Why learn:

A spreadsheet with 10,000 rows can consume the entire context window. Specific ranges keep usage predictable and controllable.

Key concepts:

Syntax: TabName!LineColumn:LineColumn. Example: Leads!A1:E100. First ask for the columns/headers to understand the structure, then consult specific data.

🔧 Managing MCPs

Master management commands: list, add, remove MCPs. Know which ones to install immediately and which ones to leave for when you need them.

What is it:

The claude mcp list command shows all configured MCPs, their status (connected/disconnected) and the tools that each one exposes. And /status is specific to MCPs.

Why learn:

Before adding a new MCP, check what is already active. Before debugging, confirm that the MCP is connected. It is the first step of any operation with MCPs.

Key concepts:

Rotate regularly to stay aware of what is consuming tokens. Disconnected MCPs appear as "disconnected" and do not consume tokens.

What is it:

claude mcp remove mcp-name disconnects and removes an MCP from the configuration. MCP stops consuming tokens immediately. You can re-add at any time.

Why learn:

Removing MCPs that you are not using is the most direct action to save tokens. And the "turn off the light when you leave the room" of the MCP world.

Key concepts:

Syntax: claude mcp remove browser. Immediate effect: stop sending schemas. You can re-add: claude mcp add browser -- [command]. No data loss.

What is it:

claude mcp add name -- server-command adds a new MCP. The name is its identifier, and the command after -- is what starts the MCP server.

Why learn:

Knowing the exact syntax of add avoids configuration errors. The -- (double dash) separates the MCP name from the server command - forgetting this is the most common error.

Key concepts:

Format: claude mcp add [name] -- [command]. Example: claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem ~/projetos. The name can be any valid string.

What is it:

Essentials (always on): Filesystem. Optional (on demand): GitHub, Browser, Database, Sheets. Sorting MCPs by need guides rotation decisions.

Why learn:

Without classification, operators treat all MCPs as essential and accumulate overhead. With sorting, you keep it to a minimum and activate on demand.

Key concepts:

Essential: constant benefit, low cost (Filesystem). Optional: one-off benefit, variable cost (Browser, GitHub). Never treat optional as essential.

What is it:

Install now: Filesystem (essential, always on) and GitHub (if you work with repos). These two cover 80% of use cases and have an acceptable combined cost (~2,500 tokens).

Why learn:

Having essential MCPs configured before you need them prevents workflow interruptions. Set it once, use it every time.

Key concepts:

Filesystem: claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem ~/projetos. GitHub: claude mcp add github -- npx -y @anthropic/mcp-github (requires GITHUB_TOKEN).

What is it:

Install when needed: Browser (only for UI/scraping), Database (only when working with data), Google Sheets (only when interacting with spreadsheets). Have the commands saved.

Why learn:

Having the installation commands ready in a reference file allows you to add MCPs in seconds when necessary. Get ready for speed.

Key concepts:

Save the commands in CLAUDE.md or a cheatsheet. When you need it: paste, run, use, remove. Complete cycle in less than 30 seconds.

===== MODULE 6.9 =====
6.9 ~12 min

🔄 Active Rotation Strategy

Learn how to rotate MCPs according to the phase of work: building, UI, backend, scraping and code review. The operator's mentality.

What is it:

When building features: keep Filesystem (always) + GitHub (for issues). Disconnect Browser and Database. Maximum focus on code with minimum overhead.

Why learn:

Building is the phase where you most need context for code. Every token saved on unnecessary MCPs goes straight to better implementation quality.

Key concepts:

Active MCPs: Filesystem + GitHub. Overhead: ~2,500 tokens. Free context: ~197,500 tokens for code. Workflow: read issue -> implement -> create PR.

What is it:

When working on UI: hold Filesystem (always) + Browser (to check visually). Disconnect GitHub and Database. Claude needs to see the result.

Why learn:

UI without visual verification and blind coding. MCP Browser allows Claude to see what you see, identify visual problems and suggest adjustments.

Key concepts:

Active MCPs: Filesystem + Browser. Overhead: ~5,500 tokens (high, but necessary). Workflow: code -> screenshot -> adjust -> screenshot -> approve.

What is it:

When working with backend/data: keep Filesystem (always) + Database (for queries and schema). Disconnect Browser and GitHub. Focus on logic and data.

Why learn:

Backend with access to the real database allows Claude to write validated queries, create migrations based on the current schema and debug data problems directly.

Key concepts:

Active MCPs: Filesystem + Database. Overhead: ~2,000 tokens. Workflow: explore schema -> write query -> validate result -> implement in code.

What is it:

For lead scraping: Browser (browse and extract) + Google Sheets (save results). Disconnect GitHub and Database. Optimized workflow for collecting data from the web.

Why learn:

Lead scraping is one of the most valuable workflows with MCPs. Browser collects, Sheets stores. Claude does in minutes what would take hours manually.

Key concepts:

Active MCPs: Browser + Sheets (+ Filesystem). Overhead: ~6,300 tokens (high). Workflow: browse directory -> extract data -> save to spreadsheet. Batch processing.

What is it:

For code review: Filesystem (read code) + GitHub (read PRs and comments). Claude reads the diff, understands the PR context and generates structured review comments.

Why learn:

Code review with full context (local code + remote PR) results in more accurate and useful reviews. Claude sees what has changed and what already existed.

Key concepts:

Active MCPs: Filesystem + GitHub. Overhead: ~2,500 tokens. Workflow: read PR -> read affected files -> generate review -> post comments. Review skill complements.

What is it:

Don't think "which MCPs do I have". Think "what phase of work am I in and what MCPs does that phase need". The phase dictates the configuration, not the other way around.

Why learn:

The operator mentality transforms MCP management from boring to automatic. You don't "manage MCPs", you "go into UI mode" or "go into backend mode".

Key concepts:

Modes: Building (FS+GH), UI (FS+Browser), Backend (FS+DB), Scraping (Browser+Sheets), Review (FS+GH). Switch modes, not individual MCPs.

===== MODULE 6.10 =====
6.10 ~12 min

🛠️ Troubleshooting

Solve the most common problems with MCPs: connection, permissions, high token, slowness and server errors.

What is it:

MCP appears as "disconnected" in /status. Most common cause: the server command is wrong or the package is not installed. Check with npx -y [package] --help.

Why learn:

Failed connection is the #1 problem with MCPs. Knowing how to diagnose quickly avoids wasting time thinking that the MCP doesn't work when it's just configuration.

Key concepts:

1. claude mcp list (confirm status). 2. claude mcp remove name. 3. Re-add with correct command. 4. Check dependencies (Node.js, npx). 5. Run /status.

What is it:

MCP connects but returns "permission denied" when trying to use tools. Cause: token/credential without necessary permissions or unauthorized path (Filesystem).

Why learn:

Permission denied is frustrating because it seems like everything is configured. The problem is usually with the scope of the token or the credentials file permissions.

Key concepts:

GitHub: Check PAT scopes. Filesystem: check allowed paths. Database: check SQL user permissions. Sheets: check sharing with service account.

What is it:

/cost shows consumption much higher than expected. Causes: many connected MCPs, queries that return large data, frequent screenshots of the MCP Browser.

Why learn:

High token usage degrades quality and costs money. Identifying the cause quickly allows you to correct it before it affects the entire session.

Key concepts:

Fix: 1. claude mcp list (see how many MCPs). 2. Remove non-essentials. 3. Use queries with LIMIT. 4. Avoid unnecessary screenshots. 5. /compact if necessary.

What is it:

Responses take much longer than normal. Causes: Slow MCP server (Browser with heavy pages), very full context, or MCP server with timeout.

Why learn:

Slowness can be due to the MCP, the context or the network. Diagnosing the correct cause avoids "fixes" that don't solve anything.

Key concepts:
