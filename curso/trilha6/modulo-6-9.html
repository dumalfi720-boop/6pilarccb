🏗️ Building Mode: Filesystem + GitHub

Main Concept

🎯Configuration to Build Features

When you're building new features, the priority is maximum context for code. Keep only Filesystem (cross-repo access) and GitHub (to read issues with requirements). Disconnect Browser and Database - you don't need web or live data to code.

  • • Active MCPs:Filesystem (~500) + GitHub (~2,000) = ~2,500 tokens overhead
  • • Free context:~197,500 tokens available for code and instructions
  • • Workflow:Read issue → Implement → Create PR → Link issue
  • • Optimization:Disconnect GitHub after reading the issue if the session will be long
Why It Matters

📊Why This Setup

  • Building needs maximum space for code and complex instructions
  • GitHub is only necessary to search for requirements (issues) and create PRs
  • Browser and Database are irrelevant to building pure code
Practical Tip

💡Practical Tip

Optimized workflow: 1) Connect GitHub. 2) 'Read issue #42 and summarize the requirements'. 3) Disconnect GitHub (mcp remove github). 4) Code with just Filesystem. 5) Reconnect GitHub. 6) 'Create PR with these changes by linking issue #42'. 7) Disconnect. Average overhead: much lower than keeping GitHub connected all the time.

Do vs Avoid

✓ What TO DO

  • ✓ Keep overhead below 3,000 tokens for building
  • ✓ Disconnect GitHub between reading issue and creating PR
  • ✓ Maximize context for code

✗ What NOT to do

  • ✗ Keep Browser on during coding
  • ✗ Keep Database connected unnecessarily
  • ✗ Connect MCPs 'just in case'

🎨 UI/Design Mode: Filesystem + Browser

Main Concept

🎯Configuration for Visual Work

When working on UI, Claude needs to SEE the result. MCP Browser is essential here but it is the most expensive. Accept the high overhead (~5,500 tokens) because visual verification is irreplaceable for UI.

  • • Active MCPs:Filesystem (~500) + Browser (~5,000) = ~5,500 tokens overhead
  • • Free context:~194,500 tokens - less but enough
  • • Workflow:Code → Screenshot → Adjustment → Screenshot → Approve
  • • Careful:Each screenshot consumes an extra 5,000-15,000 tokens!
Why It Matters

📊The Necessary Trade-off

  • UI without visual verification and blind coding
  • The high overhead of the Browser is justifiable here (single use case)
  • Minimize screenshots: ask for get_text when you don't need to SEE it visually
Practical Tip

💡Practical Tip

For UI, use an efficient verification flow: 1) Code the component. 2) 'Navigate to localhost:3000/page and take screenshot'. 3) 'Adjust header padding and font size'. 4) 'Take screenshot again to confirm'. Limit to 2-3 screenshots per cycle. More than that and waste tokens.

Do vs Avoid

✓ What TO DO

  • ✓ Accept high Browser overhead for UI tasks
  • ✓ Limit screenshots to 2-3 per tuning cycle
  • ✓ Use get_text for non-visual checks

✗ What NOT to do

  • ✗ Request screenshots for each minor change
  • ✗ Keep Browser on after UI phase
  • ✗ Do more than 3-4 screenshot cycles without pausing

⚙️ Backend/Data Mode: Filesystem + Database

Main Concept

🎯Configuration for Backend and Data

When working with backend and data, connect Database MCP for direct access to the database. Claude can explore schemas, validate queries, and debug data issues in real time.

  • • Active MCPs:Filesystem (~500) + Database (~1,500) = ~2,000 tokens overhead
  • • Free context:~198,000 tokens - excellent for backend code
  • • Workflow:Explore schema → Write query → Validate → Implement in code
  • • Careful:Use LIMIT in queries! Returned data consumes extra tokens
Why It Matters

📊Backend With Real Data

  • Claude writes SQL based on REAL schema, not guesswork
  • Live data debugging: see issues happening
  • More accurate migrations: based on the current state of the database
Practical Tip

💡Practical Tip

Start every backend session with: 'list all the database tables and describe the ones I will work on today: [table1, table2]'. This gives Claude real schema context at minimal cost. From then on, queries and migrations will be much more precise.

Do vs Avoid

✓ What TO DO

  • ✓ Get started with schema exploration
  • ✓ Use LIMIT in every query
  • ✓ Disconnect Database when finished with data

✗ What NOT to do

  • ✗ SELECT * without LIMIT on large tables
  • ✗ Connect Database during frontend coding
  • ✗ Running DDL without carefully reviewing

🕷️ Lead Scraping Mode: Browser + Sheets

Main Concept

🎯Configuration for Data Collection

For lead scraping and web data collection, combine Browser (browse and extract) with Google Sheets (store results). It is the most expensive setup but with high output value for sales and marketing teams.

  • • Active MCPs:Filesystem (~500) + Browser (~5,000) + Sheets (~800) = ~6,300 tokens overhead
  • • Free context:~193,700 tokens - tight but functional
  • • Workflow:Navigate directory → Extract data → Save to spreadsheet → Repeat
  • • Notice:This is the heaviest setup - only use when necessary
Why It Matters

📊High Cost, High Value

  • 6,300 tokens overhead is high, but the output justifies it
  • Automating collection of 50 leads can save 4-8 hours of manual work
  • The cost in tokens is insignificant compared to the value of the data collected
Practical Tip

💡Practical Tip

To maximize efficiency in scraping mode, process in batches: 'navigate to page 1 of the directory, extract the first 10 results, save to spreadsheet'. Then repeat for the next pages. This avoids accumulating too much data in the context at once and allows /compact between batches if necessary.

Do vs Avoid

✓ What TO DO

  • ✓ Process data in batches of 10-20
  • ✓ Save to spreadsheet after each batch
  • ✓ Use /compact between batches if context high

✗ What NOT to do

  • ✗ Trying to extract hundreds of results at once
  • ✗ Keep data in context without saving
  • ✗ Forgetting to disconnect Browser + Sheets after scraping

🔍 Code Review Mode: Filesystem + GitHub

Main Concept

🎯Configuration for Review

For code review, the setup is the same as building: Filesystem + GitHub. Claude reads the remote PR (GitHub MCP) and the local code (Filesystem) to generate contextualized reviews.

  • • Active MCPs:Filesystem (~500) + GitHub (~2,000) = ~2,500 tokens overhead
  • • Workflow:Read PR → Read affected files → Generate review → Post comments
  • • Combination:Use review skill (/review) + GitHub MCP for full reviews
  • • Advantage:Review with context of local code AND remote PR
Why It Matters

📊Review With Full Context

  • GitHub MCP: read the diff, comments and PR description
  • Filesystem: read complete files for context
  • Review skill: applies the team’s standard checklist
  • Result: more complete review than a common human would do
Practical Tip

💡Practical Tip

The ideal workflow: 1) 'Read PR #15 and its comments'. 2) 'Read modified files in my local project'. 3) 'Run /review with a focus on security and performance'. 4) 'Generate formatted comments to post in PR'. The combination of MCP + Skill produces senior quality reviews.

Do vs Avoid

✓ What TO DO

  • ✓ Combine GitHub MCP with Review Skill
  • ✓ Read local files for full context
  • ✓ Generate comments formatted for PR

✗ What NOT to do

  • ✗ Review only with PR diff (without local context)
  • ✗ Post comments without reviewing what Claude generated
  • ✗ Keep GitHub connected after the review

6

🧠 Operator Mentality

Main Concept

🎯Think in Modes, Not MCPs

Professional operators don't think 'which MCPs do I have'. They think 'which mode am I in and which MCPs does this mode need'. The work phase dictates the configuration, not the other way around. Switch modes and switch MCPs en bloc.

  • • Building:Filesystem + GitHub (~2,500 tokens)
  • • UI/Design:Filesystem + Browser (~5,500 tokens)
  • • Backend/Data:Filesystem + Database (~2,000 tokens)
  • • Lead Scraping:Filesystem + Browser + Sheets (~6,300 tokens)
  • • Code Review:Filesystem + GitHub (~2,500 tokens)
Why It Matters

📊The Change of Mindset

  • From: 'I'm going to add this MCP' → To: 'I'm going to enter UI mode'
  • From: 'which MCPs do I have connected?' → For: 'what stage of work am I in?'
  • From: manage individual MCPs → To: switch between pre-defined configurations
Practical Tip

💡Practical Tip

Create aliases in the shell for each mode: alias mode-build='claude mcp remove browser; claude mcp remove postgres; claude mcp add github -- npx -y @anthropic/mcp-github'. alias mode-ui='claude mcp remove github; claude mcp remove postgres; claude mcp add browser -- npx -y @anthropic/mcp-puppeteer'. Changing modes becomes a single command.

Do vs Avoid

✓ What TO DO

  • ✓ Think about work modes/phases
  • ✓ Create aliases or scripts for each mode
  • ✓ Switch mode when changing phase

✗ What NOT to do

  • ✗ Manage MCPs individually without strategy
  • ✗ Stay in the same mode when the work phase changed
  • ✗ Accumulate MCPs unrelated to the current task
