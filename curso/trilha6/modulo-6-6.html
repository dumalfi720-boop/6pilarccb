🗄️ What Database MCP Does

Main Concept

🎯Direct SQL in Claude

Database MCP connects Claude to PostgreSQL databases. It can explore schemas, execute SELECT queries, analyze data and suggest optimizations. With access to the real database, code and SQL suggestions are based on facts, not assumptions.

  • • execute_query:Run SQL directly on the connected database
  • • list_tables:List all database tables and views
  • • describe_table:Shows columns, types and constraints of a table
  • • get_schema:Returns the complete database schema
Why It Matters

📊From Assumption to Fact

  • Without MCP: Claude assumes the schema and may get column and type names wrong
  • With MCP: Claude sees the REAL schema and writes accurate SQL from the first try
  • Iteration savings: queries right on the first try instead of 3-4 corrections
Practical Tip

💡Practical Tip

The first command after connecting Database MCP must be: 'list all tables and describe the main ones'. This gives Claude an overview of the bank and allows him to write much more precise queries in subsequent requests.

Do vs Avoid

✓ What TO DO

  • ✓ Start with list_tables and describe_table for context
  • ✓ Use LIMIT in all queries
  • ✓ Connect to development/staging banks

✗ What NOT to do

  • ✗ Connect production with admin user
  • ✗ Run queries without LIMIT on large tables
  • ✗ Execute DDL (CREATE, ALTER, DROP) without reviewing

🐘 PostgreSQL Installation

Main Concept

🎯Direct Connection to Postgres

For generic PostgreSQL, the connection is made via the standard connection string. MCP supports any network-accessible PostgreSQL (local, cloud, Docker).

  • • Command: claude mcp add postgres -- npx -y @anthropic/mcp-postgres "postgresql://user:pass@host:5432/dbname"
  • • Location:postgresql://postgres:senha@localhost:5432/meu_banco
  • • Cloud:Use your provider's hostname (AWS RDS, GCP Cloud SQL, etc.)
  • • Docker:Use the container name or host.docker.internal
Why It Matters

📊Connection String Details

  • Format: postgresql://usuario:senha@host:porta/banco
  • For SSL (cloud): add ?sslmode=require to the end
  • Never commit the connection string - use environment variables
Practical Tip

💡Practical Tip

Create a specific READ-ONLY user for Claude: CREATE USER claude_reader WITH PASSWORD 'xxx'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO claude_reader; This ensures that Claude can never accidentally modify data, even in development.

Do vs Avoid

✓ What TO DO

  • ✓ Create read-only user for Claude
  • ✓ Use environment variables for connection string
  • ✓ Test connection before relying on it

✗ What NOT to do

  • ✗ Use admin/superuser user
  • ✗ Commit connection strings
  • ✗ Connect production without read-only

⚡ Supabase installation

Main Concept

🎯Specific Setup for Supabase

Supabase has a dedicated MCP that goes beyond pure SQL, offering access to auth, storage and Supabase-specific functionality.

  • • Command: claude mcp add supabase -- npx -y @supabase/mcp-server-supabase --supabase-url https://xxx.supabase.co --supabase-key sua-service-role-key
  • • URL:Find it in Supabase Dashboard → Settings → API → Project URL
  • • Key:Use service_role key (not anon key) for full access
  • • Extra:Supabase MCP includes tools for auth and storage in addition to database
Why It Matters

📊Supabase vs Generic PostgreSQL

  • Supabase MCP: access to auth, storage, realtime + database
  • PostgreSQL MCP: database only (pure SQL)
  • Use Supabase MCP if you use Supabase; PostgreSQL MCP for other Postgres
Practical Tip

💡Practical Tip

The Supabase service_role key has FULL access to the bank, bypassing RLS. Treat it with the same care as an admin password. Never expose it in the frontend, never commit it, always use it via the environment variable: export SUPABASE_SERVICE_KEY=eyJxxx.

Do vs Avoid

✓ What TO DO

  • ✓ Use service_role key for full access via MCP
  • ✓ Store key in environment variable
  • ✓ Use Dashboard to find URL and key

✗ What NOT to do

  • ✗ Use anon key (access limited by RLS)
  • ✗ Expose service_role key in code
  • ✗ Share key with other devs (each one creates their own)

💼 Use Cases

Main Concept

🎯Workflows with Database

Database MCP is most useful when you are debugging data problems, exploring unknown schemas or creating migrations. He brings the truth from the bank to Claude's context.

  • • Debug:'show the last 10 orders with failed status and their error_messages'
  • • Exploration:'describe all tables and identify relationships between them'
  • • Migration:'create a migration to add verified_at column to the users table'
  • • Performance:'run EXPLAIN ANALYZE on this query and suggest optimizations'
Why It Matters

📊Impact in Different Scenarios

  • Debug: Find data issues in seconds vs minutes with pgAdmin
  • Exploration: quickly maps unknown banks (onboarding)
  • Migration: generates DDL based on the REAL schema, not on assumptions
Practical Tip

💡Practical Tip

Powerful workflow for debugging: 'describe the orders table, show the last 5 orders with failed status, and analyze the error pattern'. Claude sees the schema, searches for real data and identifies patterns - all in one interaction. Without MCP, you would need 3-4 different tools.

Do vs Avoid

✓ What TO DO

  • ✓ Use for quick debugging of data problems
  • ✓ Explore unknown bank schemes
  • ✓ Generate migrations based on real state

✗ What NOT to do

  • ✗ Run heavy queries in production
  • ✗ Do INSERT/UPDATE/DELETE without reviewing
  • ✗ Blindly trust generated migrations (always review)

🟡 Token Impact: Moderate (~1,500 tokens)

Main Concept

🎯Controllable Cost

The fixed overhead of Database MCP is moderate (~1,500 tokens), but the real cost can be much higher depending on the data returned by the queries. A query without LIMIT on a large table can inject tens of thousands of tokens into the context.

  • • Fixed overhead: ~1,500 tokens per message
  • • Simple Query (10 rows): +500-1,000 result tokens
  • • Query without LIMIT (1000+ rows): +10,000-50,000+ result tokens!
  • • Full scheme: +2,000-5,000 tokens depending on bank
Why It Matters

📊The Danger of Unlimited Queries

  • SELECT * FROM users (no LIMIT): can return thousands of rows
  • Each row occupies ~50-100 tokens depending on the columns
  • 1,000 rows x 80 tokens = 80,000 tokens consumed at once
Practical Tip

💡Practical Tip

ALWAYS include LIMIT in your orders: 'show the last 10 orders' instead of 'show the orders'. If you need to analyze a lot of data, ask for summaries: 'count how many orders per status' instead of 'list all orders'. Aggregations (COUNT, AVG, SUM) are much more token efficient.

Do vs Avoid

✓ What TO DO

  • ✓ Always use LIMIT in queries
  • ✓ Prefer aggregations (COUNT, AVG) over SELECT *
  • ✓ Select specific columns instead of *

✗ What NOT to do

  • ✗ NEVER SELECT * without LIMIT on large tables
  • ✗ NEVER ask for 'all data' from the table
  • ✗ NEVER ignore the cost of returned data

🛡️ Production Safety

Main Concept

🎯Non-negotiable Rules

Connecting Claude to a production bank and how to give the keys to the safe. A DELETE without WHERE, an accidental DROP TABLE or a bulk UPDATE can destroy customer data. Security rules here are not suggestions - they are mandatory.

  • • PRODUCTION:ALWAYS read-only user. ZERO writing permissions
  • • STAGING:User with write allowed, but without DROP/TRUNCATE
  • • DEVELOPMENT:Can be admin for agility
  • • CREDENTIALS:NEVER in committed files. ALWAYS environment variables
Why It Matters

📊Real Disaster Scenarios

  • Claude runs 'DELETE FROM users WHERE active = false' - deletes real data
  • Claude runs 'ALTER TABLE' which breaks production schema
  • Connection string committed to Git - exposed in public repo
Practical Tip

💡Practical Tip

Create this user for production and NEVER use another: CREATE USER claude_prod_reader WITH PASSWORD 'xxx'; GRANT CONNECT ON DATABASE prod TO claude_prod_reader; GRANT USAGE ON SCHEMA public TO claude_prod_reader; GRANT SELECT ON ALL TABLES IN SCHEMA public TO claude_prod_reader; Test by trying INSERT - should fail.

Do vs Avoid

✓ What TO DO

  • ✓ Create dedicated read-only user for Claude
  • ✓ Test that the user cannot write
  • ✓ Use staging/dev for write operations

✗ What NOT to do

  • ✗ NEVER connect production with admin
  • ✗ NEVER allow writing in production via MCP
  • ✗ NEVER commit connection strings
