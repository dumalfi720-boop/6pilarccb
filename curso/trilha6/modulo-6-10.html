🔌 MCP Does Not Connect

Main Concept

🎯Connection Diagnostics

The most common problem with MCPs: you add it but the status shows 'disconnected'. The most frequent causes are: wrong server command, npm package not found, or missing dependency (Node.js, Chrome).

  • • Step 1: claude mcp list- confirm status
  • • Step 2: claude mcp remove nome- remove the problematic MCP
  • • Step 3:Test the isolated command:npx -y @anthropic/mcp-xxx --help
  • • Step 4:Re-add with the correct command and check with /status
Why It Matters

📊Most Common Causes

  • Wrong package name: check npm registry
  • Node.js not installed: check with 'node --version'
  • npx not available: 'npm install -g npx'
  • No internet: npx needs to download the package at the first time
Practical Tip

💡Practical Tip

The fix that solves 80% of connection problems: remove + re-add. If MCP was running before and stopped, a simple 'claude mcp remove name' followed by 'claude mcp add name -- command' resets everything. And the equivalent of 'turn off and on again' for MCPs.

Do vs Avoid

✓ What TO DO

  • ✓ Test the server command in isolation first
  • ✓ Use remove + re-add as the first fix
  • ✓ Check dependencies (Node.js, Chrome)

✗ What NOT to do

  • ✗ Assuming the problem is with the MCP without diagnosing it
  • ✗ Try using disconnected MCP tools
  • ✗ Ignore installation errors

🚫 Permission Denied

Main Concept

🎯Permission Errors

MCP connects successfully but returns 'permission denied' or 'access denied' when using tools. This usually indicates that the token/credential does not have the necessary scopes or the path is not authorized.

  • • GitHub:Check PAT scopes (needs repo, read:org)
  • • Filesystem:Check if the requested path is among the authorized paths
  • • Database:Check SQL user permissions (GRANT SELECT)
  • • Sheets:Check whether the spreadsheet has been shared with the service account
Why It Matters

📊Diagnosis By MCP

  • GitHub: github.com → Settings → Developer settings → check token scopes
  • Filesystem: claude mcp remove + re-add with correct paths
  • Database: connect manually and test user permissions
  • Sheets: open the spreadsheet and check sharing
Practical Tip

💡Practical Tip

For each MCP, create a simple permission test: GitHub → 'list 1 issue from repo'. Filesystem → 'list files in ~/projects'. Database → 'SELECT 1'. Sheets → 'read spreadsheet cell A1'. If the test passes, the permission is OK. If it fails, the problem is with your credentials.

Do vs Avoid

✓ What TO DO

  • ✓ Create simple permission tests for each MCP
  • ✓ Check token and permission scopes
  • ✓ Resolve permissions BEFORE trying to use MCP

✗ What NOT to do

  • ✗ Assume that connection = permission (they are different things)
  • ✗ Using tokens with excessive permissions to 'resolve'
  • ✗ Ignore permission errors and try again

📈 High Token Usage

Main Concept

🎯Excessive Context Consumption

/cost shows consumption much higher than expected. The most common causes are: many MCPs connected simultaneously, queries that return large volumes of data, or frequent screenshots of the MCP Browser.

  • • Diagnosis 1: claude mcp list- how many MCPs are active?
  • • Diagnosis 2:Add the overheads: Filesystem(500) + GitHub(2000) + Browser(5000) = 7500/msg
  • • Fix 1:Remove MCPs not essential for the current task
  • • Fix 2:Use LIMIT in queries and avoid unnecessary screenshots
Why It Matters

📊Quick Diagnostic Table

  • 1 active MCP but high token → large returned data (use LIMIT)
  • 3+ active MCPs → accumulated overhead (remove non-essential)
  • Active browser + screenshots → each screenshot costs 5k-15k tokens
  • Everything normal but high token → long session, run /compact
Practical Tip

💡Practical Tip

Diagnostic workflow in 30 seconds: 1) /cost - see the number. 2) claude mcp list - count active MCPs. 3) Calculate theoretical overhead. 4) If overhead > 30% of the context used, remove MCPs. 5) If overhead is low but token is high, and the session is long - run /compact.

Do vs Avoid

✓ What TO DO

  • ✓ Diagnose the cause before taking action
  • ✓ Remove non-essential MCPs as first fix
  • ✓ Use /compact for long sessions

✗ What NOT to do

  • ✗ Blame the model when token is high
  • ✗ Ignore the cumulative overhead of MCPs
  • ✗ Continue using without investigating the cause

🐌 Slow Responses

Main Concept

🎯Diagnosing Latency

Responses take much longer than normal. It can be caused by: slow MCP server, very full context, or external server with timeout.

  • • Diagnosis 1: /cost- context is full? (>80% = slow)
  • • Diagnosis 2:Testing without MCPs - still slow? And the model/network
  • • Diagnosis 3:Test each individual MCP - which one is slow?
  • • Fix:/compact if context high, restart MCP if from server, wait if from network
Why It Matters

📊Causes By Priority

  • Most likely: full context (>80%) → Fix: /compact
  • Second: MCP server stuck → Fix: remove + re-add
  • Third: slow network → Fix: wait or change network
  • Fourth: external server (GitHub API, DB) slow → Fix: try again later
Practical Tip

💡Practical Tip

Quick test to isolate the cause: disconnect all MCPs and send a simple message. If it responds quickly, the problem is one of the MCPs. Reconnect one by one until you find the slow one. If it's still slow without MCPs, run /cost - probably full context.

Do vs Avoid

✓ What TO DO

  • ✓ Isolate the cause: context vs MCP vs network
  • ✓ Test without MCPs for baseline
  • ✓ Use /compact as first fix for full context

✗ What NOT to do

  • ✗ Assume that it is 'Claude who is slow'
  • ✗ Send more messages while it's slow
  • ✗ Ignore it and hope it resolves on its own

❌ Server Not Found

Main Concept

🎯MCP Server Not Found

Error adding MCP: 'server not found', 'command not found' or 'module not found'. This indicates that the npm package does not exist, the name is wrong, or the Node.js environment is not configured.

  • • Fix 1:Check package name:npm info @anthropic/mcp-xxx
  • • Fix 2:Confirm Node.js:node --version(needs v18+)
  • • Fix 3:Confirm npx:npx --version
  • • Fix 4:Isolated test:npx -y @anthropic/mcp-xxx --help
Why It Matters

📊Common Causes

  • Package name changed (MCP ecosystem evolves quickly)
  • Outdated Node.js (needs v18+ for most MCPs)
  • npx not installed or not in PATH
  • Firewall/proxy blocking npm package download
Practical Tip

💡Practical Tip

If the package is not found, it may be that the name has changed. Search npm: npx -y npm search @anthropic/mcp to see all of Anthropic's official MCP packages. For third-party MCPs, check the official MCP documentation on GitHub.

Do vs Avoid

✓ What TO DO

  • ✓ Check exact package name in npm
  • ✓ Keep Node.js up to date (v18+)
  • ✓ Test isolated npx before adding to Claude

✗ What NOT to do

  • ✗ Make up package names
  • ✗ Use Node.js < v18
  • ✗ Ignore npx errors

📜 Operator Checklist

Main Concept

🎯Universal troubleshooting

When something doesn't work with MCPs, follow this checklist in order. It solves 95% of problems without the need for external research.

  • • 1. /status- MCP appears? Is it connected?
  • • 2. claude mcp list- Is the configuration correct?
  • • 3. claude mcp remove nome + re-add- Reset connection
  • • 4. /doctor- General diagnosis of Claude Code
  • • 5.Restart the Claude Code session completely
  • • 6.Check dependencies (Node.js, Chrome, credentials)
Why It Matters

📊The Three Golden Rules

  • Rule #1: Always start with /status - 50% of problems are seen here
  • Rule #2: Remove + re-add solves 80% of remaining problems
  • Rule #3: If nothing works, restart the session completely
Practical Tip

💡Practical Tip

Save this checklist as an alias in the shell: alias mcp-fix='echo "1. /status" && echo "2. claude mcp list" && echo "3. claude mcp remove NAME + re-add" && echo "4. /doctor" && echo "5. Restart session" && echo "6. Check deps"'. When something breaks, run 'mcp-fix' to remember the steps in the right order.

Do vs Avoid

✓ What TO DO

  • ✓ Follow the checklist IN THE ORDER presented
  • ✓ Stop when the problem is solved (don't run them all if the 1st solves it)
  • ✓ Document recurring problems and their solutions

✗ What NOT to do

  • ✗ Jump straight to advanced solutions
  • ✗ Trying random fixes without diagnosing
  • ✗ Reinstall Claude Code as first resort
