🧭 Decision Framework: Skill vs MCP

🔁 Repeatable = Skill

Main Concept

🎯The Principle of Repeatability

If a task follows a repetitive and predictable pattern - such as code review, component creation, test generation - it is a perfect candidate for Skill. Skills are Markdown instructions that cost ZERO in token tax because they are loaded once and do not add fixed overhead per message.

  • • Code review with standard checklist -> Skill (/review)
  • • Create React component with team pattern -> Skill (/create-component)
  • • Generate unit tests following convention -> Skill (/generate-tests)
Why It Matters

📊Why Skill is Best for Standards

  • Zero token tax: Skills are loaded on demand, not in every message
  • Faster: does not depend on external server or API
  • More reliable: no connection, timeout or authentication problems
Practical Tip

💡Practical Tip

Before creating an MCP, ask: 'does this task use data that is already in my project?' If yes, and Skill. Claude already has access to local files via its native tools (Read, Write, Bash). MCP is only necessary when the data is OUTSIDE the project.

Do vs Avoid

✓ What TO DO

  • ✓ Use Skills for repetitive tasks with a fixed pattern
  • ✓ Convert long prompts you retype into Skills
  • ✓ Prefer Skills when possible as they cost zero tokens

✗ What NOT to do

  • ✗ Create MCPs for tasks that Skills solve
  • ✗ Ignore Skills because 'MCPs seem more powerful'
  • ✗ Retyping the same complex prompts instead of creating a Skill

📡 Live Data = MCP

Main Concept

🎯The Live Data Principle

If a task needs real-time data from an external service - querying a database, reading GitHub issues, accessing an updated spreadsheet, browsing a website - it REQUIRES MCP. Skills do not access APIs and Claude will manufacture data if you ask for it without an MCP connected.

  • • Consult the user table in the database -> MCP Database
  • • Read GitHub issue #42 with comments -> MCP GitHub
  • • Check if the deployment is visually correct -> MCP Browser
Why It Matters

📊The Danger of Not Using MCP When Needed

  • Without MCP, Claude INVENTS data - seems confident but is hallucinating
  • Fabricated data can lead to incorrect implementations based on assumptions
  • Claude won't warn you that he doesn't have access - he simply generates something plausible
Practical Tip

💡Practical Tip

If you asked for data and Claude responded without using any MCP tool, BE DISTRUSTABLE. Check if the MCP is connected with /status. If Claude is 'inventing' database data or issues that do not exist, it is because the MCP is not active and he is hallucinating.

Do vs Avoid

✓ What TO DO

  • ✓ Use MCP whenever you need live external data
  • ✓ Check with /status that the MCP is connected before requesting data
  • ✓ Distrust data that Claude 'knows' without using MCP tools

✗ What NOT to do

  • ✗ Request database data without Database MCP connected
  • ✗ Trust 'GitHub issues' without active GitHub MCP
  • ✗ Assume that Claude has access to the internet without MCP Browser
TOPIC 3
3

🎨 Formatting = Skill

Main Concept

🎯Standardization and Templates

Tasks that involve formatting, standardization, boilerplate generation or application of templates are 100% Skills domain. These are instructions that shape Claude's OUTPUT, not fetch data. No MCP is needed to define how the code should be formatted.

  • • Team commit message pattern -> Skill
  • • React/Vue/Svelte component template -> Skill
  • • PR description format with checklist -> Skill
Why It Matters

📊Skills Are Perfect for Formatting

  • Templates are pure text - perfect for Markdown (Skills format)
  • Zero external dependency - works offline, without API, without token
  • Easily versionable - commit to Git along with the code
Practical Tip

💡Practical Tip

Create a folder .claude/commands/templates/ with formatting skills: commit.md, pr.md, component.md, test.md. This standardizes the entire team's output and costs zero extra tokens. And the investment with the highest ROI in the Claude Code ecosystem.

Do vs Avoid

✓ What TO DO

  • ✓ Create Skills for each team formatting standard
  • ✓ Versioning Skills in Git for the entire team to use
  • ✓ Use Skills to enforce code style and conventions

✗ What NOT to do

  • ✗ Create MCPs to generate templates (total overkill)
  • ✗ Retype formatting patterns in each session
  • ✗ Use long prompts when a 10-line Skill resolves
TOPIC 4
4

🕷️ Scraping = MCP

Main Concept

🎯External Data Collection

Web scraping, collecting website data, checking URLs, extracting information from pages - all this requires Browser MCP. Claude does not have native internet access. Without MCP, he simply cannot access any website.

  • • Extract prices from an e-commerce -> MCP Browser
  • • Collect names from a corporate directory -> MCP Browser
  • • Check if your deploy rendered correctly -> MCP Browser
Why It Matters

📊MCP Browser is Irreplaceable for Scraping

  • The only MCP that gives Claude 'eyes' - he sees the page as a real user
  • Navigates JavaScript-rendered pages (SPA, React apps) correctly
  • You can click, fill out forms and interact like a human
Practical Tip

💡Practical Tip

For efficient scraping, always tell Claude exactly what to extract: 'navigate to url.com, extract company name and contact email from page'. Vague requests like 'see what's on this site' result in huge screenshots that consume thousands of tokens with no proportional value.

Do vs Avoid

✓ What TO DO

  • ✓ Use MCP Browser for any web access
  • ✓ Be specific about what data to extract
  • ✓ Disconnect Browser after finishing scraping

✗ What NOT to do

  • ✗ Try scraping without MCP Browser (Claude invented data)
  • ✗ Ask for 'see all' instead of specific data
  • ✗ Leave MCP Browser turned on after the scraping task
TOPIC 5
5

⚡ Efficiency = Skill

Main Concept

🎯When Tokens Matter

In long sessions, plans with limited usage or tasks that require many iterations, each token saved multiplies. Skills cost ZERO in fixed overhead while MCPs cost hundreds to thousands per message. When efficiency is a priority, prefer Skills.

  • • Skill in 50 messages: 0 extra tokens = 0 total overhead
  • • MCP with 2,000 tokens overhead in 50 messages: 100,000 extra tokens
  • • The difference (100k tokens) is available context for quality code and instructions
Why It Matters

📊The Multiplier Effect

  • Token tax is PER MESSAGE - multiplies by the total number of messages in the session
  • An average session has 30-50 messages - 2k overhead turns into 60k-100k tokens
  • On plans with a monthly limit, this can represent 5-10% of the entire budget
Practical Tip

💡Practical Tip

Before connecting an MCP, ask: 'how many messages will this session have?' If it is a fast session (5-10 msgs), the overhead of MCPs is tolerable. If it's a long session (30-50+ msgs), evaluate whether it's worth the cumulative cost. Manually copying data to the session is often more efficient than keeping an MCP powered on.

Do vs Avoid

✓ What TO DO

  • ✓ Calculate total overhead: tokens_per_msg x estimated_messages
  • ✓ Prefer Skills for long sessions
  • ✓ Disconnect MCPs when entering pure coding phase

✗ What NOT to do

  • ✗ Ignore the multiplier effect in long sessions
  • ✗ Maintain MCPs 'just in case' in sessions of 50+ messages
  • ✗ Underestimating the impact of 2,000 tokens per message

📋 Complete Decision Table

Main Concept

🎯Quick Reference: Skill vs MCP

This table covers the most common scenarios. The general rule is simple: if you need live external data, and MCP. If it is instruction, pattern or template, it is Skill. When in doubt, start with Skill - you can always add MCP later if necessary.

  • • Code review→ Skill |Read GitHub issues→MCP
  • • Generate component→ Skill |Consult bank→MCP
  • • Format commit→ Skill |Browse website→MCP
  • • PR Template→ Skill |Read spreadsheet→MCP
  • • Deployment checklist→ Skill |Verify deployment visually→MCP
Why It Matters

📊The Ultimate Question

  • Ask: 'Do you need live external data?' YES = MCP, NO = Skill
  • This single question solves 90% of decision cases
  • The remaining 10% are hybrid: use Skill for instruction + MCP for data
Practical Tip

💡Practical Tip

Save this table in your CLAUDE.md as a permanent reference. When in doubt, consult the table. Over time, the decision becomes automatic and you stop thinking 'Skill or MCP?' - you just know. That is the mark of an experienced operator.

Do vs Avoid

✓ What TO DO

  • ✓ Save decision table in CLAUDE.md
  • ✓ Follow the rule 'live data = MCP, rest = Skill'
  • ✓ Start with Skill and escalate to MCP if necessary

✗ What NOT to do

  • ✗ Complicating the decision more than necessary
  • ✗ Use MCP when Skill resolves
  • ✗ Forgetting that Skills are always more efficient in tokens
