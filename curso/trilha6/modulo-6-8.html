📋 claude mcp list

Main Concept

🎯See All Active MCPs

The commandclaude mcp listshows all configured MCPs, their connection status and the tools that each one exposes. It is the equivalent of /status but specific to MCPs.

  • • List name of each configured MCP
  • • Shows status: connected or disconnected
  • • Displays the available tools of each active MCP
  • • Run before adding new MCPs to avoid duplicates
Why It Matters

📊When to Use

  • Before adding: check if the MCP is already configured
  • Troubleshooting: confirm that the MCP is connected
  • Audit: see how many MCPs are consuming tokens
Practical Tip

💡Practical Tip

Make 'claude mcp list' the first command when something doesn't work with MCPs. In 80% of cases, the problem is that the MCP is 'disconnected' and needs to be re-added. The list shows this immediately.

Do vs Avoid

✓ What TO DO

  • ✓ Run before adding new MCPs
  • ✓ Use as a first troubleshooting step
  • ✓ Periodically check how many MCPs are active

✗ What NOT to do

  • ✗ Assuming MCPs are connected without checking
  • ✗ Add duplicate MCPs
  • ✗ Ignore MCPs with disconnected status

2

🗑️ claude mcp remove

Main Concept

🎯Disconnect Unnecessary MCPs

The commandclaude mcp remove nome-do-mcpdisconnects and removes an MCP from the configuration. The effect is immediate: the MCP stops consuming tokens in the next message. You can re-add at any time.

  • • Syntax:claude mcp remove browser
  • • Immediate effect: overhead tokens stop being charged
  • • Reversible: you can re-add with 'claude mcp add' at any time
  • • No data loss: removing does not erase system settings or credentials
Why It Matters

📊The Most Direct Action in Economics

  • Removes 5,000 tokens/msg? Disconnect Browser
  • Remove 2000 tokens/msg? Disconnect GitHub when not using it
  • The savings are multiplied by each message remaining in the session
Practical Tip

💡Practical Tip

Make a habit of 'cleaning MCPs' before long coding sessions. Run 'claude mcp list', identify MCPs that you will not use, and remove all except Filesystem. This is like closing browser tabs you're not using - it frees up resources for what matters.

Do vs Avoid

✓ What TO DO

  • ✓ Remove MCPs that you will not use in the next phase
  • ✓ Create a habit of cleaning before long sessions
  • ✓ Use remove as an optimization tool

✗ What NOT to do

  • ✗ Keep MCPs 'for later' by consuming tokens
  • ✗ Being afraid to remove (you can re-add)
  • ✗ Ignore the cumulative impact of idle MCPs
TOPIC 3
3

➕ claude mcp add

Main Concept

🎯Add New MCPs

The commandclaude mcp add nome -- comando-do-serveradds a new MCP. The name is how you reference the MCP, and the command after -- is what starts the MCP server.

  • • Format:claude mcp add [nome] -- [comando]
  • • The -- (double dash) separates the name from the command - DON'T forget it!
  • • Name can be any string: filesystem, github, my-bank
  • • Server arguments go after the command: npx -y @pkg /path
Why It Matters

📊Common Mistakes

  • Forget the --: 'claude mcp add github npx...' doesn't work
  • Duplicate name: check with 'mcp list' before add
  • Wrong command: test 'npx -y @pkg --help' isolated first
Practical Tip

💡Practical Tip

Save your MCP add commands in a quick reference file (cheatsheet.txt or CLAUDE.md). When you need to add an MCP, copy and paste instead of typing from scratch. This eliminates typing errors and speeds up MCP rotation to less than 10 seconds.

Do vs Avoid

✓ What TO DO

  • ✓ Saving add commands to a cheatsheet
  • ✓ Test the server command in isolation before
  • ✓ Check with /status after adding

✗ What NOT to do

  • ✗ Forget the -- between name and command
  • ✗ Add without checking for duplicates
  • ✗ Enter long memory commands
TOPIC 4
4

⭐ Essential vs Optional

Main Concept

🎯Classification by Need

Classifying MCPs between essential (always connected) and optional (on demand) is the basis of the management strategy. Essentials have low overhead and constant benefit. Options have variable overhead and specific benefits.

  • • ESSENTIAL (always on):Filesystem (~500 tokens) - constant benefit, minimum cost
  • • OPTIONAL (on demand):GitHub (~2,000 tokens) - connect when working on issues/PRs
  • • OPTIONAL (on demand):Database (~1,500 tokens) - connect when working with data
  • • OPTIONAL (on demand):Browser (~5,000 tokens) - connect ONLY for web tasks
  • • OPTIONAL (on demand):Google Sheets (~800 tokens) - connect when interacting with spreadsheets
Why It Matters

📊Why Classify

  • Unclassified: all MCPs seem essential and you accumulate overhead
  • With classification: 1 fixed MCP + 1-2 rotating = controlled overhead
  • Decisions become automatic: 'do I need data from the bank? Connect Database MCP'
Practical Tip

💡Practical Tip

Simple rule: if the MCP costs less than 1,000 tokens AND you use it in >50% of sessions, it is essential (keep always on). If it costs more than 1,000 tokens OR you use it in <50% of sessions, it is optional (connect on demand). Currently, only Filesystem passes the 'essential' test.

Do vs Avoid

✓ What TO DO

  • ✓ Classify each MCP as essential or optional
  • ✓ Keep only essentials always on
  • ✓ Have add commands ready for optionals

✗ What NOT to do

  • ✗ Treat everyone as essential
  • ✗ Keep options turned on 'just in case'
  • ✗ Add new MCPs without sorting first
TOPIC 5
5

🚀 Install Immediately

Main Concept

🎯MCPs to Configure Now

These are the MCPs you should install and configure now, before you need them. Having the setup ready before you need it avoids interruptions in the workflow.

  • • Filesystem (required): claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem ~/projetos
  • • GitHub (if you use GitHub): claude mcp add github -- npx -y @anthropic/mcp-github(requires GITHUB_TOKEN)
  • • Combined overhead: ~2,500 tokens - acceptable for any session
  • • These two cover 80% of MCPs use cases
Why It Matters

📊Why Now

  • Configuring it in advance avoids interruptions when you really need it
  • Filesystem is essential for any type of project
  • GitHub is essential for any dev who uses GitHub (most)
Practical Tip

💡Practical Tip

Run these commands now and never think about it again: 1) claude mcp add filesystem -- npx -y @anthropic/mcp-filesystem ~/projetos 2) export GITHUB_TOKEN=your_token (add to .bashrc) 3) claude mcp add github -- npx -y @anthropic/mcp-github. That's it, 80% of your MCP setup is done.

Do vs Avoid

✓ What TO DO

  • ✓ Install Filesystem now if you don't already have it
  • ✓ Configure GitHub if working with repos
  • ✓ Add GITHUB_TOKEN to shell profile

✗ What NOT to do

  • ✗ Postpone installation 'for when you need it'
  • ✗ Install all MCPs at once (start with these 2)
  • ✗ Skip setting environment variables
TOPIC 6
6

⏰ Install When You Need

Main Concept

🎯MCPs On Demand

These MCPs should be installed only when the task requires it. Keep the commands ready on a cheatsheet to install in seconds when necessary.

  • • Browser: claude mcp add browser -- npx -y @anthropic/mcp-puppeteer→ Use for UI/scraping
  • • Database: claude mcp add postgres -- npx -y @anthropic/mcp-postgres "conn_string"→ Use for data
  • • Supabase: claude mcp add supabase -- npx -y @supabase/mcp-server-supabase --supabase-url URL --supabase-key KEY
  • • Sheets: claude mcp add google-sheets -- npx -y @anthropic/mcp-google-sheets→ Use for spreadsheets
Why It Matters

📊The Add-Use-Remove Cycle

  • Add: paste cheatsheet command (5 seconds)
  • Use: do the task that needs MCP
  • Remove: claude mcp remove name (5 seconds)
  • Complete cycle in less than 30 seconds of overhead
Practical Tip

💡Practical Tip

Create a file ~/mcp-cheatsheet.txt with all add commands ready for each MCP. When you need an MCP, open the file, copy the command and paste. This reduces rotational friction to almost zero and makes the active rotation strategy (module 6.9) practical in everyday life.

Do vs Avoid

✓ What TO DO

  • ✓ Maintain cheatsheet with add commands ready
  • ✓ Practice the add-use-remove cycle until it becomes automatic
  • ✓ Remove immediately after use

✗ What NOT to do

  • ✗ Trying to remember commands from memory
  • ✗ Install on demand without having credentials configured
  • ✗ Forgetting to remove after use
