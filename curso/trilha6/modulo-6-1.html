🏷️ What is Token Tax

Main Concept

🎯Main Concept

Token tax is the cost in tokens that each connected MCP adds to EVERY interaction with Claude, even when you are not actively using that MCP. When an MCP is connected, its complete schema (tool definitions, parameters, descriptions) is automatically injected into the context of each message sent to Claude.

  • • Each MCP exposes tools with descriptions and parameters that are serialized as input tokens
  • • These tokens are sent in EVERY message, not just when you use MCP
  • • 5 connected MCPs can consume 10,000+ fixed tokens per message, before any code or instructions
Why It Matters

📊Why This Matters

  • Token tax is invisible - you don't see the schemas being sent, but they consume context
  • Operators who ignore token tax unknowingly miss 10-30% of context
  • The quality of responses degrades proportionally to the context consumed by overhead
Practical Tip

💡Practical Tip

Do a test: run /cost at the beginning of a session without MCPs, send 5 messages. Then add 3-4 MCPs and send the same 5 messages. Compare token consumption. The difference is the actual token tax of your MCPs. This number should guide your decisions on which MCPs to keep connected.

Do vs Avoid

✓ What TO DO

  • ✓ Measure the real token tax of your MCPs with /cost
  • ✓ Disconnect MCPs you are not actively using
  • ✓ Prioritize MCPs with low overhead (Filesystem) over high (Browser)

✗ What NOT to do

  • ✗ Connect all MCPs 'just in case'
  • ✗ Ignoring the impact of token tax on quality
  • ✗ Assuming that MCPs are 'free' just because the connection is easy

📊 Overhead Table by MCP

Main Concept

🎯Token Tax Table

Each MCP has a different fixed cost based on the number of tools it exposes and the complexity of the schemas. This table shows the approximate overhead per message for the most common MCPs:

  • • Filesystem:~500 tokens (LOW) - 3-4 simple tools, lower ecosystem overhead
  • • Google Sheets:~800 tokens (LOW-MODERATE) - 4-5 tools with range parameters
  • • Database/Supabase:~1,500 tokens (MODERATE) - 5-8 tools with SQL schemas
  • • GitHub:~2,000 tokens (MODERATE) - 10+ tools with many parameters
  • • Browser/Chrome:~5,000+ tokens (HIGH) - 10+ complex tools with screenshots
Why It Matters

📊Cumulative Impact

  • Filesystem + GitHub + Browser = ~7,500 sticky tokens per message
  • In 50 messages, that and 375,000 extra tokens - almost 2x the context window
  • Each MCP removed saves your overhead multiplied by the number of messages in the session
Practical Tip

💡Practical Tip

Paste this table into your CLAUDE.md as a quick reference. When deciding which MCPs to connect, look at the numbers and calculate the total overhead. Rule of thumb: if the total overhead exceeds 5,000 tokens, you are already noticeably sacrificing quality.

Do vs Avoid

✓ What TO DO

  • ✓ Memorize approximate overheads
  • ✓ Calculate total overhead before connecting MCPs
  • ✓ Prioritize Low MCPs over High MCPs

✗ What NOT to do

  • ✗ Connect 4+ MCPs simultaneously without calculating
  • ✗ Ignore the difference between 500 and 5000 tokens
  • ✗ Treat all MCPs as having the same cost

🫁 Context as Oxygen

Main Concept

🎯The Oxygen Tank Metaphor

Imagine Claude's context as an oxygen tank inside an astronaut helmet. The tank has 200,000 liters (tokens). Everything Claude needs to 'think' - his code, instructions, history, CLAUDE.md - consumes oxygen from this tank. MCPs are extra equipment connected to the same tank.

  • • Full tank (200k tokens) = Claude thinks with maximum clarity, detailed and precise answers
  • • Each MCP is like a piece of equipment sucking oxygen: Filesystem = thin tube, Browser = thick hose
  • • When the tank is 70%+ consumed, Claude starts to 'choke': shorter answers, less detailed, more errors
Why It Matters

📊Why This Metaphor Works

  • Make something abstract tangible - tokens become oxygen that you can visualize
  • Naturally generates caution - no one would waste oxygen on equipment they are not using
  • Creates correct urgency - disconnecting unnecessary MCPs becomes 'closing the valve' of idle equipment
Practical Tip

💡Practical Tip

Before each session, ask: 'how much oxygen will each piece of equipment consume?' If the task is purely code, do not connect Browser (thick hose). If it is a backend without GitHub, do not connect GitHub MCP. Every liter of oxygen saved in overhead goes towards quality of thought.

Do vs Avoid

✓ What TO DO

  • ✓ View context as a finite and precious resource
  • ✓ Question each MCP: 'do I need this equipment now?'
  • ✓ Disconnect MCPs = close valves on idle equipment

✗ What NOT to do

  • ✗ Treat context as infinite
  • ✗ Keep MCPs 'just in case' as if they were free
  • ✗ Wait for quality degradation to act

📏 Rule of Thumb: Max 2-3 MCPs

Main Concept

🎯The practical limit

Based on the average overhead of popular MCPs, the rule of thumb is: keep a maximum of 2-3 MCPs connected simultaneously. With 2-3 MCPs, the total overhead is between 1000-4000 tokens - acceptable for most tasks. With 4+, the overhead can exceed 8,000 tokens and degradation becomes noticeable.

  • • 1 MCP (ideal):500-2000 tokens overhead, minimal impact on quality
  • • 2-3 MCPs (acceptable):1,500-5,000 tokens, balance between functionality and quality
  • • 4-5 MCPs (excessive):5,000-10,000+ tokens, noticeable degradation in complex responses
  • • 6+ MCPs (wasteful):10,000++ tokens, seriously compromised quality
Why It Matters

📊Why 2-3 and the Sweet Spot

  • Covers 95% of workflows: Filesystem (always) + 1-2 MCPs specific to the task
  • Maximum overhead of ~5,000 tokens = just 2.5% of the 200k context
  • Allows quick rotation: removing 1 and adding another takes seconds
Practical Tip

💡Practical Tip

Treat the 2-3 MCP rule as a hard limit, not a suggestion. If you need 4 MCPs for a task, divide the task into phases and rotate: phase 1 with MCPs A+B, phase 2 with MCPs C+D. Never connect everything at the same time 'for convenience'.

Do vs Avoid

✓ What TO DO

  • ✓ Respect the limit of 2-3 simultaneous MCPs
  • ✓ Rotate MCPs by work phase
  • ✓ Keep Filesystem as '1 fixed MCP' and rotate the others

✗ What NOT to do

  • ✗ Connect 4+ MCPs simultaneously
  • ✗ Justifying excess with 'but I need everyone'
  • ✗ Sacrificing quality for convenience

📈 Monitoring with /cost

Main Concept

🎯Measuring Real Impact

The /cost command is your main tool for monitoring token tax in practice. It shows tokens consumed per message, accumulated session cost and percentage of context used. Use before and after connecting MCPs to see the direct impact.

  • • Rotate/costat the beginning of the session for baseline
  • • After connecting MCPs, run again and compare the tokens per message
  • • The difference between the two measurements is the actual token tax of the connected MCPs
Why It Matters

📊Monitoring and Power

  • Without medication, token tax and theory. With /cost, it is a concrete number that you optimize
  • Enables data-driven decisions: 'MCP browser costs 5k tokens but saved 30 minutes of manual QA'
  • Creates feedback loop: connect -> measure -> optimize -> measure again
Practical Tip

💡Practical Tip

Make a habit of running /cost every 15-20 messages when using MCPs. If the context is above 60% AND you have MCPs connected, consider whether you can disconnect any before running /compact. Disconnecting a 2000 token MCP before compacting is how to free up space in the oxygen tank.

Do vs Avoid

✓ What TO DO

  • ✓ Run /cost regularly with active MCPs
  • ✓ Compare sessions with and without MCPs
  • ✓ Using /cost data to justify MCP decisions

✗ What NOT to do

  • ✗ Ignore /cost when using MCPs
  • ✗ Assume that MCPs do not affect consumption
  • ✗ Expect problems to start monitoring

🎯 Impact on Response Quality

Main Concept

🎯The Degradation Curve

The quality of Claude's responses follows a predictable curve based on the use of context. With connected MCPs, you get into degradation lanes faster because the fixed overhead consumes context from the beginning.

  • • 0-50% context:Excellent answers - detailed, precise, with examples. Claude at full potential
  • • 50-70% context:Good answers - still accurate but less detailed. Start considering /compact
  • • 70-90% context:Visible degradation - shorter answers, fewer examples, possible omissions
  • • 90%+ context:Severe degradation - frequent errors, garbled responses, lost context between messages
Why It Matters

📊MCPs Accelerate the Curve

  • Without MCPs: you reach 50% of the context after ~40-60 messages
  • With 3 MCPs (~5k overhead): you reach 50% after ~30-40 messages
  • With 5+ MCPs (~10k+ overhead): you reach 50% after ~20-30 messages - almost half of normal
Practical Tip

💡Practical Tip

When you notice that Claude's answers are becoming shorter or less precise, run /cost immediately. If the context is above 70%, the likely cause is saturated context - not 'the model went dumb'. The solution: /compact + remove unnecessary MCPs. You will be surprised by the improvement.

Do vs Avoid

✓ What TO DO

  • ✓ Correlate response quality with use of context
  • ✓ Use /cost when responses appear degraded
  • ✓ Compact and remove MCPs when context > 70%

✗ What NOT to do

  • ✗ Blaming the model when quality drops (generally and context)
  • ✗ Ignore signs of degradation
  • ✗ Keep asking for complex tasks with context > 80%
