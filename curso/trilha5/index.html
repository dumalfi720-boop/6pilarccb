🔀 Skills vs MCPs vs Plugins

Understand the fundamental differences between Skills, MCPs and Plugins. Learn when to use each one and how they complement each other in the Claude Code ecosystem.

What is it:

Skills are Markdown files that contain structured instructions for Claude to perform specific tasks. They live in the .claude/commands/ folder and are invoked as slash commands.

Why learn:

Skills turn long, repetitive prompts into one-line commands. Instead of retyping complex instructions, you write once and reuse infinitely.

Key concepts:

A skill is basically a saved prompt with structure. It defines Purpose, Inputs, Steps and Quality Checks. Claude reads the markdown and runs it as if you had entered everything manually.

What is it:

MCPs (Model Context Protocol) are servers that connect Claude to external tools and services: databases, APIs, remote file systems, browsers and more.

Why learn:

MCPs expand what Claude can do. Without MCPs, it only reads and writes local files. With MCPs, it queries databases, surfs the web, manages infrastructure.

Key concepts:

MCPs are separate processes that run alongside Claude. They expose tools via standardized protocol. They are configured in .claude/settings.json or settings.local.json.

What is it:

Plugins are packages that combine skills, configurations and even MCPs into a single installable package. You install it once and get a full set of features.

Why learn:

Plugins are the future of the Claude Code ecosystem. Instead of configuring everything manually, you install a plugin and gain professional skills ready to use.

Key concepts:

A plugin can contain multiple skills, CLAUDE.md configs and bundled MCPs. Distributed via marketplace or Git repositories. Install with one command, use immediately.

What is it:

Direct comparison: Skills are Markdown with instructions (zero infra), MCPs are servers with tools (requires a running process), Plugins are distributable packages (combine everything).

Why learn:

The confusion between these three concepts is the #1 mistake of beginners. Knowing when to use each one avoids unnecessary complexity and ensures you choose the right tool.

Key concepts:

Skills: light, pure text, anyone can create them. MCPs: powerful, require setup, connect services. Plugins: packaged, easy installation, combines skills + MCPs.

What is it:

Practical decision guide: use Skill when you want to automate a prompt. Use MCP when you need to connect to an external service. Use Plugin when you want to distribute or install ready-made features.

Why learn:

Choosing the wrong one means waste. Create an MCP when a skill resolves and over-engineering. Create a skill when you need a database and it is insufficient.

Key concepts:

Simple rule: if it is prompt instruction, Skill. If you need an API/external service, MCP. If you want to share/distribute, Plugin. Always start with the simplest.

What is it:

Decision tree: Do you need an external connection? -> MCP. And reusable instruction? -> Skill. Do you want to distribute to others? -> Plugins. Do you need everything together? -> Plugin with MCP.

Why learn:

Having a mental framework avoids analysis paralysis. When a need arises, you follow the flowchart and arrive at the answer in seconds.

Key concepts:

KISS principle applied: start with Skill (simplest), scale to MCP (when needed), package in Plugin (when you want to distribute). Never start with the most complex.

📁 Where Skills Live

Discover the .claude/commands/ directory, how auto-discovery works, and how to organize skills between Claude Code and Claude Cowork.

What is it:

The .claude/commands/ directory is at the root of the project and where Claude Code searches for custom skills. Each .md file inside becomes a slash command automatically.

Why learn:

Without knowing where to put it, you can't create skills. It is the mandatory starting point for any automation with skills.

Key concepts:

Create the directory: mkdir -p .claude/commands/. Each .md file inside becomes /filename. Subfolders create namespaces: .claude/commands/review/security.md becomes /review/security.

What is it:

Claude Code automatically scans the .claude/commands/ directory when starting a session. New .md files are detected without the need for reloading or configuration.

Why learn:

Knowing that discovery is automatic means you can create skills on-the-fly. Create the file, and it will be available the next time you type /.

Key concepts:

The scan takes place at the beginning of the session. If you create a skill during the session, you may need to start a new sailing session. /help lists all discovered skills.

What is it:

Each skill becomes a slash command: /filename. You type / and Claude shows all available skills. Select and execute with an Enter.

Why learn:

Slash commands are the interface between you and your skills. It's the fastest way to invoke complex instructions without typing anything other than /name.

Key concepts:

File review.md becomes /review. File screenshot-to-site.md becomes /screenshot-to-site. Use descriptive names and kebab-case. Arguments are passed after the command.

What is it:

Claude Cowork (web interface) comes with built-in skills that do not exist in Claude Code CLI. These are visual skills such as creating dashboards, slides and formatted documents.

Why learn:

Knowing Cowork's native skills avoids recreating what already exists. Some tasks are better performed in Cowork than in the CLI.

Key concepts:

Cowork has pre-built visual skills. Some are exclusive to the web interface. The challenge is synchronizing skills between Code and Cowork (module 5.9 addresses this).

What is it:

The plugin marketplace is a centralized repository where you can find community-created skills and plugins, ready to install and use.

Why learn:

Don't reinvent the wheel. Before creating a skill from scratch, check if someone has already created something similar. The marketplace accelerates adoption and shares best practices.

Key concepts:

SkillsMP, SkillHub.club, and awesome-claude-code are popular sources. Plugins are evaluated by the community. Always review the code before installing (security in module 5.10).

What is it:

The recommended organization: .claude/commands/ for project skills, ~/.claude/commands/ for global skills. Subfolders by category: review/, generate/, analyze/.

Why learn:

Scale organization. With 2 skills, any structure works. With 20, you need categories. Planning the structure early avoids mess later.

Key concepts:

Project: .claude/commands/review.md. Global: ~/.claude/commands/review.md. Categories: .claude/commands/review/security.md, .claude/commands/review/performance.md. Git: version the project's skills.

📝 Skill Template

The anatomy of a well-written skill: Purpose, Inputs, Steps, Quality Checks and Output. Learn to write instructions that Claude follows precisely.

What is it:

The Purpose section defines in a clear sentence what the skill does. It is the north that guides the entire execution and prevents Claude from deviating from his objective.

Why learn:

Without a clear Purpose, Claude interprets the skill in different ways with each execution. A well-written Purpose guarantees consistency and predictability.

Key concepts:

Format: "# Purpose\nTransform [input] into [output] following [criteria]." Be specific. "Do code review" is vague. "Identify security vulnerabilities in Python code" is accurate.

What is it:

The Inputs section lists everything the skill needs to work: files, variables, context, parameters. Defines what is mandatory and what is optional.

Why learn:

Poorly defined inputs lead to incomplete executions. Claude asks for missing information or assumes wrong values. Clear inputs = clean execution.

Key concepts:

Use $ARGUMENTS to capture user input. List inputs with type: "- Screenshot (PNG/JPG image)", "- Technological Stack (string, optional)". Set defaults for optionals.

What is it:

The Steps section defines the sequence of actions that Claude must follow. Each step is a clear, actionable instruction, numbered in order of execution.

Why learn:

Steps are the heart of the skill. Without them, Claude decides what to do on his own. With clear Steps, you control exactly the sequence of actions.

Key concepts:

Use numbering: 1. Analyze the input, 2. Identify patterns, 3. Generate output. Each step must be atomic (one action). Include decisions: "If X, do Y. Otherwise, do Z."

What is it:

The Quality Checks section defines criteria that the output must meet before being delivered. Claude self-validates the result against these criteria.

Why learn:

Quality Checks transform good skills into excellent skills. Claude reviews his own work and corrects problems before handing it in. And automatic self-review.

Key concepts:

Checklist: "- [ ] Output contains all required sections", "- [ ] No generic placeholders remaining", "- [ ] Consistent formatting". Claude checks off each item.

What is it:

The Output section defines exactly how the result should be formatted: file, format, structure, name. Eliminates ambiguity about what the skill produces.

Why learn:

Without defined Output, Claude decides the format. Sometimes it generates Markdown, sometimes plain text, sometimes JSON. Defining the output ensures consistency.

Key concepts:

Specify: format (JSON, CSV, HTML), structure (required fields), destination (file, stdout, clipboard). Example: "Output: leads.csv file with Name, Email, Rating columns."

What is it:

Writing techniques to maximize the effectiveness of instructions: imperative verbs, specificity, concrete examples and explicit restrictions.

Why learn:

The quality of the skill is proportional to the quality of the instructions. Vague instructions generate inconsistent results. Precise instructions generate predictable results.

Key concepts:

Use imperative verbs: "Analyze", "Identify", "Generate". Avoid ambiguity: "good code" vs "code without warnings, with typed types, coverage > 80%". Include examples of expected output.

📸 Skill: Screenshot to Website

Practice skill that transforms design screenshots into functional HTML/Tailwind code. Layout analysis, semantic HTML and comparison loop.

What is it:

Skill that receives a design screenshot (Figma, Dribbble, existing website) and generates HTML + Tailwind CSS code that faithfully replicates the look.

Why learn:

Turning design into code is one of the most repetitive tasks in web development. This skill automates 80% of the work, leaving you to focus on the final details.

Key concepts:

Input: screenshot PNG/JPG. Output: HTML file with Tailwind. Claude analyzes the layout, colors, typography and spacing of the screenshot and generates the equivalent code.

What is it:

The first step of the skill: analyze the screenshot and identify grid, sections, components, visual hierarchy, color palette and typography used.

Why learn:

Without structured analysis, Claude generates code that looks right but doesn't scale. Systematic analysis ensures that the HTML structure reflects the design logic.

Key concepts:

Claude identifies: header, hero, features, footer. Maps grid: 12 columns, flexbox, CSS grid. Extracts colors in hex/tailwind. Identifies fonts and sizes.

What is it:

The skill instructs Claude to use semantic tags (nav, main, section, article, aside, footer) instead of generic divs. Accessibility and SEO for free.

Why learn:

Semantic HTML and professional quality requirement. The skill ensures that the generated code is not "div soup" but structured and accessible HTML.

Key concepts:

nav for navigation, main for main content, section for sections, article for independent content. Alt text on images. Aria labels where necessary.

What is it:

The skill generates code using Tailwind CSS utility classes instead of custom CSS. Result: maintainable code, responsive and consistent with the design system.

Why learn:

Tailwind is the industry standard for rapid prototyping. The code generated by the skill is immediately usable in any Tailwind project.

Key concepts:

Responsive classes: sm:, md:, lg:. Spacing: p-4, m-2, gap-6. Layout: flex, grid, grid-cols-3. The skill maps the visual design to equivalent Tailwind classes.

What is it:

The skill includes a self-review loop: Claude compares the generated result with the original screenshot, identifies differences and refines the code until it reaches high fidelity.

Why learn:

The first draft is never perfect. The comparison loop ensures that Claude iterates automatically, correcting spacing, colors and proportions without you asking.

Key concepts:

Self-review step: "Compare the generated HTML with the screenshot. Identify 3 differences. Correct and repeat until fidelity score > 90%." Maximum of 3 iterations.

What is it:

Final checklist: Valid HTML, responsive (mobile-first), semantic, accessible, no inline CSS, pure Tailwind, images with alt text, functional links.

Why learn:

The checklist ensures that no quality aspect is forgotten. And the difference between prototype code and production code.

Key concepts:

Checklist: responsiveness (3 breakpoints), accessibility (alt, aria), performance (no unnecessary libraries), consistency (design system), validity (W3C validator).

🔍 Skill: Lead Research

Lead research skill: niche confirmation, research methodology, Hot/Warm/Cold classification, deduplication and structured CSV output.

What is it:

Skill that searches and organizes potential leads for a specific niche. Generates classified list with contact information, company and estimated level of interest.

Why learn:

Manually searching for leads takes hours. This skill systematizes the process: defines criteria, research, classifies and formats. Consistent results in minutes.

Key concepts:

Input: target niche/industry. Output: CSV with classified leads. Claude researches, validates and organizes leads following a structured methodology.

What is it:

First step: confirm with the user the exact niche, target personas, company size and geography. Avoid searching for wrong leads.

Why learn:

Search without a defined niche generates irrelevant leads. Initial confirmation ensures that all research effort is directed correctly.

Key concepts:

Confirmation questions: "Which industry?", "Company size (SME, Enterprise)?", "Geography (Brazil, LATAM, Global)?", "Decision maker's role?" Without this, you won't move forward.

What is it:

The methodology defines where and how to look for leads: LinkedIn, company websites, industry directories, events, sector publications. Systematic process with prioritized sources.

Why learn:

Without methodology, research is random and incomplete. With methodology, you guarantee broad coverage and uniform quality of the data collected.

Key concepts:

Primary sources: LinkedIn, Crunchbase, corporate websites. Secondary: events, publications, associations. For each lead: name, position, company, email, telephone, LinkedIn.

What is it:

Classification system: Hot (clear signs of interest/need), Warm (compatible profile, no active signals), Cold (demographic match only).

Why learn:

Not all leads are the same. Sorting by temperature allows you to prioritize outreach and allocate time to leads most likely to convert.

Key concepts:

Hot: published about the problem, hiring for the area, mentioned the topic. Warm: company with the right profile, right position. Cold: right industry but no signs of timing.

What is it:

Cleaning step: identify and remove duplicate leads based on email, name+company or LinkedIn URL. Keep only the most complete record.

Why learn:

Duplicate leads generate double contact, which is unprofessional and wastes time. Deduplication guarantees a clean and reliable list.

Key concepts:

Criteria: identical email, identical name+company, identical LinkedIn URL. Merge: keep the record with more fields filled in. Report how many duplicates were removed.

What is it:

The output is a CSV with standardized columns: Name, Position, Company, Email, Telephone, LinkedIn, Classification, Notes. Ready to import into CRM.

Why learn:

CSV structured and importable directly into HubSpot, Salesforce, Pipedrive. No manual formatting. The skill generates the file ready for use.

Key concepts:

Mandatory columns: Name, Company, Classification. Optional: Email, Telephone, LinkedIn, Position, Notes. UTF-8 encoding. Comma separator. Header on the first line.

🔒 Skill: Senior Code Review

Professional review skill in 4 layers: security, bugs, performance and quality. Output with severity levels and positive feedback.

What is it:

First layer of the review: search for SQL injection, XSS, exposed secrets, weak authentication, CORS misconfiguration, vulnerable dependencies.

Why learn:

Security is the most critical layer. A single SQL injection can compromise the entire database. The security review should always be the first.

Key concepts:

Checklist: sanitized inputs, parameterized queries, secure tokens, HTTPS enforced, security headers, secrets in env vars (never hardcoded).

What is it:

Second layer: search for race conditions, null pointer exceptions, off-by-one errors, resource leaks, missing error handling, untreated edge cases.

Why learn:

Bugs in production cost 10x more than bugs found in review. The skill systematizes the search for known bug patterns that human eyes often miss.

Key concepts:

Patterns: missing null checks, async without await, try without proper catch, potential infinite loops, division by zero, array out of bounds.

What is it:

Third layer: N+1 queries, unnecessary loops, cache misses, missing indexes, O(n2) algorithms when O(n) is possible, excessive bundle size.

Why learn:

Poor performance kills user experience. The skill identifies performance problems that only appear under load, before reaching production.

Key concepts:

Patterns: queries in loops, lack of pagination, unnecessary renders (React), heavy imports, lack of lazy loading, absent memoization.

What is it:

Fourth layer: naming conventions, DRY violations, overlong functions, excessive coupling, lack of typing, missing tests, inline documentation.

Why learn:

Code that works but is unreadable becomes technical debt. The quality layer ensures that the code is maintainable in the long term, not just today.

Key concepts:

Principles: functions < 30 lines, descriptive names, single responsibility, comments explain "why" not "what", strong typing, test coverage.

What is it:

The output classifies each finding as Critical (blocks deployment), Important (must fix), Minor (nice to have). Includes location, description and fixing suggestion.

Why learn:

Severity allows you to prioritize fixes. Criticals first, importants later, minors when there is time. Without classification, everything seems equally urgent.

Key concepts:

Format: [CRITICAL] file:line - description - fix suggestion. [IMPORTANT] file:line - description. [MINOR] file:line - description. Summary at the end with count.

What is it:

The skill includes a positive feedback section: highlighting what was done well. Good patterns, elegant abstractions, robust error handling, complete tests.

Why learn:

Reviews are only negative and demotivating. Including positive feedback reinforces good practices and makes the review more constructive. Balance between criticism and recognition.

Key concepts:

“Positive Highlights” section: list 3-5 things well done. Examples: "Good separation of concerns", "Complete error handling", "Tests cover edge cases". Always include.

🔌 Plugins: The Game-Changer

Claude Code's plugin system: what they are, why they matter, examples such as Supabase plugin, marketplace and the "install once, always use" paradigm.

What is it:

Plugins are packages that package skills, CLAUDE.md configs, MCPs and scripts into a single installable. They are the evolution of individual skills into complete solutions.

Why learn:

Plugins drastically simplify setup. Instead of configuring 5 skills + 2 MCPs + CLAUDE.md manually, you install a plugin and everything works.

Key concepts:

A plugin = bundle of resources. May contain: skills (.md), config (CLAUDE.md patches), MCPs (bundled servers), scripts (setup/teardown). Versioned as software.

What is it:

Plugins democratize advanced capabilities. One expert creates it, thousands use it. And the same model that made npm, VS Code extensions and WordPress plugins explode.

Why learn:

The plugin ecosystem will define who is productive and who is left behind. Those who adopt early will have a massive competitive advantage in productivity.

Key concepts:

Network effect: more plugins = more value. Specialization: vertical plugins for each industry/stack. Composability: combine plugins for complex workflows.

What is it:

The Supabase plugin adds skills to manage database, auth, storage and edge functions directly from Claude Code. Real example of a powerful plugin.

Why learn:

And the perfect case study: a plugin that combines MCP (connection to Supabase) with skills (create tables, configure auth, deploy functions). Shows the power of the model.

Key concepts:

Install once, access: table creation, migrations, RLS configuration, edge functions deployment, storage management. All via slash commands.

What is it:

The plugin marketplace centralizes discovery, installation and updating. Similar to npm but for Claude Code features.

Why learn:

The marketplace is where you find plugins tested and trusted by the community. It's also where you publish your own plugins for others to use.

Key concepts:

Browse, install, update. Categories: development, database, deployment, testing, documentation. Each plugin has a README, rating and number of installations.

What is it:

The fundamental principle: configure once, use on every project. Global plugins are located in ~/.claude/ and work in any repository automatically.

Why learn:

Repeating setup on each project is wasteful. Global plugins ensure that your favorite tools are always available, in any context.

Key concepts:

Global: ~/.claude/commands/ (available in all projects). Location: .claude/commands/ (project specific). Combine both for maximum coverage.

What is it:

The plugin ecosystem is growing exponentially. New plugins emerge daily, covering more and more use cases and technology stacks.

Why learn:

Those who contribute early gain visibility and influence. Creating popular plugins establishes a reputation in the community and can become a source of income.

Key concepts:

🔄 Synchronizing Skills Code and Cowork

The challenge of keeping skills synchronized between Claude Code (CLI) and Claude Cowork (web). Three practical options and the skill-creator tool.

What is it:

Claude Code (CLI) and Claude Cowork (web) have separate skill systems. A skill created in Code does not automatically appear in Cowork and vice versa.

Why learn:

Many users use both interfaces. Without sync, you maintain two versions of each skill, which creates inconsistency and wasted effort.

Key concepts:

Code skills: .claude/commands/ (local files). Cowork skills: configured in the web interface. There is no native automatic synchronization between them.

What is it:

Configure Cowork to read your project's .claude/commands/ folder. This way, both interfaces read from the same location, maintaining natural sync.

Why learn:

It is the simplest and most direct solution. Does not require extra tools. Works out of the box for file-based skills.

Key concepts:

In Cowork, select the project folder. The skills in .claude/commands/ are automatically available. Limitation: only works with the selected project.

What is it:

Package your skills as a plugin and install it in both Code and Cowork. The plugin ensures that both have the same version of the skills.

Why learn:

Plugins are the most robust solution for sync. An update to the plugin updates both interfaces. Versioning included.

Key concepts:

Create a repository for the plugin. Structure: /commands/, /config/, README.md. Install on Code and Cowork. Update via git pull or marketplace.

What is it:

Recreate skills directly in the Cowork interface. More laborious option but allows you to take advantage of exclusive Cowork features such as interactive UI.

Why learn:

Some skills work better in Cowork than in Code. Visual, interactive skills or skills that require UI benefit from specific rebuilds.

Key concepts:

Copy the logic, adapt the format. Cowork has specific fields for inputs, steps and outputs. Maintain the essence of the skill, adapt the interface.

What is it:

Tool that automates the creation of skills based on a description in natural language. You describe what you want and it generates the .md in the correct format.

Why learn:

Writing skills from scratch requires knowledge of the format. skill-creator abstracts this, allowing anyone to create professional skills quickly.

Key concepts:

Input: "I want a skill that performs security reviews on Python APIs." Output: .md file complete with Purpose, Steps, Quality Checks and Output. Ready to use.

What is it:

Set of good practices: version skills in Git, use naming conventions, document each skill, test before publishing, maintain changelog.

Why learn:

Disorganized skills become tech debt. Good practices from the beginning ensure that your skills catalog scales without becoming a mess.

Key concepts:

Git: version everything in .claude/commands/. Naming: descriptive kebab-case. Docs: each skill has a header with description. Testing: test with real data before committing.

===== MODULE 5.10 =====
5.10 ~15 min

🛡️ Skills Security

Security risks when installing third-party skills: access to filesystem, shell and environment. How to check hidden instructions and review plugins.

What is it:

Skills can instruct Claude to read, modify or delete files on your system. A malicious skill can access sensitive files outside the project.

Why learn:

Blindly trusting third-party skills and running unknown scripts. You need to understand the risks to make informed decisions about what to install.

Key concepts:

Risks: reading ~/.ssh/, ~/.aws/credentials, .env files. Writing: modifying configs, creating backdoors. Mitigation: sandbox, restricted permissions, code review.

What is it:

Skills can instruct Claude to execute shell commands. A malicious skill can run scripts that install malware, send data or change system settings.

Why learn:

Shell access is the most serious risk. With bash, Claude can do anything you can do in the terminal. Including destructive things.

Key concepts:

Risks: curl | bash from malicious URLs, rm -rf, data exfiltration via curl/wget. Mitigation: use --allowedTools to restrict, review each command before approving.

What is it:

Skills can access environment variables that contain API keys, authentication tokens, database credentials and other secrets.

Why learn:

Environment variables are the default location for secrets. A skill that reads $DATABASE_URL or $OPENAI_KEY can exfiltrate credentials without you noticing.

Key concepts:

Risks: echo $API_KEY, env | curl -X POST. Mitigation: use .env files with restricted permissions, do not export unnecessary secrets, monitor network requests.

What is it:

Rule number 1: always read the raw .md file before installing. Skills are pure text, you can read and understand everything they instruct Claude to do.

Why learn:

It's the only way to guarantee that the skill does what it says. Trusting the marketplace description without reading the code is the equivalent of installing npm packages without auditing.

Key concepts:

Open the .md raw. Look for: shell instructions, access to files outside the project, network requests, reading env vars. If something looks suspicious, don't install it.

What is it:

Prompt injection via skills: instructions hidden in HTML comments, invisible unicode or formatting that seems harmless but instructs Claude to do something different.

Why learn:

Sophisticated attacks use prompt injection techniques into skills. Knowing the attack vectors allows you to identify them before they cause damage.

Key concepts:

Vectors: HTML comments with instructions, zero-width unicode characters, "ignore previous instructions" hidden in long text. Use editor with special character visualization.

What is it:

Security checklist to evaluate plugins before installing: trusted source, audited code, no suspicious network requests, minimum permissions, active community.

Why learn:

A systematic evaluation framework avoids impulse decisions. Having a checklist ensures that you do not forget any safety aspects.

Key concepts:

Checklist: reliable source? Open source? Positive reviews? No suspicious network calls? Minimum permissions? Actively maintained? If any item fails, do not install.

💰 Math Token: 90% Savings

The math behind the token economy with skills. Cost comparison, ROI calculation and scaling strategy to maximize savings.

What is it:

Direct comparison: typing a 500-word prompt costs ~700 input tokens. Invoking a skill with /review costs ~20 input tokens (the content of the skill is processed internally).

Why learn:

Each token costs money. Re-typing the same prompt 100 times costs 100x. Invoking a skill 100 times costs almost the same as 1 time. The economy is dramatic.

Key concepts:
