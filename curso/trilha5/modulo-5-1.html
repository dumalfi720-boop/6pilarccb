MODULE 5.1

💙 Skills vs MCPs vs Plugins

Understand the three ways to extend Claude Code: Skills (Markdown instructions), MCPs (external tools via protocol) and Plugins (installable packages). Each one has its place, and confusing them is the most common mistake made by those just starting out.

📝 What are Skills

Skills areinstructions in Markdownthat you save in the folder.claude/commands/of your project. They work like custom slash commands: you type/project:nome-da-skilland Claude follows the instructions step by step. It's not code, it's not an API, it's not a plugin. And pure text that guides the agent's behavior.

🎯Main Concept

One Skill and onestructured and reusable prompt. Instead of typing the same instructions every time you need a code review, you create a skill that defines exactly what to check, in what order, and how to format the output. Claude reads Markdown and runs it as if you had typed everything manually.

  • • Format:.md file with instructions in natural language
  • • Activation:Via slash command /project:name or /user:name
  • • Cost:Zero installation, zero dependencies, zero configuration

💡Practical Tip

Think of Skills as "cake recipes" for Claude. Each skill has ingredients (inputs), steps (instructions) and expected result (output). The beauty is that anyone on the team can use the same skill and get consistent results, regardless of how well they know how to prompt.

TOPIC 2
2

🔌 What are MCPs

MCPs (Model Context Protocol) areexternal serverswhich give Claude access to tools that he doesn't have natively: databases, APIs, remote file systems, browsers, etc. Unlike Skills, MCPs involve running code, background processes and communication via a standardized protocol.

🎯Main Concept

One MCP and oneserver that exposes toolsfor Claude to use. When you configure Supabase's MCP, for example, Claude gains access to tools such as "query_database", "create_table", "insert_row". He calls out these tools during the conversation as if they were native.

  • • Format:Node/Python server that implements the MCP protocol
  • • Activation:Configured in .claude/settings.json or via claude mcp add
  • • Cost:Requires server installation, configuration and maintenance

💡Practical Tip

MCPs are for when Claude needs theminteract with the external world. If the task can be solved with text instructions alone (code review, refactoring, analysis), use a Skill. If you need to connect to Slack, a database, or an external API, then you need an MCP.

TOPIC 3
3

📦 What are Plugins

Plugins areinstallable packagesthat combine Skills, configurations and possibly MCPs in a single bundle. Think of them as "apps" for Claude Code: you install it once and get a full set of features. Plugins can come from a marketplace or be created by you.

🎯Main Concept

A Plugin packages multiple Skills (and optionally MCPs) into something that can beinstalled, updated and sharedwith a single command. It's natural evolution: you start with a single skill, it grows, you add more, and eventually package everything as a plugin.

  • • Format:Repository or package with standardized structure
  • • Activation:Installed via marketplace or manually via git clone
  • • Cost:More complex to create, easier to distribute

💡Practical Tip

Don't start with plugins. Start by creating simple Skills, validate that they work, and only package them as a plugin when you want to share them with the team or community. Most workflows can be solved with 2-3 individual Skills without needing a full plugin.

TOPIC 5

🤔 When to Use Each One

The golden rule is simple:always start with Skill. Only escalate to MCP or Plugin when the Skill does not resolve the issue. Most people jump straight to MCPs when a 20 line skill would do the trick.

🎯Main Concept

  • • Use Skill when:The task is about analyzing, generating, formatting or transforming text/code. Ex: code review, generate tests, refactor, create documentation.
  • • Use MCP when:Claude needs to access something outside the local file system. Ex: querying a database, sending a message on Slack, accessing the GitHub API.
  • • Use Plugin when:You have a set of Skills + configs that you want to distribute as a package. Ex: "DevOps plugin" with deployment, monitoring and incident response skills.

💡Practical Tip

Take the "Markdown test": if you can describe the entire task in a .md file with clear instructions, it's a Skill. If at any point you think "but he needs to access X to get Y", then you need an MCP. And if you have 5+ Skills that always go together, consider packaging them as a Plugin.

🧭 Decision Framework

Use this mental flowchart every time you need to automate something in Claude Code. In less than 30 seconds you know exactly which way to go, without wasting time on wrong approaches.

🎯Main Concept

Question 1: Do you need to access an external service?

SIM → MCP (or Plugin with built-in MCP)

NO → Go to Question 2

Question 2: Is it a single task or a set of tasks?

UNICA → Simple Skill (.md)

SET → Go to Question 3

Question 3: Do you need to share with the team/community?

YES → Plugin (packages everything)

NO → Multiple Skills in the .claude/commands/ folder

💡Practical Tip

80% of the time the answer is "Skill". That's no exaggeration - most development tasks are about manipulating text and code, which is exactly what Skills do. Save MCPs and Plugins for the 20% that really need external integration or distribution.
