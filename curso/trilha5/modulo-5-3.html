TOPIC 1: Purpose
1

🎯 Purpose

The first section of every skill defineswhat it does and why it exists. Without a clear purpose, Claude interprets the instructions ambiguously and the result varies with each execution.

🎯Main Concept

Purpose must answer three questions in a maximum of 3 lines:

  • •WHAT:What task does this skill perform? (ex: "Do security code review")
  • •WHY:What problem does it solve? (e.g. "Prevents vulnerabilities before merge")
  • •FOR WHOM:Who will use it? (ex: "Devs before opening PR")

Example in .md:

# Purpose
Realizar code review focado em seguranca para prevenir vulnerabilidades
antes do merge. Destinado a desenvolvedores que querem validar PRs
sem depender exclusivamente de revisores humanos.

💡Practical Tip

A vague Purpose like "help with code" will produce generic results. A specific Purpose such as "identify SQL injection, XSS and auth bypass in Python/FastAPI code" produces accurate and actionable results. The more specific, the better.

TOPIC 2: Inputs
2

📥 Inputs

The Inputs section defineswhat the skill needs to receiveto function. It can be a file, a directory, a URL, free text or specific parameters passed via$ARGUMENTS.

🎯Main Concept

# Inputs
- **Target**: $ARGUMENTS (arquivo ou diretorio para revisar)
- **Scope**: Se nenhum argumento, usar arquivos modificados no git diff
- **Language**: Auto-detectar a partir da extensao do arquivo
- **Severity**: Default "all" - pode filtrar com "critical-only"

💡Practical Tip

Always set a fallback for when $ARGUMENTS is empty. "If no file specified, analyze all files modified in the last commit" is much better than letting Claude guess what you want.

TOPIC 3: Steps
3

📋 Steps

The Steps are the core of the skill:exact sequence of actionsthat Claude must execute. Steps numbered, clear and in logical order. Each step must be atomic - a single, well-defined action.

🎯Main Concept

# Steps
1. Ler o arquivo/diretorio alvo e identificar a linguagem
2. Mapear todos os endpoints, funcoes publicas e pontos de entrada
3. Para cada ponto de entrada, verificar:
   - Input validation (sanitizacao, tipos, limites)
   - Authentication/Authorization (quem pode acessar)
   - Data exposure (dados sensiveis no response)
4. Classificar cada finding por severidade (Critical/High/Medium/Low)
5. Gerar sugestao de fix para cada finding Critical e High
6. Compilar relatorio no formato de saida definido

💡Practical Tip

The 5-8 step rule: less than 5 is probably too vague, more than 8 is too complex for a single skill. If you need more steps, consider dividing it into two skills that complement each other.

TOPIC 4: Quality Checks
4

✅ Quality Checks

Quality Checks arevalidation criteriawhich Claude must check before delivering the result. And the difference between a skill that “works sometimes” and one that delivers consistently.

🎯Main Concept

# Quality Checks
- [ ] Todos os arquivos do target foram analisados
- [ ] Cada finding tem severidade, descricao e localizacao
- [ ] Findings Critical/High tem sugestao de fix
- [ ] Nenhum falso positivo obvio (ex: test files flagged)
- [ ] Relatorio segue o formato de Output definido
- [ ] Se zero findings, confirmar com "No issues found"

💡Practical Tip

Quality Checks are your safety net. Claude really checks these criteria before finalizing. If you don't include checks, it will deliver the first result that seems reasonable, without validating completeness or accuracy.

TOPIC 5: Output
5

📤 Output

The Output section definesexactly how the result should be formatted. Without this, each run produces a different format and you lose the standardization that makes skills useful.

🎯Main Concept

# Output
Formato: Markdown estruturado

## Security Review: [filename]
**Score**: X/10
**Findings**: N issues (C critical, H high, M medium, L low)

### Critical Issues
| # | Location | Issue | Suggested Fix |
|---|----------|-------|---------------|
| 1 | line:col | desc  | fix code      |

### Summary
[1-2 frases com avaliacao geral e proximos passos]

💡Practical Tip

Include a concrete example of the expected output in the skill. Claude follows examples much better than abstract descriptions. Show how a table should look, how the score should be calculated, how the summary should be written.

TOPIC 6: Writing Effective Instructions
6

✍️ Writing Effective Instructions

The quality of a skill depends 100% on the quality of the instructions. Here are the principles that separate mediocre skills from skills that produce professional results.

🎯Main Concept

  • •Be imperative:"Analyze" instead of "Could you analyze". Direct commands produce better results.
  • •Be specific:"Check SQL injection in raw queries" instead of "check security".
  • •Use negatives:"DO NOT include test files" is clearer than "focus on production files".
  • •Include examples:A concrete example is worth more than 10 lines of explanation.
  • •Set limits:"Maximum 5 findings per category" avoids giant outputs.

💡Practical Tip

Test your skill 3 times with different inputs before considering it "ready". In the first run you discover ambiguities, in the second you adjust edge cases, in the third you confirm consistency. A skill that only works in the ideal case is not a skill, it is a draft.
