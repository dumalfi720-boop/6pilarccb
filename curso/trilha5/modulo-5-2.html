📁 The .claude/commands/ Folder

The directory.claude/commands/It is the heart of the Skills system. Every file.mdPlaced here automatically becomes a slash command accessible within Claude Code. No need to register, configure or restart anything.

🎯Main Concept

There are two levels of Skills:

  • •Project: .claude/commands/at the root of the project → accessible via/project:nome
  • •User: ~/.claude/commands/on home → accessible via/user:nomein any project

💡Practical Tip

Project skills go into git and are shared with the team. User skills are personal and work across all your projects. Place generic skills (code review, generate tests) in the user, and specific skills (deploy this project, standards in this codebase) in the project.

TOPIC 2
2

🔍 Self-Discovery

Claude Code automatically scans command folders when starting a session. Every file.mdfound becomes a slash command without any extra configuration. The file name becomes the command name.

🎯Main Concept

The mapping is direct: file name → command name.

  • •code-review.md → /project:code-review
  • •generate-tests.md → /project:generate-tests
  • •Subfolders create namespaces:devops/deploy.md → /project:devops:deploy

💡Practical Tip

Use descriptive names with hyphens:senior-code-review.mdand better thanscr.md. You'll type Tab to autocomplete anyway, and clear names help the entire team understand the purpose without opening the file.

TOPIC 3
3

⚡ Slash Commands in Action

Once the skill is in the right folder, using it is as simple as typing/followed by the prefix and name. Claude loads the .md content and follows the instructions as if you had pasted everything into the chat.

🎯Main Concept

  • •/project:name- Skills of the current project (folder .claude/commands/)
  • •/user:name- Personal skills (folder ~/.claude/commands/)
  • •Arguments:Use$ARGUMENTSin .md to receive parameters after the command

💡Practical Tip

The placeholder$ARGUMENTSis powerful: when you type/project:code-review src/auth.ts, Claude replaces $ARGUMENTS with "src/auth.ts" within the skill instructions. Use this to create parameterized skills that work with any file or input.

TOPIC 4
4

🏢 Cowork Built-ins

Claude Code Cowork (collaborative/web version) comes with pre-installed Skills that cover the most common use cases. These built-ins serve as a reference on how to structure your own skills.

🎯Main Concept

  • •Code Review:Complete analysis with security, performance, bugs and suggestions
  • •Generate Tests:Automatic creation of unit and integration tests
  • •Refactor:Guided refactoring with functionality preservation
  • •Document:Generation of technical documentation from code

💡Practical Tip

Study the built-ins before creating your own. They show the expected quality standard: clear instructions, numbered steps, quality criteria and defined output format. Copy the structure and adapt the content to your use case.

TOPIC 5
5

🛒Plugin Marketplace

In addition to individual Skills, there is a growing ecosystem of plugins that package multiple skills. The marketplace allows you to install complete sets of features with a single command.

🎯Main Concept

  • •Official plugins:Maintained by Anthropic or verified partners
  • •Community plugins:Created by the community, quality varies
  • •Installation:Via CLI or web marketplace, automatically registered

💡Practical Tip

Before installing a plugin, read the source code. Plugins have access to your file system and can execute commands. Check what each skill does, what tools it uses, and whether the repository has active maintenance. Abandoned plugin and security risk.

TOPIC 6
6

🗂️ Recommended File Structure

Good Skills organization makes the difference between a clean project and file chaos. Here is the recommended structure for scaling from 2 to 20 skills without losing control.

🎯Main Concept

.claude/
├── commands/                  # Skills do projeto
│   ├── code-review.md         # /project:code-review
│   ├── generate-tests.md      # /project:generate-tests
│   ├── devops/
│   │   ├── deploy.md          # /project:devops:deploy
│   │   └── rollback.md        # /project:devops:rollback
│   └── research/
│       ├── lead-research.md   # /project:research:lead-research
│       └── competitor.md      # /project:research:competitor
├── settings.json              # MCPs e configuracoes
└── CLAUDE.md                  # Instrucoes persistentes

💡Practical Tip

Use subfolders to group skills by domain (devops, research, frontend). This creates automatic namespaces in slash commands and makes navigation easier. When the project grows to 10+ skills, you will be thankful you organized it from the beginning.
