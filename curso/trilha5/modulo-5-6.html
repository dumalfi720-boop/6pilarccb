MODULE 5.6

💙 Skill: Senior Code Review

The most complete code review skill: analyzes security, bugs, performance and code quality with the eyes of a senior engineer. Include positive feedback so as not to demotivate the team.

TOPIC 1: Security
1

🔒 Security Analysis

The first pass of Senior Code Review focuses onsecurity vulnerabilities. And the analysis is more critical because security bugs can compromise the entire system, not just one feature.

🎯Main Concept

  • •Injection:SQL injection, XSS, command injection, path traversal
  • •Auth:Broken authentication, missing authorization checks, IDOR
  • •Date:Secrets hardcoded, PII exposure, insecure storage
  • •Deps:Dependencies with known CVEs, outdated versions

💡Practical Tip

Include in the skill: "If you find hardcoded secrets (API keys, passwords, tokens), mark them as CRITICAL and suggest using environment variables. Never include the secret value in the report." This prevents the review report itself from becoming a leak.

TOPIC 2: Bugs
2

🐛 Bug Detection

The second pass seekslogical bugs, untreated edge cases and potential errorswhich can cause crashes or unexpected behavior in production.

🎯Main Concept

  • •Null/undefined:Accesses without null check, optional chaining absent
  • •Race conditions:Concurrent access without synchronization
  • •Error handling:Empty try/catch, silenced errors, promises without .catch
  • •Edge cases:Empty list, empty string, negative numbers, overflow

💡Practical Tip

Ask Claude to mentally simulate the execution of the code with extreme inputs: list of 1 million items, 10MB string, request with 1ms timeout. This forces the detection of edge cases that no one thought of during development.

TOPIC 3: Performance
3

⚡ Performance Analysis

The third pass identifiesperformance bottlenecks, unnecessary complexity and optimization opportunitieswhich can impact the user experience and infrastructure costs.

🎯Main Concept

  • •Complexity:O(n²) where O(n) is possible, nested loops unnecessary
  • •N+1 queries:Loop that makes individual queries instead of batches
  • •Memory leaks:Event listeners not removed, closures retaining references
  • •Caching:Repeated computations that could be cached

💡Practical Tip

Add to the skill: "Only report performance issues that have a measurable impact. Do not suggest premature micro-optimizations. Focus on N+1 queries, algorithmic complexity and memory leaks." This avoids reports full of useless suggestions like "use const instead of let".

TOPIC 4: Quality
4

📐 Code Quality

The fourth pass evaluatesreadability, maintenance and adherence to standards. Code that works but is impossible to maintain is a technical debt that grows exponentially.

🎯Main Concept

  • •Naming:Variables and functions with descriptive and consistent names
  • •DRY:Duplicate code that should be extracted in function
  • •SRP:Functions doing many things, classes with mixed responsibilities
  • •Types:TypeScript with any, too generic types, missing interfaces

💡Practical Tip

Adapt quality standards to the project: "Follow the standards defined in the project's CLAUDE.md. If they do not exist, use the conventions already present in the codebase." This prevents the review from suggesting patterns that conflict with the team's style.

TOPIC 5: Output Format
5

📊 Output Format

The final report is structured in clear sections withnumerical score, categorized findings and recommended actions. Standard format that the entire team learns to read and act quickly.

🎯Main Concept

# Output Format

## Code Review: [file/PR name]
**Overall Score**: X/10
**Verdict**: APPROVE / REQUEST CHANGES / NEEDS DISCUSSION

### 🔒 Security (X issues)
| Severity | Location | Issue | Fix |
|----------|----------|-------|-----|

### 🐛 Bugs (X issues)
| Severity | Location | Issue | Fix |

### ⚡ Performance (X issues)
| Severity | Location | Issue | Fix |

### 📐 Quality (X issues)
| Severity | Location | Issue | Fix |

### 👏 What's Good
- [positive feedback items]

### 📋 Action Items
1. [ordered by priority]

💡Practical Tip

The "Verdict" of 3 options is intentional: APPROVE (can merge), REQUEST CHANGES (has a blocker) and NEEDS DISCUSSION (not a blocker but needs to talk). This aligns with GitHub's PR workflow and makes the team's decision easier.

TOPIC 6: Positive Feedback
6

👏 Positive Feedback

A code review that only points out problemsdemotivates the team. The "What's Good" section is mandatory in the skill to highlight good practices, smart decisions and improvements compared to previous versions.

🎯Main Concept

  • •Good practices:"Excellent use of early returns to reduce nesting"
  • •Design decisions:"Good choice to use Strategy pattern here - makes future extension easier"
  • •Tests:"Test coverage includes important edge cases"
  • •Documentation:"Complete JSDoc in public functions, will facilitate onboarding"

💡Practical Tip

Include in the skill: "Always list at least 3 positive points, even in code with many problems. If you don't find anything specific, recognize the effort: 'Well-structured code in logical folders' or 'Clear and descriptive Commit messages'." Reviews that are only critical destroy team culture.
