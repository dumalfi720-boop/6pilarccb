TOPIC 1: Structured Bug Report Deep Dive
1

🐛 Structured Bug Report - Advanced

O Structured Bug ReportAdvanced adds layers of context that speed up investigation. In addition to the basics (bug, expected, current), you include the environment, history of recent changes and initial hypotheses.

🎯Advanced Template

Investigue este bug com profundidade total:

## Contexto
- Bug: [descricao precisa]
- Ambiente: [dev/staging/prod] [OS] [Node/Python version]
- Primeira ocorrencia: [quando comecou]
- Frequencia: [sempre/intermitente/condicional]

## Sintomas
- Esperado: [comportamento correto]
- Atual: [comportamento errado]
- Erro exato: [mensagem/stack trace - cole aqui]

## Historico
- Mudancas recentes: [commits ou deploys recentes que podem ter causado]
- Ja tentei: [o que voce ja fez para resolver]

## Investigacao
1. Identifique a causa raiz (nao o sintoma)
2. Trace o fluxo de dados desde a entrada ate o ponto de falha
3. Verifique se ha outros lugares no codigo com o mesmo padrao vulneravel
4. Proponha fix minimo + teste de regressao
5. Avalie impacto: essa correcao pode quebrar algo?

💡Practical Tip

The "I've tried" field is crucial. It prevents Claude from suggesting solutions that you already know don't work, saving you time and tokens. Always list what you have already tested.

TOPIC 2: Log Analysis
2

📊 Log Analysis

The prompt forLog Analysisturns Claude into a specialized log analyzer. It identifies patterns, anomalies and correlations that would be difficult to find manually in extensive logs.

🎯Prompt Template

Analise estes logs e encontre o problema:

[COLE OS LOGS AQUI]

Preciso que voce:
1. Identifique a sequencia de eventos que levou ao erro
2. Separe logs normais de logs anomalos
3. Encontre o PRIMEIRO evento que divergiu do comportamento esperado
4. Correlacione timestamps para identificar latencias anormais
5. Identifique se ha padroes repetitivos (loops, retries excessivos)
6. Proponha a causa raiz baseada na evidencia dos logs

Formato: Timeline com [timestamp] [evento] [normal/anomalo] [significado]

💡Practical Tip

For very long logs, filter before pasting. Usegrep -A5 -B5 "ERROR\|WARN"to extract only the relevant lines with context. Claude analyzes focused logs better than giant logs.

TOPIC 3: Stack Trace Handling
3

📚 Stack Trace Handling

O Stack Trace Handlingteaches Claude how to interpret stack traces efficiently, separating his project's code from external libraries and identifying exactly where the problem is.

🎯Prompt Template

Analise este stack trace:

[COLE O STACK TRACE]

1. Separe as linhas do NOSSO codigo das linhas de bibliotecas externas
2. Identifique a linha EXATA do nosso codigo onde o erro originou
3. Leia o arquivo e a funcao apontados pelo stack trace
4. Explique o que estava acontecendo no momento do erro
5. Identifique a causa raiz (tipo errado? null? async nao tratado?)
6. Proponha a correcao minima

Foque apenas no nosso codigo - ignore frames de node_modules/stdlib.

💡Practical Tip

Always paste the complete stack trace, not just the error message. The intermediate frames often contain the most important clue - especially in callback and promise errors.

TOPIC 4: Error Categorization
4

🏷️ Error Categorization

A error categorizationis a prompt that organizes multiple errors by type, priority and relationship between them. Ideal for when you have several simultaneous bugs and need to know where to start.

🎯Prompt Template

Estou vendo multiplos erros. Categorize e priorize:

[COLE TODOS OS ERROS/LOGS]

Para cada erro:
1. Categoria: [Tipo | Runtime | Build | Config | Network | Auth | Data]
2. Severidade: [Critico | Alto | Medio | Baixo]
3. Relacionamento: Este erro pode ser CAUSADO por outro erro na lista?
4. Ordem de resolucao: Qual resolver primeiro?

Identifique se ha um erro RAIZ que, ao ser resolvido, corrige os outros em cascata.

💡Practical Tip

Cascading errors are extremely common. A configuration error can generate 10 different runtime errors. Always resolve the root error first - often 80% of errors go away on their own.

TOPIC 5: Root Cause Analysis
5

🔬 Root Cause Analysis

O Root Cause Analysisforces Claude to go beyond the superficial fix and find out why the bug exists in the first place. This prevents the same type of bug from appearing again elsewhere in the code.

🎯Prompt Template

Faca uma analise de causa raiz completa usando os 5 Porques:

Bug: [descricao]

Para cada "por que":
1. Por que o erro aconteceu? → [resposta]
2. Por que [resposta 1] aconteceu? → [resposta]
3. Por que [resposta 2] aconteceu? → [resposta]
4. Por que [resposta 3] aconteceu? → [resposta]
5. Por que [resposta 4] aconteceu? → [causa raiz]

Depois:
- Existe esse mesmo padrao em outros lugares do codigo?
- Que mudanca SISTEMICA preveniria esse tipo de bug?
- Que regra podemos adicionar ao CLAUDE.md para evitar?

💡Practical Tip

After finding the root cause, add prevention to theCLAUDE.md. Example: If the bug was caused by a lack of input validation, add: "Always validate inputs before processing. Never assume that external data is valid."

TOPIC 6: Complete Debugging Workflow
6

🔄 Complete Debugging Workflow

O complete debugging workflowcombines all previous prompts into a logical sequence. Follow this order to resolve any bug systematically, from the simplest to the most complex.

🎯Debugging sequence

1
Categorize- If multiple errors, categorize and find the root
2
Analyze logs/stack- Use Log Analysis or Stack Trace as appropriate
3
Bug Report- Document the bug with the structured template
4
Root Cause- 5 Whys to find the real root cause
5
Fix + Test- Minimum correction with regression test
6
Prevention- Update CLAUDE.md to prevent recurrence

💡Practical Tip

For simple bugs, skip straight to step 3 (Bug Report). The full sequence is for complex or intermittent bugs that resist superficial fixes. Always end with prevention - the best debug is what never needs to happen again.
