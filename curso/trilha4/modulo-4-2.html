TOPIC 1: 95% Confidence Deep Dive
1

🎯 95% Confidence - Deep Dive

O 95% ConfidenceIt goes beyond the basics when you delve into it. In this topic, we explore advanced variations: adjusting the confidence percentage, targeting specific areas of the project and combining with CLAUDE.md instructions for complete automation.

🎯Advanced Variation - Directed

Antes de tocar em qualquer coisa, estude especificamente:
1. A camada de autenticacao (auth/, middleware/, guards/)
2. Como os tokens sao gerados, validados e renovados
3. Os testes existentes para auth
4. As variaveis de ambiente relacionadas

Preciso de 95% de confianca de que voce entende o fluxo completo de auth antes de modificar qualquer coisa. Me mostre um diagrama do fluxo atual.
  • • Direct the studyprevents Claude from wasting time reading irrelevant files
  • • Request oneflow diagramforces Claude to synthesize understanding

💡Practical Tip

For large projects, use99% Confidencefor critical areas (payments, auth, sensitive data) and90% Confidencefor less critical areas (UI, formatting). The percentage communicates the expected depth of study.

TOPIC 2: Project Kickoff
2

🚀 Project Kickoff - New Projects

O Project Kickoffis the prompt for when you are starting a project from scratch. He guides Claude to create the initial framework with solid standards from the start, avoiding early technical debt.

🎯Prompt Template

Estou comecando um projeto novo. Antes de criar qualquer arquivo:

1. Me faca perguntas sobre: stack, escopo, requisitos, deploy target
2. Baseado nas respostas, proponha a estrutura de pastas
3. Defina os padroes: naming conventions, estrutura de imports, error handling
4. Crie o CLAUDE.md com essas convencoes documentadas
5. So entao comece a implementar, um passo de cada vez

Projeto: [descricao do projeto]
Stack desejada: [tecnologias]
  • • Force Claude toask before committingarchitectural decisions
  • • Create theCLAUDE.md from the beginning- conventions are documented

✓ What TO DO

  • ✓Answer Claude's questions in detail
  • ✓Review the proposed folder structure
  • ✓Validate CLAUDE.md before proceeding

✗ What NOT to do

  • ✗Skip the questions and have everything created at once
  • ✗Letting Claude choose the stack without your guidance
  • ✗Start coding without defining conventions first
TOPIC 3: Codebase Onboarding
3

🗺️ Codebase Onboarding - Existing Projects

O Codebase Onboardingand for when you join a project that already exists. He has Claude map out the architecture, identify patterns, and create a mind map of the project for you.

🎯Prompt Template

Estou entrando neste projeto pela primeira vez. Faca um onboarding completo:

1. Leia package.json / requirements.txt / go.mod (ou equivalente)
2. Mapeie a estrutura de pastas e o proposito de cada diretorio
3. Identifique os padroes de arquitetura (MVC, Clean, Hexagonal, etc)
4. Liste os endpoints/rotas principais
5. Identifique o fluxo de dados (entrada → processamento → saida)
6. Aponte areas de risco ou divida tecnica visivel

Me de um resumo executivo de 1 paragrafo seguido do mapeamento detalhado.

💡Practical Tip

Save the Codebase Onboarding output toCLAUDE.mdof the project. Thus, future sessions start with this mapping and do not need to redo the study. Update as the project evolves.

TOPIC 4: First Session Strategy
4

🎬 First Session Strategy

A first sessionsets the tone for all work with Claude on a project. If you start badly, every subsequent session inherits problems. Here is the ideal sequence for the very first interaction with any project.

🎯First Session Sequence

1
Run /init- Creates the base CLAUDE.md automatically
2
Use Codebase Onboarding- Map the complete architecture
3
Enriches CLAUDE.md- Add mapping to context file
4
Do a small task- Validate that Claude understands the project with a simple change

💡Practical Tip

The small validation task should touch a central area of ​​the project. If Claude gets it right, you know that the understanding is correct. If you make a mistake, you identify gaps in CLAUDE.md and correct them before carrying out larger tasks.

TOPIC 5: Question Patterns
5

❓ Strategic Question Patterns

Do theright questionsto Claude at the beginning of the session saves hours of rework. These are the question patterns that extract the maximum amount of relevant information before starting any work.

🎯Questions You Should Ask

Perguntas de Arquitetura:
- "Quais sao as dependencias criticas deste modulo?"
- "Se eu mudar X, o que pode quebrar?"
- "Existe algum padrao nao-obvio neste codigo que eu deveria saber?"

Perguntas de Contexto:
- "Como essa feature se conecta com o resto do sistema?"
- "Quais testes cobrem essa area?"
- "Existe documentacao inline que eu deveria ler?"

Perguntas de Risco:
- "Qual o pior cenario se essa mudanca tiver um bug?"
- "Tem alguma race condition ou estado compartilhado aqui?"
- "Essa area tem historico de bugs recentes nos commits?"

💡Practical Tip

Add your favorite questions toCLAUDE.mdas an automatic checklist. Example: "Before making any change, always answer: which files will be affected? What tests are there? What is the risk?"

TOPIC 6: Avoiding Blind Spots
6

🚫 Avoiding Blind Spots

Claude haspredictable blind spotsat the beginning of sessions. It tends to assume common defaults, ignore project-specific settings, and does not check dependency versions. Knowing this allows you to prevent common mistakes.

🎯Common Blind Spots and How to Prevent them

  • • Dependency versions- Instruct: "Check the version in package.json before using any API"
  • • Environment settings- Instruct: "Read .env.example before assuming environment variables"
  • • Team conventions- Instruct: "Follow the style of existing files, do not invent new patterns"
  • • Existing tests- Instruct: "Run the tests before and after any change"

💡Practical Tip

All these prevention instructions should go to theCLAUDE.md. This way, Claude reads automatically every session and you don't need to repeat it. CLAUDE.md is your weapon against blind spots.
