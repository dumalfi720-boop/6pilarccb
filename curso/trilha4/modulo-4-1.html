TOPIC 1: 95% Confidence
1

🎯 95% Confidence - The Start Prompt

The prompt95% Confidenceand the most important thing on the panel. It forces Claude to study the project before acting, eliminating generic responses and ensuring that any action is based on real code. Use it whenever you start a session or face a complex task.

🎯Prompt Template

Antes de fazer qualquer coisa, leia os arquivos relevantes e entenda o projeto. Nao comece a codar ate ter 95% de confianca de que entende:
1. A estrutura do projeto
2. Os padroes e convencoes existentes
3. Como a mudanca solicitada se encaixa
4. Quais arquivos serao afetados

Depois de estudar, me diga o que voce entendeu e seu plano. So execute depois que eu aprovar.
  • • Strengthreading before action- eliminates the addiction of generating generic code
  • • Create aapproval checkpoint- you validate understanding before execution
  • • Ideal forfirst sessionsin existing projects or tasks that involve multiple files

💡Practical Tip

Put this prompt in yourCLAUDE.mdas standard instruction. Thus, every session begins with automatic study of the project. You can adjust the percentage (90%, 95%, 99%) according to the criticality of the task.

✓ What TO DO

  • ✓Use at the beginning of every session with complex tasks
  • ✓Read the plan that Claude presents before approving
  • ✓Ask you to list the files you will modify

✗ What NOT to do

  • ✗Approving the plan without reading it - this defeats the purpose
  • ✗Use for trivial tasks like renaming a variable
  • ✗Skip the study and ask them to go straight to the code
TOPIC 2: Plan Mode Blueprint
2

📋 Plan Mode Blueprint - The Planning Prompt

O Plan Mode BlueprintIt's the prompt you use when you need to build something meaningful. It activates Claude's planning mode, generating a detailed blueprint before any implementation. Perfect for new features, large refactorings or architectural changes.

🎯Prompt Template

Use o modo ultrathink para planejar esta tarefa. Quero um blueprint completo antes de qualquer codigo:

1. Objetivo: [descreva o que precisa ser feito]
2. Liste todos os arquivos que serao criados ou modificados
3. Para cada arquivo, descreva as mudancas especificas
4. Identifique dependencias e ordem de implementacao
5. Liste riscos e pontos de atencao
6. Proponha testes para validar

NAO implemente nada ainda. Apenas o plano.
  • • Activate theextended thinkingfor in-depth analysis before execution
  • • Generates aimplementation mapthat you can review, adjust and approve
  • • Reducereworkby identifying problems before implementation

💡Practical Tip

After receiving the blueprint, useShift+Tabto enter multiline mode and make detailed adjustments before approving the implementation. You can ask to implement one step at a time.

✓ What TO DO

  • ✓Use for features that play more than 3 files
  • ✓Request incremental implementation after approving the plan
  • ✓Save the blueprint as a reference during development

✗ What NOT to do

  • ✗Use for simple changes that do not require a plan
  • ✗Ignore the risks identified in the blueprint
  • ✗Asking to implement everything at once without reviewing
TOPIC 3: Structured Bug Report
3

🐛 Structured Bug Report - The Debug Prompt

O Structured Bug Reportturns any problem into a systematic investigation. Instead of saying "it's broken", you provide a format that forces Claude to analyze the bug like a senior engineer: reproduce, isolate, diagnose and propose a solution.

🎯Prompt Template

Investigue este bug de forma estruturada:

Bug: [descricao do problema]
Comportamento esperado: [o que deveria acontecer]
Comportamento atual: [o que esta acontecendo]
Passos para reproduzir: [se conhecidos]

Siga este processo:
1. Leia os arquivos relevantes e entenda o fluxo
2. Identifique a causa raiz (nao apenas o sintoma)
3. Proponha a correcao minima necessaria
4. Explique por que essa correcao resolve sem efeitos colaterais
5. Sugira um teste para prevenir regressao

NAO corrija antes de explicar a causa raiz.
  • • Structure the investigation intoroot cause vs symptom
  • • Demandsminimum correction- avoids unnecessary rewrites
  • • Includesregression testto prevent the bug from returning

💡Practical Tip

Include the error log or stack trace directly at the prompt. The more concrete context you provide, the less time Claude spends reading irrelevant files. UseShift+Tabto paste multiline logs.

✓ What TO DO

  • ✓Include logs, stack traces and actual error messages
  • ✓Require explanation of the cause before accepting the correction
  • ✓Request a regression test along with the fix

✗ What NOT to do

  • ✗Just say "it's broken, fix it"
  • ✗Accepting correction without understanding the root cause
  • ✗Let Claude rewrite entire files for a one-off bug
TOPIC 4: Brutal Audit
4

🔍 Brutal Audit - The Quality Prompt

O Brutal Auditand your relentless reviewer. He instructs Claude to find everything that is wrong, weak or poorly done in the code. Use at the end of a development session to ensure that what has been built is production quality.

🎯Prompt Template

Faca uma auditoria brutal do codigo que acabamos de escrever. Seja implacavel:

1. Bugs potenciais e edge cases nao tratados
2. Falhas de seguranca (injecao, exposicao de dados, auth)
3. Problemas de performance (N+1, memory leaks, loops desnecessarios)
4. Codigo morto ou duplicado
5. Violacoes de padroes do projeto
6. Testes faltando ou insuficientes

Para cada problema encontrado, classifique como:
- CRITICO: Precisa corrigir antes de deploy
- ALTO: Deveria corrigir, risco real
- MEDIO: Melhoraria a qualidade
- BAIXO: Sugestao de melhoria

NAO suavize. Prefiro saber agora do que descobrir em producao.
  • • Sort byseverityhelps prioritize fixes
  • • Coppersecurity, performance, standards and testingin a single pass
  • • The "brutal" tone instructs Claude todon't be complacentwith problems

💡Practical Tip

Run Brutal Audit before every commit or pull request. After receiving the report, ask to correct only the CRITICAL and HIGH items first. MEDIUM and LOW items can become issues for later.

✓ What TO DO

  • ✓Run before every important commit
  • ✓Correct CRITICAL and HIGH before proceeding
  • ✓Use as a recurring quality checklist

✗ What NOT to do

  • ✗Ignore results because "works in dev"
  • ✗Ask to soften feedback
  • ✗Fix everything at once without prioritizing by severity
TOPIC 5: Teach Me
5

📚 Teach Me - The Learning Prompt

O Teach MeTurn Claude into your personal tutor. Instead of just solving the problem, he explains the concept behind it, with examples from his own code. Use it when you find something you don't understand or when you want to delve deeper into a concept.

🎯Prompt Template

Me ensine [CONCEITO/PADRAO] usando exemplos do nosso proprio projeto.

1. Explique o conceito de forma simples (como se eu tivesse 5 anos de experiencia)
2. Mostre onde ja usamos isso no projeto (aponte arquivos e linhas)
3. Explique POR QUE usamos dessa forma e nao de outra
4. De um exemplo pratico de como eu aplicaria isso na proxima tarefa
5. Liste armadilhas comuns que devo evitar

Nao simplifique demais - quero entender de verdade, nao apenas copiar.
  • • Useexamples of your projectinstead of generic examples
  • • Explains the"why"beyond the "how"
  • • Includespractical trapsto avoid common mistakes

💡Practical Tip

Use Teach Me whenever Claude applies a pattern that you don't know. Instead of just accepting the code, understand the concept. This speeds up your learning exponentially because the examples are from your own context.

✓ What TO DO

  • ✓Use when you find new concepts in your code
  • ✓Ask for concrete examples of the current project
  • ✓Ask follow-up questions to delve deeper

✗ What NOT to do

  • ✗Accepting code you don't understand without asking
  • ✗Use as a substitute for official documentation on critical concepts
  • ✗Asking for very basic explanations that unnecessarily consume context
TOPIC 6: Combination Strategy
6

🔄 Combination Strategy

The 5 prompts are not used in isolation. THEcombination strategydefines the right order and timing for each person during a typical work session. Master this sequence and you will have a complete and predictable workflow.

🎯Recommended Daily Sequence

1
95% Confidence- Start of the session. Claude studies the project and presents what he understood.
2
Plan Mode Blueprint- Plan the day's task. Generate the blueprint before coding.
3
Structured Bug Report- During development, when you find bugs.
4
Brutal Audit- Before commit. Review everything that was built.
5
Teach Me- Any time. When you find something you don't understand.

📊Why This Order Works

  • Natural progression- Study → Planning → Execution → Debug → Review → Learning
  • Each prompt feeds into the next- The 95% study informs the blueprint, which guides development, which generates bugs to investigate
  • Audit as a quality gate- Nothing goes into production without going through brutal auditing

💡Practical Tip

You do not need to use all 5 prompts in every session. For quick bug fixing sessions, just use Structured Bug Report + Brutal Audit. For learning sessions, focus on Teach Me. The key is knowing which prompt to use at which time.
