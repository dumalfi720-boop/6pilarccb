TOPIC 1: Feature Blueprint
1

📐 Feature Blueprint

O Feature Blueprinttransforms a vague idea into a concrete implementation plan. It forces Claude to decompose the feature into components, define interfaces and map dependencies before writing a line of code.

🎯Prompt Template

Crie um blueprint para esta feature antes de implementar:

Feature: [descricao detalhada]

Preciso de:
1. Decomposicao em sub-tarefas (max 5 passos)
2. Para cada sub-tarefa: arquivos envolvidos, mudancas especificas
3. Interfaces/tipos que serao criados ou modificados
4. Dependencias entre os passos (o que precisa vir primeiro)
5. Estimativa de complexidade por passo (simples/medio/complexo)
6. Testes necessarios por passo

Formato: tabela com colunas [Passo | Arquivos | Mudanca | Dependencia | Complexidade | Testes]

NAO implemente. Apenas o blueprint.
  • • The formattablemakes it easier to view and review the plan
  • • Limit to5 stepsforce decomposition without over-engineering

💡Practical Tip

After receiving the blueprint, say"Implemente o passo 1"and review before proceeding. Never say "implement everything". Step-by-step incremental implementation is the key to maintaining quality.

TOPIC 2: Build With Tests
2

🧪 Build With Tests

O Build With Testsensures that each piece of new code is already tested. Instead of building and testing later, you instruct Claude to write the test first and then the implementation that satisfies it.

🎯Prompt Template

Implemente [FEATURE] usando TDD:

1. PRIMEIRO: Escreva os testes que definem o comportamento esperado
   - Happy path (caso normal)
   - Edge cases (limites, valores nulos, erros)
   - Integracão com componentes existentes

2. DEPOIS: Implemente o codigo minimo que faz todos os testes passarem

3. FINALMENTE: Rode os testes e me mostre o resultado

Regras:
- Cada teste deve ter um nome descritivo que explica O QUE testa
- Use o framework de testes que ja existe no projeto
- Nao use mocks desnecessarios - prefira testes de integracao quando possivel

💡Practical Tip

If the tests pass the first time, be suspicious. Ask Claude to add more aggressive edge cases. Tests that never fail may be testing the wrong thing.

✓ What TO DO

  • ✓Write test BEFORE production code
  • ✓Include edge cases and error scenarios
  • ✓Run tests and check that they pass

✗ What NOT to do

  • ✗Build everything and "test later"
  • ✗Accept tests that only cover happy path
  • ✗Use mocks for everything instead of real tests
TOPIC 3: Incremental Building
3

🧱 Incremental Construction

A incremental constructionand the principle of never asking Claude to implement more than one step at a time. Each step is reviewed, tested and approved before proceeding. This eliminates massive rewrites and maintains full control.

🎯Incremental Flow

Passo 1: "Implemente apenas [componente A]. Nao toque em mais nada."
         → Revise → Teste → Aprove

Passo 2: "Agora implemente [componente B] que se conecta com A."
         → Revise → Teste → Aprove

Passo 3: "Integre A e B. Rode os testes de integracao."
         → Revise → Teste → Aprove

REGRA: Cada passo deve ser commitavel isoladamente.
  • • Every stepcommittable alonemeans you can revert without losing everything
  • • To say"Don't touch anything else"prevents unwanted changes to other files

💡Practical Tip

Usegit diffafter each step to check that Claude did not modify files that he should not have. If it has changed, revert it and ask again with more explicit instructions.

TOPIC 4: Test-First Approach
4

🔄 Test-First Approach

The approachtest-firstgoes beyond classic TDD. You use tests as an executable specification: you define the desired behavior in tests and ask Claude to make the tests pass. This reverses the flow and generates more focused code.

🎯Prompt Template

Eu ja escrevi os testes que definem o comportamento esperado.
Eles estao em [caminho/do/teste].

Sua tarefa:
1. Leia os testes e entenda o que cada um espera
2. Implemente o codigo de producao que faz TODOS passarem
3. NAO modifique os testes - eles sao a especificacao
4. Se algum teste parece impossivel de satisfazer, me avise antes de mudar

Rode os testes ao final e me mostre o resultado.

💡Practical Tip

Write the most critical tests yourself (business rules, security) and let Claude write the edge case and integration tests. You maintain control over the “what” while delegating the “how”.

TOPIC 5: Approval Workflow
5

✅ Approval Workflow

O approval workflowdefines checkpoints where you stop, review and decide whether Claude should continue. Without these gates, Claude could build an entire feature in the wrong direction before you know it.

🎯Template with Gates

Implemente [FEATURE] com os seguintes gates de aprovacao:

GATE 1 - Plano: Me mostre o plano antes de codar. PARE e espere aprovacao.
GATE 2 - Interfaces: Crie os tipos/interfaces. PARE e espere aprovacao.
GATE 3 - Implementacao core: Implemente a logica principal. PARE e espere aprovacao.
GATE 4 - Testes: Escreva e rode os testes. PARE e espere aprovacao.
GATE 5 - Integracao: Conecte com o resto do sistema. PARE e espere aprovacao.

Em NENHUM momento pule um gate. Sempre espere meu "ok" ou "aprovado" para continuar.

💡Practical Tip

For simple tasks, use only 2 gates (plan and implementation). For critical tasks, use 5. Adjust the gate granularity according to the risk of change.

TOPIC 6: Quality Gates
6

🏁 Quality Automatic Gates

Os automatic quality gatesThese are instructions that you place on CLAUDE.md so that Claude applies quality checks automatically, without you having to ask each time.

🎯Quality Gates for CLAUDE.md

# Quality Gates (CLAUDE.md)

Antes de considerar qualquer tarefa como "completa":
- [ ] Testes escritos E passando
- [ ] Sem erros de linting
- [ ] Sem TODO/FIXME sem issue associada
- [ ] Tipos corretos (sem any desnecessario)
- [ ] Error handling para todos os caminhos de falha
- [ ] Sem dados hardcoded (use env vars ou config)
- [ ] git diff revisado - nenhuma mudanca inesperada

💡Practical Tip

Add these quality gates to your CLAUDE.md and Claude will automatically check each item before saying it is finished. If you notice that a gate is always ignored, make the instruction more explicit.
