TOPIC 1: Architecture Decision Record
1

📄 Architecture Decision Record (ADR)

O Architecture Decision Recorddocuments architectural decisions in a structured way. Each important decision (database, framework, API standard) receives a record that explains the context, the options evaluated and the rationale for the choice.

🎯Prompt Template

Crie um ADR (Architecture Decision Record) para esta decisao:

Decisao: [o que precisa ser decidido]
Contexto: [por que estamos tomando essa decisao agora]

Formato do ADR:
# ADR-[numero]: [titulo]
## Status: [Proposto | Aceito | Deprecado | Substituido]
## Contexto: [situacao que motiva a decisao]
## Opcoes Consideradas:
  - Opcao A: [descricao] | Pros: [...] | Cons: [...]
  - Opcao B: [descricao] | Pros: [...] | Cons: [...]
  - Opcao C: [descricao] | Pros: [...] | Cons: [...]
## Decisao: [opcao escolhida e por que]
## Consequencias: [o que muda com essa decisao]
## Metricas: [como vamos medir se a decisao foi boa]

💡Practical Tip

Create a folderdocs/adr/in the project and save each ADR as a file. When someone asks "why do we use X?", the answer is documented. Reference the ADRs on CLAUDE.md.

TOPIC 2: API Design
2

🔌 API Design Prompt

O API Design Promptguides Claude to design APIs that are consistent, versionable, and easy to use. It covers naming, status codes, error formats, pagination and authentication.

🎯Prompt Template

Projete a API para [RECURSO/FEATURE]:

1. Liste todos os endpoints necessarios (CRUD + acoes especiais)
2. Para cada endpoint defina:
   - Metodo HTTP e path (RESTful)
   - Request body/params (com tipos e validacoes)
   - Response body (com exemplos)
   - Status codes possiveis (sucesso e erro)
   - Autenticacao necessaria
3. Defina o formato padrao de erro
4. Defina a estrategia de paginacao
5. Considere versionamento (v1, v2)
6. Identifique rate limits necessarios

Siga os padroes das APIs existentes no projeto. Consistencia > perfeicao.

💡Practical Tip

The final phrase "Consistency > perfection" is intentional. Claude tends to propose the "ideal" design that may be inconsistent with what already exists. Forcing it to follow existing standards prevents a Frankenstein API.

TOPIC 3: Comparing Options
3

⚖️ Comparison of Options

The prompt forcomparison of optionsforces Claude to objectively analyze alternatives with explicit trade-offs. Ideal for decisions such as "which bank to use", "which standard to adopt" or "build vs buy".

🎯Prompt Template

Compare estas opcoes para [DECISAO]:
- Opcao A: [descricao]
- Opcao B: [descricao]
- Opcao C: [descricao]

Para cada opcao, avalie em escala 1-5:
| Criterio        | Opcao A | Opcao B | Opcao C |
|-----------------|---------|---------|---------|
| Complexidade    |         |         |         |
| Performance     |         |         |         |
| Escalabilidade  |         |         |         |
| Manutencao      |         |         |         |
| Curva de aprend.|         |         |         |
| Custo           |         |         |         |

IMPORTANTE: Nao tenha favoritos. Seja objetivo.
Para a opcao RECOMENDADA, explique o trade-off principal que estamos aceitando.

💡Practical Tip

The instruction "don't play favorites" is essential because Claude has a bias for popular technologies. Asking him to explain the trade-off of the recommended option ensures that you make the informed decision, not him.

TOPIC 4: Scalability Analysis
4

📈 Scalability Analysis

The prompt forscalability analysismakes Claude evaluate how the current architecture behaves under increasing load and identify bottlenecks before they become problems in production.

🎯Prompt Template

Analise a escalabilidade da nossa arquitetura atual:

Cenarios de carga:
- Atual: [X usuarios/requests por dia]
- 10x: [10X usuarios] - o que quebra primeiro?
- 100x: [100X usuarios] - o que precisa ser re-arquitetado?

Para cada gargalo identificado:
1. Componente afetado
2. Tipo de gargalo (CPU, memoria, I/O, rede, banco)
3. Quando vai comecar a ser problema (qual volume)
4. Solucao proposta (cache, sharding, queue, scale out)
5. Esforco estimado para implementar

Priorize: o que devo resolver AGORA vs o que pode esperar?

💡Practical Tip

The question “what should I resolve NOW vs what can wait” is the most valuable. Premature optimization is the enemy - you want to know what needs immediate attention and what can be planned for in the future.

TOPIC 5: Maintenance & Technical Debt
5

🔧 Maintenance and Technical Division

The prompt formaintenance and technical divisionmakes Claude inventory the existing technical debt, calculate the "interest cost" (time spent working around the problems) and prioritize what to pay first.

🎯Prompt Template

Faca um inventario da divida tecnica do projeto:

1. Identifique todas as areas com divida tecnica
2. Para cada area:
   - Tipo: [codigo legado | falta de testes | arquitetura inadequada | dependencia desatualizada]
   - Impacto diario: [quanto tempo perdemos trabalhando ao redor disso]
   - Risco: [o que acontece se nao resolvermos]
   - Esforco para pagar: [horas/dias estimados]
   - ROI: [impacto / esforco]
3. Ordene por ROI (maior ROI = pagar primeiro)

Regra: Priorize divida que ATRAPALHA o trabalho diario sobre divida "estetica".

💡Practical Tip

Rotate this inventory monthly and compare it with the previous one. If debt is growing, you need to allocate more time to paying it off. If it is decreasing, your strategy is working.

TOPIC 6: Architecture Documentation
6

📚 Architectural Documentation

A architectural documentationIt's the prompt that makes Claude generate living documentation of the project's architecture. Unlike static documentation, this documentation is generated from the actual code and remains up to date.

🎯Prompt Template

Gere documentacao arquitetural baseada no codigo REAL do projeto:

1. Diagrama de componentes (em texto/mermaid)
   - Quais sao os modulos principais
   - Como eles se comunicam
   - Quais sao as dependencias

2. Fluxos criticos (top 3)
   - Request → Response para os endpoints mais importantes
   - Fluxo de dados do input ao banco

3. Decisoes arquiteturais ativas
   - Padroes usados e por que (inferir do codigo)
   - Convencoes de naming e organizacao

4. Pontos de extensao
   - Onde adicionar novas features
   - Onde NAO tocar (areas criticas)

Baseie TUDO no codigo real. Nao invente nada que nao esteja no projeto.

💡Practical Tip

Save the generated documentation and add it toCLAUDE.mdas a reference. Regenerate with each major architectural change. The phrase "base EVERYTHING on real code" prevents fanciful documentation.
