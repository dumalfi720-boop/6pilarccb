📖 /help - The Instant Help Center

The command/helpIt's the first one you should learn in Claude Code. It works as a living index of everything the tool offers: slash commands, command line flags, keyboard shortcuts and available settings. Instead of memorizing documentation, you consult it directly in the terminal.

Main Concept

🎯Main Concept

O /helpIt's not just a list of commands. It is a navigation map that updates as new features are added to Claude Code. When an update brings new slash commands or flags, /help automatically reflects this.

  • • List allslash commandsavailable in the current session (e.g. /clear, /compact, /cost, /doctor, /init, etc.)
  • • Show theCLI flagswhich you can use when starting Claude Code (ex: --model, --allowedTools, --print)
  • • Displayskeyboard shortcutssuch as Escape (cancel generation), Shift+Tab (multiple line mode) and others
Data/Research

📊Why This Matters

  • Less than 20%of CLI tool users explore all available commands
  • Active discovery- Running /help periodically reveals features you didn't know existed
  • Zero external dependence- No need to leave the terminal to consult online documentation
Practical Tip

💡Practical Tip

Rotate/helpright when opening Claude Code for the first time after an update. New versions often add commands and flags that can completely change your workflow. Real example: the /doctor command was added silently and resolves configuration issues that previously required reinstallation.

Combine withclaude --helpin the shell (outside the session) to see the startup flags. They are complementary contexts: one shows what to do within the session, the other shows how to configure the entry.

Do vs Avoid

✓ What TO DO

  • ✓ Run /help as the first command in new sessions
  • ✓ Check /help after each Claude Code update
  • ✓ Use as a quick reference before trying a command you don't remember

✗ What NOT to do

  • ✗ Trying to memorize all the commands without ever consulting /help
  • ✗ Search for documentation online when the answer is in the terminal itself
  • ✗ Ignore unknown flags - they might be exactly what you need

🧹 /clear - Clearing the Session

O /clearclears the terminal screen without erasing the context that Claude has accumulated. And how to clean your desk without throwing away the documents you are using. The session continues right where it left off, but visually you have a clean canvas to focus on the next step.

Main Concept

🎯Main Concept

There is a common confusion between/clear e /compact. They are completely different things:

  • • /clearonly affects the visual display of the terminal. Claude still has the entire message history in memory. Tokens consumed remain the same.
  • • /compact(next topic) actually changes the internal context, summarizing previous messages to free up space in the context window.
  • • Analogy: /clear and clear the whiteboard. /compact and photograph the frame, save the photo and delete the frame to have new space.
Practical Tip

💡Practical Tip

Use/clearwhen Claude has just generated a huge block of code or done several operations in sequence and his screen is full of output. Visual cleaning helps you focus on the next instruction without distractions.

Ideal time: after Claude performs a major refactoring, or after you review an extensive diff. Clear the screen and move on to the next step with renewed focus.

Data/Research

🔍Technical Difference

  • /clear- Resets the terminal display. No tokens are released. No information is lost. Session cost does not change.
  • Ctrl+L- Does the same thing in many terminals, but within Claude Code /clear is more reliable because it resets the visual state of the REPL itself.
  • Zero impact- You can run /clear 100 times and the session remains identical. There is no risk of losing context.
Do vs Avoid

✓ What TO DO

  • ✓ Use /clear when the screen is cluttered and you need visual focus
  • ✓ Combine /clear with /cost to get a clear view of current expenses
  • ✓ Use between task transitions within the same session

✗ What NOT to do

  • ✗ Confusing /clear with /compact thinking it frees context
  • ✗ Use /clear hoping Claude "forgets" something you said
  • ✗ Run /clear when you need to copy previous output (clears the screen!)

📦 /compact - Optimizing Context

O /compactIt is probably the most important command for those who work in long sessions. It compresses conversation history, generating an intelligent summary that preserves decisions and critical context while freeing up space in the context window. Without it, long sessions degrade in quality and eventually reach the limit.

Main Concept

🎯Main Concept

Each message exchanged with Claude Code consumes context window tokens (typically 200k tokens). When you get close to the limit, three things happen: Claude starts to "forget" instructions from the beginning, the quality of the responses drops, and eventually you get a context full error.

  • • How it works:Claude reads the entire history, generates a condensed summary preserving key decisions, edited files, and current work status, and replaces the full history with this summary
  • • What is preserved:Architectural decisions, modified file names, errors found and resolved, current task status, CLAUDE.md instructions
  • • What is lost:Granular details of previous conversations, intermediate code that was replaced, attempts that failed (unless relevant)
Data/Research

📊Compaction Numbers

  • 70-80%of the context used - ideal time to run /compact
  • ~60-70%typical reduction in context usage after compression
  • Multiple compressionsare possible in the same session, but each one loses more details
  • Automatic- Claude Code also automatically compresses when approaching the limit, but proactively compressing gives better results
Practical Tip

💡Practical Tip

You can pass custom instructions to /compact. Instead of just running/compact, try/compact foco nos arquivos de API e nas decisoes de banco de dados. This directs the summary to preserve exactly what matters for your next step.

Rule of thumb: if you've already run /cost and seen that the context is above 70%, run /compact immediately. Waiting for automatic compression often results in worse summaries because the system is under pressure.

Alert

⚠️Attention

After /compact, Claude does not remember granular details of previous conversations. If you need it to remember something specific, mention it explicitly before packing or use the custom instruction parameter. Information on CLAUDE.md is always preserved because it is reloaded with each interaction.

Do vs Avoid

✓ What TO DO

  • ✓ Proactively compact when reaching 70-80% of context
  • ✓ Use custom instructions: /compact to preserve architectural decisions
  • ✓ Check /cost before and after to confirm the reduction

✗ What NOT to do

  • ✗ Waiting for automatic compression (results in worse summaries)
  • ✗ Compress with only 20-30% usage (unnecessary, loses details without gain)
  • ✗ Assume that Claude remembers everything after compressing (always re-confirm critical points)

💰 /cost - Monitoring Spending and Tokens

O /costdisplays the accumulated cost of the session in dollars, count of tokens consumed, and usage statistics. And your consumption meter in real time. Professional operators check /cost regularly, just as a pilot checks the fuel panel.

Main Concept

🎯Main Concept

Each interaction with Claude Code consumes input tokens (your message + context) and output tokens (Claude's response). /cost shows exactly how much has been spent so far, allowing you to make informed decisions about when to compact, when to start a new session, or when a task is becoming too expensive.

  • • Cost in USD:Accumulated value of the current session, calculated based on the input and output tokens consumed
  • • Token count:Total input and output tokens, with breakdown by category (cache hits, new reads)
  • • Context indicator:Shows how much of the context window is being used, serving as a direct trigger for /compact decisions
Data/Research

📊Typical Costs by Task Type

  • $0.02-0.10- Quick questions, specific corrections, code explanations
  • $0.10-0.50- Implementation of small features, debugging, refactoring a file
  • $0.50-2.00- Complex features, multi-file refactoring, extensive debugging
  • $2.00+- Long sessions with many iterations, projects from scratch, multiple compressions
Practical Tip

💡Practical Tip

Get into the habit of spinning/costevery 10-15 interactions. If the context is above 70%, it's time to /compact. If the cost is rising quickly, evaluate whether you are being specific enough in the instructions. Vague prompts generate long responses that consume more tokens.

For plans with usage limits (Claude Pro with $100/month, for example), /cost becomes your best friend. You can estimate how many sessions fit into your monthly budget and adjust the intensity of use.

Do vs Avoid

✓ What TO DO

  • ✓ Check /cost regularly (every 10-15 interactions)
  • ✓ Use as trigger for /compact when context exceeds 70%
  • ✓ Compare costs between sessions to identify usage patterns

✗ What NOT to do

  • ✗ Ignore /cost and be surprised by your consumption at the end of the month
  • ✗ Obsessively checking every message (creates unnecessary anxiety)
  • ✗ Trying to save tokens by writing incomprehensible prompts (it costs more for retries)

🔎 /status - Checking the Environment Status

O /statusshows the complete state of your current session: which model is in use, which MCPs are connected, which tools are available, and which settings are active. It is the first troubleshooting command when something does not work as expected.

Main Concept

🎯Main Concept

/status works as a quick diagnosis of your environment. When an MCP is unresponsive, when a tool is unavailable, or when the model appears to be behaving differently, /status reveals what is happening behind the scenes.

  • • Active model:Confirm whether you are using Opus, Sonnet or another model. Essential when you want to ensure you have the right model for the task
  • • Connected MCPs:Lists all active MCP servers and their tools. If an MCP failed to connect, it appears here
  • • Available tools:All tools that Claude can use in the current session, including native ones (Read, Write, Bash, etc.) and those from MCPs
  • • Settings:Shows working directory, active permissions and other relevant settings
Data/Research

🔍When to Use /status

  • MCP does not work:Run /status to check if the MCP is in the connected list. If not, the problem is in the configuration
  • Different responses than expected:Check out the active model. You can be in Sonnet when you want Opus
  • Tool not recognized:Check if the tool appears in the list. If it does not appear, the MCP that provides it may have failed
  • Login:Run /status upon opening to confirm that everything is configured correctly before starting work
Practical Tip

💡Practical Tip

Combine/statuswith/costto get a complete view of the session. /status shows the "what" (environment, tools, model) and /cost shows the "how much" (tokens, cost, context). Together, they give a complete X-ray.

If an MCP appears as disconnected in /status, try running/doctorto diagnose and attempt to reconnect automatically. /doctor is the mechanic, /status is the instrument panel.

Do vs Avoid

✓ What TO DO

  • ✓ Run /status as the first troubleshooter when something fails
  • ✓ Check active model before critical tasks
  • ✓ Check connected MCPs at the beginning of each session

✗ What NOT to do

  • ✗ Assuming the environment is configured without checking
  • ✗ Ignore disconnected MCPs and try to use their tools anyway
  • ✗ Confusing /status with /cost (they are complementary, not substitutes)

🔄 Combining Commands in Everyday Life

Each individual command has its role. But the real power comes when youcombines into a conscious workflow. Experienced Claude Code operators develop a natural rhythm: they check the environment, work, monitor consumption, optimize the context and clean the screen. This cycle repeats itself naturally throughout a productive session.

Main Concept

🎯The Recommended Flow

This is the flow that experienced traders naturally follow. It is not a rigid rule, but a rhythm that maximizes productivity and minimizes waste:

  • • 1. /status- When opening the session, confirm that the environment is correct (model, MCPs, tools)
  • • 2. Work- Performs tasks normally (10-20 interactions)
  • • 3. /cost- Checks context consumption and usage
  • • 4. /compact- If context > 70%, compact with relevant instructions
  • • 5. /clear- Clears the screen for the next block of work
  • • 6. Repeat- Return to step 2 until you complete the task
Timeline
1

Start of Session

First 30 seconds

Rotate/statusto confirm model and MCPs. If something is wrong, fix it now. Rotate/helpIf you recently updated Claude Code to see new features.

2

Work Block

10-20 interactions

Do your tasks normally. Implement features, debug, refactor. Don't interrupt the flow to check metrics for each message. Let the work flow naturally.

3

Consumption Checkpoint

~30 seconds pause

Rotate/cost. If context > 70%, run/compactwith customized instructions preserving the relevant context. Rotate/clearto clear the screen and start the next block with focus.

Practical Tip

💡Practical Tip

Do not treat these commands as bureaucracy. Treat as operational hygiene. Just like a developer git commits regularly to not lose work, you check /cost and run /compact regularly to not lose context.

If you work with CLAUDE.md (and you should), your persistent instructions are reloaded even after /compact. This means that your design rules, code patterns, and preferences survive compression. /compact only affects chat history, not project settings.

Do vs Avoid

✓ What TO DO

  • ✓ Develop a natural rhythm: work > check > optimize > work
  • ✓ Use /status + /cost as the session "dashboard"
  • ✓ Keep important instructions in CLAUDE.md (survives anything)

✗ What NOT to do

  • ✗ Work entire sessions without ever checking /cost or /status
  • ✗ Interrupt workflow to check metrics for each message
  • ✗ Rely only on automatic compression to manage context
