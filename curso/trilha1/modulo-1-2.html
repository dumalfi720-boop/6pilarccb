📝 /commit - Smart Commit with AI

The command/commitanalyzes all changes in your repository (staged and unstaged), understands what was changed and automatically generates a descriptive commit message. It is not a generic summary: Claude reads the real diff, understands the context of the changes and writes a message that describes the"why"of change, not just the “what”.

Main Concept

🎯Main Concept

/commit is not just a shortcut togit commit -m "...". It runs a complete pipeline: runsgit status, analyzes thegit diff, checks recent commit history to maintain style consistency, and generates a message that follows repository conventions.

  • • Real diff analysis:Claude reads each changed line, understands the intent of the change and generates a message that captures the purpose, not just lists files
  • • Automatic conventions:If the repository already uses Conventional Commits (feat:, fix:, refactor:), Claude detects it and follows the same pattern
  • • Review before confirming:The generated message is presented for you to approve, edit or reject before the commit is executed
  • • Co-authored-by:Automatically adds a Claude co-author tag to the commit message, maintaining traceability
Data/Research

📊What /commit does underneath

  • git status- Identifies modified, new and deleted files (staged and unstaged)
  • git diff- Read the exact content of the changes to understand the context
  • git log- Checks the latest commits to follow the project's messaging style
  • Smart staging- If there are unstaged files, Claude asks which ones you want to include
  • Protection- Never commit sensitive files (.env, credentials) without explicit warning
Practical Tip

💡Practical Tip

/commit generates drastically better messages when changes are atomic. If you did 5 different things in the same batch of files, the message will try to cover everything and be generic. Prefer to commit related changes together:

First implement the feature, run/commit. Then do the style fix, rotate/commitagain. Two clear commits are worth more than one confusing commit. Claude understands each isolated change with much more precision.

Do vs Avoid

✓ What TO DO

  • ✓ Make atomic changes before running /commit
  • ✓ Review the generated message before confirming
  • ✓ Let Claude detect and follow repo conventions

✗ What NOT to do

  • ✗ Accumulate dozens of unrelated changes and run /commit
  • ✗ Accept the message without reading (there may be inaccuracies)
  • ✗ Commit .env files or credentials without checking staging

🔍 /review - Automatic Code Review

O /reviewperforms a complete code review of current changes, analyzing layers: security, bugs, performance and code quality. It works like having a senior reviewer look at your diff before you commit. Each problem found is classified by severity so you know exactly where to focus.

Main Concept

🎯Main Concept

/review is not a linter. He understandssemantics. While a linter checks syntactic rules (identation, semicolons), /review understands what the code does and identifies logical problems, security vulnerabilities, performance bottlenecks and patterns that will cause maintenance headaches.

  • • Layer 1 - Security:SQL injection, XSS, exposed secrets, bypass authentication, unsanitized inputs, excessive permissions
  • • Layer 2 - Bugs:Null pointer exceptions, race conditions, off-by-one errors, unhandled edge cases, impossible states
  • • Layer 3 - Performance:N+1 queries, unnecessary loops, excessive allocations, lack of caching, avoidable re-renders
  • • Layer 4 - Quality:Unclear names, oversized functions, lack of error handling, duplication of logic
Data/Research

📊Severity Classification

  • Critical- Problems that must be corrected before any merge. Security vulnerabilities, bugs that cause data loss, production crashes
  • Important- Problems that significantly affect quality. Degraded performance, incorrect business logic, insufficient error handling
  • Minor- Suggestions for improvement. Better names, simplified logic, refactoring opportunities, missing documentation
Practical Tip

💡Practical Tip

Adopt the rule:Never run /commit in production code without first running /review. The cost of review tokens is insignificant compared to the cost of a bug in production. /review picks up things you don't see because you're immersed in the code.

After receiving the review, you can ask Claude to fix the identified problems directly. Say something like "fix the Critical and Important issues in the review" and it will apply the fixes automatically. Then rotate/reviewagain to confirm that everything has been resolved.

Do vs Avoid

✓ What TO DO

  • ✓ Run /review before /commit in production code
  • ✓ Focus on Critical items first, then Important
  • ✓ Ask Claude to fix the problems found

✗ What NOT to do

  • ✗ Ignore Critical items because "it works in my environment"
  • ✗ Use /review as a fallback for automated testing
  • ✗ Thinking that a clean review means zero bugs (it's a layer, not the only one)

🚀 /pr - Instant Pull Request

O /pranalyzes all commits from your current branch, generates a concise title and a formatted description with Summary and Test Plan, and creates the Pull Request on GitHub automatically viagh CLI. What used to take 5-10 minutes writing description and formatting now takes seconds.

Main Concept

🎯Main Concept

/pr is not a simple wrapper forgh pr create. It analyzes the branch's entire commit history since it diverged from the base branch, understands the complete set of changes, and generates a description that covers everything coherently.

  • • Complete analysis:Read all commits in the branch, not just the last one. Understands the narrative of changes as a whole
  • • Short title:Generates a headline of less than 70 characters that captures the essence of the change
  • • Summary + Test Plan:The description follows a standard format with bullet points of what changed and a checklist on how to test
  • • Automatic creation:Pushing the branch and creating the PR happen in sequence, without manual interaction with GitHub
Timeline

Optimal PR Flow

1

/review

First of all

Run the review to identify and fix problems. Do not open PR with pending Critical issues.

2

/commit

Clean commits

Ensure all changes are committed with clear messages. /pr uses the commit history to generate the description.

3

/pr

Pull Request ready

Claude analyzes the commits, generates title and body, pushes the branch and creates the PR on GitHub. You receive the PR URL in the response.

Alert

⚠️Prerequisite

/pr requires theGitHub CLI (gh)installed and authenticated. Rotategh auth statusto check. If not configured, rungh auth loginfirst. Without the authenticated gh CLI, /pr will fail when trying to create the PR.

Practical Tip

💡Practical Tip

The better your commit messages are, the better the PR description will be. /pr uses commits as the primary source of information. If you used /commit to generate descriptive messages throughout development, the PR practically writes itself.

The default format generated includes## Summarywith bullet points of changes and## Test planwith verification checklist. You can customize by asking Claude to add specific sections before creating the PR.

Do vs Avoid

✓ What TO DO

  • ✓ Follow the /review > /commit > /pr flow in sequence
  • ✓ Check that gh CLI is authenticated before using /pr
  • ✓ Review the generated title and body before confirming creation

✗ What NOT to do

  • ✗ Create PR with uncommitted changes (/pr does not see uncommitted changes)
  • ✗ Skip /review before /pr in production code
  • ✗ Use /pr without checking which branch you are on

🏗️ /init - Project Initialization

O /initautomatically generates a fileCLAUDE.mdfor your project. Claude explores the folder structure, identifies the technology stack, discovers build/test commands, detects code conventions and creates a project memory file that will be loaded in all future sessions.

Main Concept

🎯Main Concept

CLAUDE.md is the equivalent of giving Claude a "project manual". Without it, each new session starts from scratch - Claude needs to re-discover the stack, conventions and patterns. With CLAUDE.md, he already knows how the project works.

  • • Automatic exploration:Claude navigates through the project files, read package.json, Makefile, Dockerfile, configs and understands the stack
  • • Convention detection:Identifies naming patterns, folder structure, code style and frameworks in use
  • • Useful commands:Discover how to run tests, build, lint, deploy and other project commands
  • • Persistence:CLAUDE.md is automatically loaded in every Claude Code session in that directory
Data/Research

📊What /init Detects

  • Languages ​​and frameworks:TypeScript, React, Node.js, Python, Go, Rust and their respective configs
  • Package Managers:npm, yarn, pnpm, pip, cargo and their lockfiles
  • Build/test scripts:package.json scripts, Makefile targets, docker-compose services
  • Folder structure:src/, lib/, tests/, docs/ and how the code is organized
  • Settings:ESLint, Prettier, tsconfig, .env.example and other configs
Practical Tip

💡Practical Tip

/init provides a good starting point, butyou must review and customize the result. Add specific rules that only you know: "never use ORM X", "always use pattern Y for endpoints", "tests must cover scenarios A, B and C". The more context you add to CLAUDE.md, the better all future sessions will be.

You can have CLAUDE.md at different levels: in the project root (project rules), in the home directory (global rules), and in subdirectories (module-specific rules). All are loaded and combined.

Do vs Avoid

✓ What TO DO

  • ✓ Run /init as soon as you start using Claude Code in a project
  • ✓ Review and enrich the CLAUDE.md generated with your preferences
  • ✓ Commit CLAUDE.md to the repository to share with the team

✗ What NOT to do

  • ✗ Accept the generated CLAUDE.md without reading it (may contain inaccuracies)
  • ✗ Put secrets or tokens in CLAUDE.md (it goes to git)
  • ✗ Work on projects without CLAUDE.md (sessions start from scratch every time)

🩺 /doctor - Environmental Diagnosis

O /doctorruns an automatic diagnostic checklist on your Claude Code installation. Checks version, authentication, MCP status, access permissions, dependencies and configurations. When something stops working and you don't know why, /doctor is the first command you should run.

Main Concept

🎯Main Concept

/doctor works like an automated mechanic. It runs a series of checks and, for each problem found, points out the likely cause and suggests the command or action to correct it. It is not a generic diagnosis: it is specific to your environment.

  • • Claude Code version:Checks if you are on the latest version and suggests updating if not
  • • Authentication:Confirms that your session is correctly authenticated with the Anthropic API
  • • MCPs:Tests each configured MCP server, checks if it is responding and identifies connection failures
  • • Permissions:Checks read/write permissions on the required directories
  • • Dependencies:Check if Node.js, git, gh CLI and other dependencies are installed and accessible
Data/Research

🔍Common Problems that /doctor Detects

  • MCP disconnected:An MCP server has silently crashed. /doctor identifies which one and suggests restarting
  • Outdated version:You are using an old version that has no important features or fixes
  • Expired authentication:The API token has expired or been revoked. /doctor points and suggests re-authentication
  • Insufficient permissions:Claude Code cannot read/write necessary directories
  • Missing dependency:A required tool (git, node, gh) is not in the PATH or not installed
Practical Tip

💡Practical Tip

Rotate/doctorwhenever any integration stops working unexpectedly. Before you try to reinstall, reconfigure or search on Google, /doctor will probably identify the problem and give you the solution in seconds.

It is also useful to run /doctor periodically as preventative maintenance. Small problems (like an MCP that fails intermittently) can be detected and resolved before causing disruptions to your workflow.

Do vs Avoid

✓ What TO DO

  • ✓ Run /doctor as the first step in any troubleshooting
  • ✓ Follow the correction suggestions that /doctor provides
  • ✓ Use periodically as preventive maintenance

✗ What NOT to do

  • ✗ Reinstall Claude Code before running /doctor
  • ✗ Ignore /doctor warnings (small problems grow)
  • ✗ Trying to debug MCPs manually without first consulting /doctor

🔗 Complete Flow with Slash Commands

Each individual slash command solves a specific problem. But the real power comes when youchain in a production pipeline. The standard flow/init > desenvolver > /review > /commit > /prtransforms your development workflow into a system where each step automatically adds quality.

Main Concept

🎯The Production Pipeline

Think of slash commands not as isolated commands, but as steps in a production line. Each step receives the output of the previous one and adds value. The final result is revised code, well documented, with clear commits and professional PRs.

  • • /init (once):Creates the project's CLAUDE.md. All future sessions begin with context. Do it on day one and never start from scratch again
  • • Development:Implement the feature or fix normally with Claude. Use the Modulo 1.1 commands (/cost, /compact, /clear) to manage the session
  • • /review:Review everything before committing. Fixes Critical and Important. Ensure quality before commit
  • • /commit:Generates descriptive message that captures the “why” of the change. Atomic and clear commits
  • • /pr:Create the PR with full description, Summary and Test Plan. Push and create in a single step
Complete Pipeline Timeline

Pipeline in Action: Real Example

1

Initialization (one time)

First day on the project

Rotate/initat the root of the project. Review the generated CLAUDE.md, add your rules and preferences, commit to the repo. From now on, every Claude Code session in this project carries this context.

2

Development

Working on the feature/fix

Write code, debug, refactor. Use the commands from Module 1.1 to manage the session:/costto monitor,/compactwhen necessary,/clearfor focus.

3

Review + Commit + PR

Finishing in 3 commands

Rotate/review, fix issues. Rotate/commit, confirm the message. Rotate/pr, receive the URL. Three commands and the work is delivered with professional quality.

4

Troubleshooting (when necessary)

Something went wrong

If any step fails, run/doctorto diagnose. /doctor identifies configuration problems, failing MCPs, expired authentication, and other environmental problems that may prevent other commands from working.

Practical Tip

💡Practical Tip

The sequence/review > /commit > /prit should become as automatic as saving a file. Over time, you internalize the flow and it takes less than 2 minutes from start to PR created. Compare that to the 15-20 minutes of writing commit message, mentally reviewing, writing PR description, and navigating the GitHub interface.

Each step in the pipeline also works in isolation. You can use /review without /commit, or /commit without /pr. The complete pipeline is ideal, but each command adds value individually. Start using one at a time and add others as you feel comfortable.

Do vs Avoid

✓ What TO DO

  • ✓ Internalize the /review > /commit > /pr pipeline as usual
  • ✓ Use /init on the first day and keep CLAUDE.md updated
  • ✓ Combine with Module 1.1 commands for full sessions

✗ What NOT to do

  • ✗ Skip /review thinking your code doesn't need review
  • ✗ Trying to memorize the pipeline instead of gradually practicing it
  • ✗ Abandon the pipeline when a step fails (use /doctor and try again)
