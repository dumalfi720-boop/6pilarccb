⚡ -p (Direct Prompt) - Execution without Interactive Session

The flag-p(or--print) is the gateway to using Claude Code in non-interactive mode. Instead of opening a chat session, you send a single prompt, receive the response, and return to the shell. No REPL, no history, no persistent session. Think of her as acurlfor artificial intelligence.

Main Concept

🎯Main Concept

Claude Code's interactive mode is great for long work sessions. But when you need a specific response, run a script or integrate with other tools, interactive mode gets in the way. The flag-presolve this: execute, respond and exit.

  • • Single execution:Sends the prompt, receives the response on stdout and the process ends. There is no session to manage
  • • Compatible with pipes:You can feed Claude with data via stdin using pipe, making it part of any Unix pipeline
  • • Ideal for automation:Cron jobs, CI/CD scripts, git hooks, pre-commit checks, everything works with -p
  • • Stateless:Each execution is independent. There is no memory between calls. If you need context, pass everything in the prompt or via pipe
Code block

💻Practical Examples

Simple direct prompt:

claude -p "explique o que faz o comando chmod 755"

Combining with pipe to parse a file:

cat src/api/auth.py | claude -p "encontre vulnerabilidades de seguranca neste codigo"

Generating code and saving directly to a file:

claude -p "gere um Dockerfile para uma app Node.js com pnpm" > Dockerfile

Pipeline with git diff for automatic review:

git diff --staged | claude -p "faca code review dessas mudancas. seja direto e aponte problemas"
Practical Tip

💡Practical Tip

The flag-pdoes not read CLAUDE.md from the directory automatically in all situations. If you need project instructions to be respected in calls with -p, pass the content via pipe or include it in the prompt. For more robust automations, consider using--system-promptalong with -p.

Create aliases in your.bashrc ou .zshrcfor commands you run frequently. Example:alias cr='git diff --staged | claude -p "code review direto e conciso"'

Do vs Avoid

✓ What TO DO

  • ✓ Use -p for specific tasks: explain, generate, convert, analyze
  • ✓ Combine with pipes to feed Claude with real data
  • ✓ Integrate into CI/CD scripts and git hooks for automation

✗ What NOT to do

  • ✗ Use -p for tasks that require multiple iterations (open interactive session)
  • ✗ Expect -p to remember previous calls (each execution is isolated)
  • ✗ Send huge files via pipe without limiting the size (consumes unnecessary tokens)

🧩 --model - Choosing the Right Model

The flag--modelallows you to select which AI model will process your instructions. Each model has a distinct capacity, speed and cost profile. Choosing the right model for each task is the difference between spending cents and spending dollars for the same result.

Main Concept

🎯Main Concept

Claude Code supports three main families of models. The choice directly impacts the quality of the response, execution speed and cost per token. There is no "best" model at all. There is the right model for each situation.

  • • Opus (opus):The most capable. Ideal for architectural decisions, complex debugging, refactoring entire systems and tasks that require deep reasoning. Slower and more expensive
  • • Sonnet (sonnet):The balanced one. Covers 80% of day-to-day use cases: feature implementation, standard debugging, code review, documentation. Good speed at a reasonable cost
  • • Haiku (haiku):Fast and cheap. Perfect for simple tasks: formatting, direct questions, generating boilerplate, trivial conversions. Almost instantaneous response with minimal cost
Code block

💻Syntax and Examples

Start interactive session with Opus:

claude --model opus

Quick prompt with Haiku (cheap and instant):

claude -p "converta esse JSON para YAML" --model haiku < config.json

Sonnet for daily development:

claude --model sonnet
Data/Research

📊When to Use Each Model

  • Opus:Architectural redesign, decisions that affect the entire project, debugging bugs that no one finds, in-depth analysis of trade-offs
  • Sonnet:Implement endpoints, create components, write tests, do code review, refactor individual functions
  • Haiku:Format JSON/YAML, generate regex, answer syntax questions, create boilerplate, rename variables in bulk
  • General rule:Start with Sonnet. Step up to Opus when the task requires deep thinking. Switch to Haiku when you're a mechanic
Practical Tip

💡Practical Tip

You can configure the default model in the Claude Code configuration file so you don't need to go through--modelevery time. But keep in the habit of consciously changing models depending on the task. Using Opus to format JSON is like using a truck to go to the bakery.

Within an interactive session, you can change the model using/model. This allows you to start a task with Sonnet and, when you encounter a complex problem, switch to Opus without leaving the session.

Do vs Avoid

✓ What TO DO

  • ✓ Choose the model based on the complexity of the task, not on habit
  • ✓ Use Haiku in repetitive automations to save money
  • ✓ Reserve Opus for moments where maximum quality is necessary

✗ What NOT to do

  • ✗ Always use Opus for everything (expensive and unnecessary in most cases)
  • ✗ Use Haiku for tasks that require deep reasoning (you will make mistakes and cost more when redoing)
  • ✗ Ignore the existence of --model and always use the default without thinking

🔄 --resume - Resuming Previous Sessions

The flag--resumeallows you to resume a previous session exactly where you left off. All context, decisions made, files discussed and task status are preserved. And how to close your laptop on Friday and reopen it on Monday with everything in place.

Main Concept

🎯Main Concept

Each Claude Code session receives a unique ID. When you close a session (Ctrl+C or /exit), it is not destroyed. The history is stored locally and can be recalled at any time. THE--resumereload this complete history.

  • • No argument: claude --resumeresumes the most recent session automatically
  • • With specific ID: claude --resume SESSION_IDresumes a specific session, even if it is not the most recent
  • • Preserved context:All messages, decisions, files read and modified are in the history. Claude returns with complete knowledge of what was done
  • • Context Limit:Very long sessions may need /compact after resume, as the entire history is reloaded in the context window
Code block

💻Practical Examples

Resume the most recent session:

claude --resume

Resume a session specified by ID:

claude --resume abc123-def456-ghi789

List previous sessions to choose which one to resume:

claude --resume  # sem ID, mostra lista de sessoes recentes para selecionar

Resume and continue with direct prompt:

claude --resume -p "continue implementando os testes que faltaram"
Data/Research

🔍Usage Scenarios

  • Multi-day projects:A feature started on Monday, resumes on Tuesday without losing architectural context and decisions
  • Interruptions:Lunch, meeting, end of the day. --resume ensures that you return exactly where you left off
  • Long debugging:When a bug takes hours to track down, being able to close and reopen without starting over from scratch is essential
  • Revision:Revisit an old session to review decisions that were made and understand the reasoning
Practical Tip

💡Practical Tip

Before closing a session that you intend to resume, tell Claude something like "I'm going to pause here. When I come back, continue from the payments module integration tests." This creates a clear landmark in the history that facilitates resumption.

If the resumed session has a very full context, run/compact foco no estado atual da tarefa e proximos passosright after summarizing it. This clears the old history but preserves what is important to continue.

Do vs Avoid

✓ What TO DO

  • ✓ Use --resume for projects that last more than one session
  • ✓ Leave a "milestone" before closing the session to facilitate resumption
  • ✓ Run /compact after resume if the session was long

✗ What NOT to do

  • ✗ Start a new session for each small task in the same project
  • ✗ Resuming very old sessions without checking whether the context is still relevant
  • ✗ Assume that --resume works between different machines (history and local)

🔒 --allowedTools - Tool Control

The flag--allowedToolsrestricts which tools Claude can use during the session. Instead of having full access to Bash, Read, Write, Edit and all MCP tools, you define a specific subset. And the principle of least privilege applied to AI.

Main Concept

🎯Main Concept

By default, Claude Code has access to a wide set of tools: executing commands in the terminal (Bash), reading files (Read), writing files (Write), editing files (Edit), searching the web and all configured MCP tools. In many scenarios, you want to limit this access.

  • • Security:In production environments, you may want Claude to only read files, without executing commands or modifying anything.
  • • Predictability:In automations, restricting tools ensures that Claude only does what is expected, without surprises
  • • Audit:When you need to prove that an AI only read data and did not modify anything, --allowedTools is the evidence
Code block

💻Practical Examples

Read only (Read and Grep, without execution or writing):

claude --allowedTools Read,Grep -p "analise a estrutura deste projeto"

Reading and writing, but without Bash (does not execute commands):

claude --allowedTools Read,Write,Edit,Grep,Glob

Bash only for below operations:

claude --allowedTools Bash -p "verifique o status dos containers docker"

Include specific MCP tools:

claude --allowedTools Read,Grep,mcp_github
Alert

⚠️Attention

If you restrict it too much, Claude may not be able to complete the task. For example, if you ask it to "fix a bug" but only allow Read, it will identify the problem but will not be able to edit the file. Think of tools as permissions: sufficient access for the task, no more, no less.

Practical Tip

💡Practical Tip

For CI/CD automation, use the combination--allowedTools Read,Grep -pfor analysis tasks (code review, standards verification, auditing). This ensures that Claude never changes anything in the pipeline, only analyzes and reports.

You can use/statuswithin the session to check which tools are available. If a tool does not appear, it is because it was restricted by --allowedTools.

Do vs Avoid

✓ What TO DO

  • ✓ Restrict tools in automations and CI/CD pipelines
  • ✓ Use read-only mode for audits and analysis
  • ✓ Document which tools each automation script uses

✗ What NOT to do

  • ✗ Restrict tools in interactive development sessions (disrupts the flow)
  • ✗ Request tasks that require tools you have locked
  • ✗ Ignore --allowedTools in production environments for "practicality"

📤 --output-format - Structured Output Format

The flag--output-formatdefines how Claude Code formats its response. Instead of free text for humans to read, you can request output in structured JSON or streaming JSON. This transforms Claude Code from a conversational assistant into a programmatic component of any pipeline.

Main Concept

🎯Main Concept

There are three output formats available. Each serves a different use case:

  • • text (default):Free text formatted for human reading. Includes markdown, code blocks, lists. And what do you normally see in the terminal
  • • json:Output structured in JSON. Ideal for automations that need to parse the response programmatically. The entire result comes as a valid JSON object
  • • stream-json:Streaming JSON, where each event (generated text, tool call, result) comes as a separate JSON line. Perfect for real-time UIs that show generation progress
Code block

💻Practical Examples

JSON output for automatic processing:

claude -p "liste os 5 maiores arquivos deste projeto" --output-format json

JSON + jq to extract specific fields:

claude -p "analise este codigo" --output-format json | jq '.result'

Stream JSON for real-time interfaces:

claude -p "implemente a funcao de login" --output-format stream-json

Complete pipeline: analysis + parse + action:

cat src/*.ts | claude -p "encontre funcoes sem tipagem" \
  --output-format json --model haiku | jq '.result' > report.txt
Data/Research

📊Format Comparison

  • text:Human readable, not reliably parseable. Use when you are going to read the result in the terminal
  • json:Parseable, structured, arrives complete. Use in scripts that process the entire response at once
  • stream-json:Parseavel, in real time, arrives incrementally. Use in dashboards, web UIs, or apps that show progress
  • Rule:If a human is going to read it, use text. If a machine is going to process it, use json. If a UI is going to display in real time, use stream-json
Practical Tip

💡Practical Tip

Combine--output-format jsonwithjqto create powerful pipelines. jq is a JSON processor on the command line that allows you to accurately filter, transform and extract data from Claude's output.

For integrations with Node.js or Python, the json format allows you to doJSON.parse() ou json.loads()directly in the output, without the need for regex or manual text parsing.

Do vs Avoid

✓ What TO DO

  • ✓ Use json in all automation that needs to process the response
  • ✓ Combine with jq to extract exactly the field you need
  • ✓ Use stream-json when you need to show progress to the user

✗ What NOT to do

  • ✗ Try parsing text output with regex (fragile and unpredictable)
  • ✗ Use json in the interactive session (the output is unreadable for humans)
  • ✗ Ignore --output-format and do manual text parsing in automations

🧬 Combining Flags - Advanced CLI Recipes

Each individual flag has its value. But the real power comes when youcombines multiple flagsin a single command. These combinations create reusable recipes for specific tasks: automatic review, on-demand documentation, security auditing, and more.

Main Concept

🎯Ready Recipes

These are tested combinations that solve real everyday problems. Copy, adapt and create aliases for the ones you use most.

  • • Quick and cheap review:Combines -p + --model haiku + git diff pipe for instant and cost-effective code review
  • • Deep analysis:Combines -p + --model opus + --output-format json for high-quality structured parsing
  • • Secure audit:Combines --allowedTools Read,Grep + -p + --output-format json for read-only audits with parseable output
  • • Resumption with specific model:Combine --resume + --model opus to resume a session with the most powerful model
Code block with recipes

💻Complete Recipes

Automatic code review (fast and cheap):

git diff --staged | claude -p "code review direto. aponte bugs, \
  problemas de performance e sugestoes" --model haiku

Generate structured documentation in JSON:

cat src/api/*.ts | claude -p "documente todas as funcoes exportadas \
  com parametros, retorno e descricao" \
  --model sonnet --output-format json > docs/api.json

Read-only security audit:

claude -p "audite este projeto por vulnerabilidades de seguranca" \
  --allowedTools Read,Grep --model opus --output-format json \
  > security-audit.json

Resume session with different model:

claude --resume --model opus

Alias ​​in .zshrc for everyday use:

# Code review rapido
alias cr='git diff --staged | claude -p "code review conciso" --model haiku'

# Explicar arquivo
alias explain='claude -p "explique o que este arquivo faz" --model sonnet'

# Gerar testes
alias gentest='claude -p "gere testes unitarios para este arquivo" --model sonnet'
Revenue timeline by scenario
1

For CI/CD Scripts

Pipeline automation

Use-p + --output-format json + --allowedTools Read,Grep + --model haiku. It guarantees fast execution, parsable output, no modifications to the repo and minimum cost. The JSON can be processed by the next step in the pipeline.

2

For Daily Development

Interactive workflow

Use--model sonnetfor the default interactive session. When you encounter something complex, switch to--model opusvia /model or restart with --resume --model opus.

3

For Documentation and Reports

Structured content generation

Use-p + --model sonnet + --output-format jsonwith source file pipe. Process the JSON with jq or your favorite script to generate markdown, HTML or any desired output format.

Practical Tip

💡Practical Tip

Create a directory~/scripts/claude/with shell scripts for your favorite recipes. Each script encapsulates a combination of flags for a specific use case. This is more flexible than aliases because it allows for conditional logic, variables, and composition.

Reusable script example: areview.shwhich accepts the branch as an argument, does diff, sends it to Claude with --model haiku, formats the output with --output-format json and saves the report in a dated file.

Do vs Avoid

✓ What TO DO

  • ✓ Create aliases and scripts for combinations you use frequently
  • ✓ Document your recipes for the team to reuse
  • ✓ Test new recipes on test repos before using in production

✗ What NOT to do

  • ✗ Memorize flag combinations without creating aliases (you'll forget the syntax)
  • ✗ Use complex combinations without testing first (debugging is difficult with multiple flags)
  • ✗ Hardcoding session IDs in scripts (they change every session)
