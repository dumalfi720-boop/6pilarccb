===== MODULE 1.1 =====
1.1 ~25 min

⌨️ Essential Terminal Commands

Claude Code's internal commands that you will use every day. /help, /clear, /compact, /cost, /status and how to combine them.

Topic 1
What is it:

Command that displays all available commands, slash commands and Claude Code flags. And your quick reference manual within the session.

Why learn:

When you forget a command or want to discover new features, /help is the first place to look. Avoid having to leave the terminal to consult documentation.

Key concepts:

Type /help at any time in the session. Shows internal commands, skill slash commands, and CLI flags. It works as a living index of what Claude Code can do.

Topic 2
What is it:

Command that clears the visual history of the current session in the terminal. It doesn't erase Claude's context, just the display on the screen.

Why learn:

Long sessions become visually polluted. /clear takes a breather without losing the thread of the conversation. You maintain context but gain visual clarity.

Key concepts:

Unlike /compact, /clear only affects what appears in the terminal. Claude keeps remembering everything. Use when the screen is full but the context is still relevant.

Topic 3
What is it:

Command that compresses the conversation history, summarizing previous messages to free up space in the context window. Essential for long sessions.

Why learn:

The context window has a limit. When you reach ~70-80% usage, Claude starts to lose details. /compact summarizes the history and frees up space to continue working without losing critical information.

Key concepts:

Use /compact before long migrations or large refactorings. Monitor with /cost to know when to compress. Claude generates an intelligent summary that preserves important decisions and context.

Topic 4
What is it:

Command that shows the accumulated cost of the current session in dollars, number of tokens consumed and usage statistics. Your consumption meter in real time.

Why learn:

Without monitoring costs, you can burn credits without realizing it. /cost gives you full visibility into how much you are spending and helps you decide when to compact or start a new session.

Key concepts:

Shows entry, exit and cost tokens in USD. Use as a trigger for /compact when the context gets large. Professional operators check /cost regularly to optimize usage.

Topic 5
What is it:

Shows information about the current session: model in use, connected MCPs, available tools and active settings. Your environmental control panel.

Why learn:

When something doesn't work as expected, /status helps diagnose it. Perhaps an MCP is disconnected or the wrong model is selected. And the first troubleshooting command.

Key concepts:

Displays active model, connected MCPs, tools, and permissions. Use before critical tasks to ensure the environment is correct. Complements /cost for a complete view of the session.

Topic 6
What is it:

The practice of using internal commands together during a work session. /status at the beginning, /cost throughout, /compact when necessary, /clear to maintain visual focus.

Why learn:

Isolated commands are useful. Commands combined in a conscious flow are powerful. You develop a work rhythm that maximizes productivity and minimizes waste.

Key concepts:

Recommended flow: /status (check environment) > work > /cost (monitor) > /compact (when necessary) > /clear (clear visual). Repeat as the session progresses.

===== MODULE 1.2 =====
1.2 ~25 min

⚡ Native Slash Commands

Claude Code action commands: /commit, /review, /pr, /init and /doctor. Direct execution without typing long prompts.

What is it:

Parses all staged/unstaged changes, generates a descriptive commit message automatically, and runs git commit. Eliminates the friction of writing commit messages.

Why learn:

Bad commit messages are an epidemic. Claude analyzes the actual diff and generates messages that describe the "why" of the change, not just the "what". Clearer commits, more useful history.

Key concepts:

Claude reads the diff, analyzes staged + unstaged, generates a message in the repo pattern and executes it. You can review before confirming. Follow conventions such as Conventional Commits if the repo already uses it.

What is it:

Review your current changes looking for bugs, security vulnerabilities, performance issues and code quality. How to have a senior reviewer available 24/7.

Why learn:

No one should push without a review. When you work alone or need quick feedback, /review catches things you didn't see: race conditions, SQL injection, unnecessary imports.

Key concepts:

Analyzes changes by layers: security, bugs, performance and quality. Classifies as Critical, Important, Minor. Use /commit before all production code.

What is it:

Analyzes all branch commits, generates formatted title and description and creates Pull Request on GitHub automatically via gh CLI.

Why learn:

Creating PRs is tedious. Writing good descriptions takes time. /pr does this in seconds, parsing the entire diff and generating a summary that another dev can immediately understand.

Key concepts:

Requires gh CLI installed and authenticated. Analyzes all commits (not just the last one). Generates Summary + Test Plan. You review before publishing. Ideal flow: /review > /commit > /pr.

What is it:

Automatically generates a CLAUDE.md for the current project. Claude explores the structure, identifies stack, commands and conventions, and creates the project memory file.

Why learn:

CLAUDE.md is the foundation of Claude Code (entire Track 2). /init creates this file automatically, saving time and ensuring the essential points are covered.

Key concepts:

Explores the repo, detects language, framework, tests, build scripts. Generates CLAUDE.md with overview, stack, commands and conventions. You must review and customize the result.

What is it:

Checks the health of the Claude Code installation: version, authentication, MCPs, permissions and dependencies. Your automatic troubleshooting checklist.

Why learn:

When something doesn't connect, an MCP fails, or a command doesn't work, /doctor quickly diagnoses the problem. Avoid hours of manual debugging in configuration.

Key concepts:

Check Claude version, Node.js, API authentication, installed MCPs and their connections. Points out problems with correction suggestions. Run when any integration stops working.

What is it:

The complete workflow using slash commands in sequence: /init (setup) > code > /review > /commit > /pr. The professional workflow with Claude Code.

Why learn:

Isolated slash commands are shortcuts. Slash commands in sequence are a production system. You build a quality pipeline that ensures code is reviewed, committed and published correctly.

Key concepts:

Default pipeline: /init (once per project) > develop > /review (before committing) > /commit (with IA message) > /pr (when ready). Each step adds quality.

===== MODULE 1.3 =====
1.3 ~25 min

🚩 Command Line Flags

Full control of Claude via CLI flags: -p for direct prompts, --model for choosing model, --resume for resuming sessions and more.

What is it:

Flag that allows you to send a prompt directly to Claude without opening an interactive session. Ideal for scripts, automations and quick one-line tasks.

Why learn:

You don't always want to open a full session. With -p you can integrate Claude into shell scripts, cron jobs and CI/CD pipelines.

Key concepts:

Syntax: claude -p "your prompt here". Executes and returns results to stdout. Composable with pipe: cat file.py | claude -p "explain this code". Does not maintain session.

What is it:

Flag that allows you to select which Claude model to use: Opus (more capable), Sonnet (balanced) or Haiku (fast and cheap).

Why learn:

Different tasks require different models. Using Opus for everything is waste. Use Haiku for complex and insufficient architecture. Knowing how to choose saves money and time.

Key concepts:

Opus: architecture, complex decisions. Sonnet: daily development, 80% of cases. Haiku: quick tasks, formatting, simple questions. Syntax: claude --model sonnet.

What is it:

Flag that resumes a previous session of Claude Code with all context preserved. You continue exactly where you left off.

Why learn:

Long sessions are interrupted by updates, restarts or end of business hours. With --resume you do not lose the accumulated context. Essential for projects that last days.

Key concepts:

Syntax: claude --resume. Resume the last session. You can specify ID: claude --resume SESSION_ID. The context, decisions and files discussed are preserved.

What is it:

Flag that restricts which tools Claude can use in the session. Granular permissions control for security and predictability.

Why learn:

In production environments or when you want to limit what Claude can do (e.g. read only, without running bash), this flag gives the necessary control.

Key concepts:

Allows you to list explicitly allowed tools. Useful in automations where you want to ensure that Claude only uses Read and Grep, for example, without modifying files.

What is it:

Flag that defines Claude's output format: plain text, JSON or streaming. Essential for programmatic integrations.

Why learn:

When Claude is part of an automated pipeline, you need parseable output. Structured JSON allows other scripts to process Claude's output.

Key concepts:

Options: text (default), json, stream-json. For automations use json. For real-time UI use stream-json. Combinable with -p for full pipelines.

What is it:

The art of combining multiple flags to create powerful and specific commands. Each combination unlocks a different use case.

Why learn:

Individual flags are useful. Combined flags create real automations: automatic review scripts, documentation pipelines, on-demand codebase reviews.

Key concepts:

Examples: claude -p "..." --model haiku --output-format json (quick parsing with parsable output). claude --resume --model opus (resume with more capable model). Create aliases for your favorite combinations.

🔀 Data Pipe

Feed Claude with files, logs and stack traces via Unix pipe. Integrate with terminal tools for powerful flows.

What is it:

Technique for sending the contents of a file directly to Claude via stdin. The Unix pipe connects the output of cat with the input of Claude.

Why learn:

The fastest way to make Claude analyze a file without opening an interactive session. Works with any type of text file: code, logs, configs, CSV.

Key concepts:

Syntax: cat app.py | claude -p "explain this code". The contents of the file become part of the prompt. Works with any command that generates output: grep, find, git diff.

What is it:

Send error logs, stack traces or application output directly to Claude for analysis and automatic diagnosis.

Why learn:

Copying and pasting stack traces manually is slow and error-prone. With pipe, you send the entire log directly from the terminal. Claude receives the full context without anything cut out.

Key concepts:

Examples: cat error.log | claude -p "what is causing these errors?". Works with tail -f for realtime logs. Combine with grep to filter before sending.

What is it:

Technique of concatenating multiple files and sending everything to Claude at once. Useful when the context depends on several related files.

Why learn:

Many bugs only make sense when you see multiple files together: the controller, the model and the route. Sending them all at once gives Claude the full context.

Key concepts:

Syntax: cat model.py controller.py routes.py | claude -p "find inconsistencies". Use separators or file headers so Claude knows which file is which.

What is it:

Use redirection operators (> and >>) to save Claude's output to files. Transforms Claude's responses into permanent artifacts.

Why learn:

Claude's responses in the terminal are ephemeral. By redirecting to archive, you create documentation, reports and analyzes that can be shared or versioned.

Key concepts:

Syntax: claude -p "..." >result.md. Use >> to append. Combine with --output-format json > data.json for automated pipelines with structured output.

What is it:

Use Claude as another tool in the Unix pipeline, combining with grep, awk, sed, jq and other tools to create intelligent processing flows.

Why learn:

The Unix philosophy is to compose small, specialized tools. Claude adds intelligence to this pipeline. grep filters, awk formats, Claude understands and parses.

Key concepts:

Examples: git diff | claude -p "summarize these changes". find . -name "*.py" -exec cat {} + | claude -p "find vulnerabilities". Claude works like any other Unix command.

What is it:

Advanced multi-stage pipe combinations: filtering, Claude processing, formatting and output. Real automations for everyday life.

Why learn:

This is the operator level. You build scripts that automatically perform complete analysis: PR review, API documentation, security audit across the entire codebase.

Key concepts:

Chain Claude in shell scripts: for file in src/*.py; from cat $file | claude -p "analyse" --output-format json >> report.json; done. Combine with cron for scheduled execution.

🎯 Keyboard Shortcuts and Aliases

Quick navigation, prompt editing, execution control, and how to create custom aliases for your favorite combinations.

What is it:

Keyboard shortcuts to navigate session history, scroll up/down, and access previous responses without using the mouse.

Why learn:

In long sessions, you need to review previous answers. Navigating with a keyboard is faster than manual scrolling and keeps your hands on the keyboard.

Key concepts:

Up/down arrows for input history. Page Up/Down for scrolling. Ctrl+Home/End to start/end the session. These shortcuts follow standard terminal conventions.

What is it:

Shortcuts for editing the prompt before sending: move cursor, select text, delete words and lines. Efficient editing without a mouse.

Why learn:

Long prompts need editing. Ctrl+A (start), Ctrl+E (end), Ctrl+W (delete word), Alt+Backspace. Saves seconds that add up to hours.

Key concepts:

Ctrl+A: start of line. Ctrl+E: end of line. Ctrl+W: delete previous word. Ctrl+U: clear entire line. Ctrl+K: delete to the end. These are standard readline shortcuts.

What is it:

Shortcuts to control Claude's execution: interrupt response in progress, cancel operation and exit the session cleanly.

Why learn:

If Claude started generating something wrong, you don't need to wait for it to finish. Stop, correct the prompt, and continue. Control and productivity.

Key concepts:

Escape: interrupts the current generation. Ctrl+C: cancels and returns to the prompt. Ctrl+D: exit the session. Use Escape when the answer is going the wrong way to save tokens.

What is it:

Browse the history of prompts you've already typed in the session, reusing or modifying previous commands without typing again.

Why learn:

You often send variations of the same prompt. With history, you retrieve the previous prompt, make a small adjustment, and resend. Much faster than retyping.

Key concepts:

Up arrow: previous prompt. Down arrow: next prompt. Works like bash history. Combine with prompt editing to quickly iterate on a task.

What is it:

Technique for writing multi-line prompts in Claude Code without accidentally sending them in the middle. Essential for complex, structured prompts.

Why learn:

Good prompts often need structure: context on one line, instruction on another, constraints on a third. Multi-line allows for organized and more effective prompts.

Key concepts:

Shift+Enter: new line without sending. Backslash at the end of the line: continuation. Allows you to write structured prompts that Claude interprets better.

What is it:

Create aliases in the shell (bash/zsh) that encapsulate Claude commands with flags, prompts and settings that you use frequently.

Why learn:

If you type "claude -p --model haiku --output-format json" every time, an alias turns it into "cq" (claude quick). Automate what you repeat.

Key concepts:

Add to .bashrc/.zshrc: alias cr="claude --resume", alias cq="claude -p --model haiku", alias crev="claude -p --model sonnet 'review this code'". Reload with source ~/.zshrc.
