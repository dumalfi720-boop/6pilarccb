📄 cat | claude -p - Feeding Files Directly

The commandcat file | claude -p "instruction"It is the most direct way to send the contents of any file to Claude without opening an interactive session. The Unix pipe connects the output of thecatwith Claude's entry, and the flag-p(print) causes Claude to process and print the response directly to the terminal. Zero Africa.

Main Concept

🎯Main Concept

In Unix, the pipe (|) takes the standard output (stdout) of one command and connects it to the standard input (stdin) of the next. When you docat app.py | claude -p "explique", the entire contents of the app.py file are injected as context for Claude to process. It receives everything at once, no truncation, no manual upload.

  • • stdin as context:Claude reads everything that arrives via stdin and treats it as part of the prompt. You don't need to paste the content manually
  • • Flag -p (print):Runs in non-interactive mode. Claude processes, responds and leaves. No session, no REPL. Ideal for automations and scripts
  • • Any text file:Works with code (.py, .js, .ts), logs, configs (.yaml, .toml, .env), CSVs, markdown, and anything that cat can print
Code Examples

⌨️Real Examples in the Terminal

# Explicar um arquivo Python
$ cat app.py | claude -p "explain what this code does, function by function"

# Analisar um arquivo de configuracao
$ cat docker-compose.yml | claude -p "is this compose correct? Are there any security issues?"

# Revisar um CSV
$ cat data.csv | claude -p "summarize this data: how many rows, which columns, outliers"

# Analisar um Dockerfile
$ cat Dockerfile | claude -p "optimize this Dockerfile for layer reduction and final size"

# Verificar um .env de exemplo
$cat.env.example | claude -p "which variables are missing for production?"
Data/Research

📊When to Use cat | claude -p

  • Quick analysis:Need to understand a file you've never seen before? Pipe and ask
  • Code review solo:No pair? Let Claude review before committing
  • Documentation:Generate docstrings, READMEs or comments from source code
  • Conversion:Transform YAML to JSON, CSV to markdown table, or any format to another
Practical Tip

💡Practical Tip

The flag-pand the secret to making Claude scriptable. Without it, Claude opens an interactive session that expects input from the keyboard. With-p, it processes and exits, which means you can chain it with other commands, redirect to files and use it in shell scripts.

If the file is very large (thousands of lines), consider filtering first withhead -100 arquivo.py | claude -p "..."to send just the beginning, or usegrep -n "def " app.py | claude -p "..."to send only function signatures.

Do vs Avoid

✓ What TO DO

  • ✓ Use -p for one-shot operations (without interactive session)
  • ✓ Be specific in the prompt: "explain function by function" > "explain"
  • ✓ Filter large files before uploading (head, grep, sed)

✗ What NOT to do

  • ✗ Send binaries (images, PDFs, executables) via cat pipe
  • ✗ Send files with secrets (.env with real keys) to external APIs
  • ✗ Forgetting the -p flag and getting stuck in an interactive session within a script

🐛 Pipe with Logs and Stack Traces - Instant Debug

When your app explodes at 3am and you have a200 line stack tracein the log, the last thing you want is to copy and paste into the browser. Just send the entire log directly to Claude via pipe. It gets the full context, with nothing cut out, and identifies the root cause in seconds.

Main Concept

🎯Main Concept

Logs and stack traces are structured text. Claude is exceptional at interpreting error patterns, tracing the chain of calls and suggesting corrections. The advantage of the pipe over pasting in the web interface: you send thefull log without truncation, and Claude sees every line, every timestamp, every severity level.

  • • Full context:Nothing is cut. Claude sees the complete traceback, including the frame that caused the error and the entire call stack
  • • Temporal patterns:With timestamps in the log, Claude identifies error cascades, timeouts and race conditions
  • • Multiple errors:If the log has several distinct errors, Claude groups them and prioritizes them by severity
Code Examples

⌨️Real Examples in the Terminal

# Enviar log de erro completo
$ cat /var/log/app/error.log | claude -p "what is causing these errors? suggest a correction"

# Ultimas 50 linhas do log (erros recentes)
$ tail -50 /var/log/nginx/error.log | claude -p "diagnose these nginx errors"

# Filtrar apenas erros antes de enviar
$ grep -i "error\|exception\|fatal" app.log | claude -p "group by type and suggest corrections"

# Stack trace do Python em tempo real
$ python app.py 2>&1 | claude -p "what went wrong and how do I fix it?"

# Logs do Docker de um container
$ docker logs api-container --tail 100 2>&1 | claude -p "why is this container restarting?"

# Combinar tail -f com timeout para capturar logs ao vivo
$ timeout 10 tail -f /var/log/syslog | claude -p "anything unusual in the last 10 seconds?"
Practical Tip

💡Practical Tip

Use2>&1to capture both stdout and stderr. Many errors only appear on stderr, and without this redirection the pipe only takes stdout. Example:npm run build 2>&1 | claude -p "o que quebrou?"

For very large logs, filter with grep first and output only the relevant lines. A log of 10,000 lines where only 50 are errors does not need to be complete. The pre-filter saves tokens and improves the quality of the analysis.

Do vs Avoid

✓ What TO DO

  • ✓ Use 2>&1 to capture stderr along with stdout
  • ✓ Filter with grep before sending very large logs
  • ✓ Ask Claude to suggest a fix in addition to diagnosing

✗ What NOT to do

  • ✗ Send gigabytes of log without filtering (wastes tokens and generates noise)
  • ✗ Forget the 2>&1 and lose the errors that are on stderr
  • ✗ Send logs with exposed tokens, passwords, or API keys

📚 Multiple Files - Concatenating Context

Bugs that only make sense when you see themmultiple files togetherThey are the most difficult to debug alone. A model imports from a place, a route calls a controller, the controller uses a service. Concatenating everything into a single pipe allows Claude to see the entire stream and find inconsistencies that you wouldn't notice by looking file by file.

Main Concept

🎯Main Concept

The commandcataccepts multiple files as arguments and concatenates them in sequence at the output. When you docat a.py b.py c.py | claude -p "...", Claude receives the contents of the three files as a single block of continuous text.

  • • Cross view:Claude can check whether the function signatures used in one file match the definitions in another
  • • Inconsistencies detection:Wrong types, missing imports, different variable names between model and controller
  • • File separators:To help Claude know which content came from which file, use echo with separators between cats
Code Examples

⌨️Real Examples in the Terminal

# Concatenacao simples de multiplos arquivos
$ cat model.py controller.py routes.py | claude -p "find inconsistencies between these 3 files"

# Com separadores para o Claude saber qual arquivo e qual
$ (echo "=== model.py ===" && cat model.py && echo "=== controller.py ===" && cat controller.py && echo "=== routes.py ===" && cat routes.py) | claude -p "analyze the integration between these files"

# Todos os arquivos .py de um diretorio
$ cat src/*.py | claude -p "do a general code review of this Python project"

# Schema do banco + models da aplicacao
$ cat schema.sql models.py | claude -p "are the models consistent with the SQL schema?"

# Package.json + tsconfig para verificar consistencia
$ cat package.json tsconfig.json | claude -p "Are the TS dependencies and config consistent?"
Practical Tip

💡Practical Tip

The separator trick makes a huge difference in the quality of the response. When Claude receives=== model.py ===between the contents, it knows exactly where each file begins and ends. Without separators, he tries to guess from the context, and doesn't always get it right.

For larger projects, create an alias in your shell:alias catfiles='for f in "$@"; do echo "=== $f ===" && cat "$f"; done'. Then usecatfiles model.py routes.py | claude -p "...".

Alert

⚠️Pay attention to the total size

Claude's context window has a limit (200k tokens in Sonnet, more in Opus). Concatenating 50 files of 500 lines each may exceed the limit. Calculate: every 4 characters is equivalent to approximately 1 token. A 1000 line file with 80 characters per line consumes around 20k tokens. Please monitor the total size before shipping.

Do vs Avoid

✓ What TO DO

  • ✓ Use separators (echo "=== file ===") between contents
  • ✓ Concatenate only the files relevant to the analysis
  • ✓ Check total size before sending (wc -l *.py)

✗ What NOT to do

  • ✗ Concatenate entire repositories without filtering (overflows context)
  • ✗ Send without separators and hope Claude guesses the limits
  • ✗ Include node_modules, .git or other files generated in cat

💾 Redirecting Output - Saving Claude Results

So far, Claude's answer appears on the screen and disappears when you close the terminal. With redirect operators> e >>, you transform ephemeral responses intopermanent artifacts: markdown files, JSON, reports, documentation. Claude generates, the shell saves.

Main Concept

🎯Main Concept

Redirection operators are fundamental in Unix and combine perfectly with Claude in-p. THE>creates or overwrites a file with the output. THE>>adds (append) to the end of an existing file without deleting the previous content.

  • • > (overwrite): claude -p "..." > resultado.mdcreates the file from scratch, overwriting it if it already exists
  • • >> (append): claude -p "..." >> relatorio.mdadds it to the end of the file, preserving the previous content
  • • --output-format:Use--output-format jsonto obtain structured JSON output, ideal for programmatic processing
Code Examples

⌨️Real Examples in the Terminal

# Gerar documentacao e salvar em markdown
$ cat src/api.py | claude -p "generate complete documentation of this API in markdown" > docs/api-docs.md

# Criar changelog a partir dos commits
$ git log --oneline -20 | claude -p "generate a formatted CHANGELOG" > CHANGELOG.md

# Append de review em relatorio existente
$ cat models.py | claude -p "security review" >> security-report.md

# Saida em JSON estruturado
$ cat app.py | claude -p "list all functions with their parameters" --output-format json > funcoes.json

# Gerar testes e salvar direto no arquivo de testes
$ cat utils.py | claude -p "generate unit tests with pytest for all functions" > tests/test_utils.py

# Combinar pipe de entrada + redirecionamento de saida
$ cat schema.sql | claude -p "generate the corresponding SQLAlchemy models" > models.py
Data/Research

📊Use Cases with Redirection

  • Automatic documentation:Generate READMEs, docstrings and wikis from code and save directly
  • Review reports:Append of reviews of multiple files in a single report
  • Code generation:Claude generates tests, migrations, or boilerplate and saves it in the right file
  • Structured data:Extract information from unstructured text and save to JSON or CSV
Practical Tip

💡Practical Tip

Use| tee arquivo.mdinstead of> arquivo.mdwhen you want to see the output on screen AND save to file at the same time. THEteeduplicates the stream: sends it to stdout (screen) and to the file simultaneously.

Example:cat app.py | claude -p "review" | tee review.md. You see the result scrolling on the screen and when it finishes, the file is saved there.

Do vs Avoid

✓ What TO DO

  • ✓ Use > to create new files, >> to append to existing ones
  • ✓ Use tee to view and save at the same time
  • ✓ Use --output-format json when you need to process the output

✗ What NOT to do

  • ✗ Use > on important files without backup (overwrites everything!)
  • ✗ Confuse > (overwrite) with >> (append) and lose content
  • ✗ Save Claude's output in production without reviewing first

🔧 Unix Integration - Claude in the Ecosystem

The Unix philosophy is clear:small, specialized tools that are assembled via pipe. The Claude Code with-pfits this model perfectly. It does not replace grep, awk, sed or jq. He composes himself with them. grep filters, awk formats, jq transforms JSON, and Claude interprets, analyzes and generates insights. Each tool does what it does best.

Main Concept

🎯Main Concept

Claude becomes another tool in the pipeline. The output of any Unix command can feed Claude, and the output of Claude can feed any other command. This composition is what makes the pipe so powerful. You are not limited to "question and answer". You are buildingintelligent processing pipelines.

  • • grep + Claude:grep filters the relevant lines, Claude interprets the meaning
  • • git diff + Claude:git shows what changed, Claude explains the impact
  • • jq + Claude:jq extracts fields from JSON, Claude analyzes the data
  • • find + xargs + Claude:find finds the files, xargs concatenates, Claude processes
Code Examples

⌨️Real Examples in the Terminal

# git diff + Claude para resumo de mudancas
$ git diff HEAD~5 | claude -p "summarize these changes in bullet points for the PR"

# find + xargs para buscar vulnerabilidades
$ find . -name "*.py" -not -path "./venv/*" | xargs cat | claude -p "find security vulnerabilities"

# jq + Claude para analisar dados JSON
$ cat response.json | jq '.data[] | {name, status}' | claude -p "which items have an abnormal status?"

# grep + Claude para entender padroes
$ grep -rn "TODO\|FIXME\|HACK" src/ | claude -p "prioritize these TODOs for urgency"

# awk + Claude para analise de performance
$ awk '{print $7, $10}' access.log | claude -p "which endpoints are slower?"

# curl + Claude para analisar resposta de API
$ curl -s https://api.example.com/health | claude -p "is everything healthy in this response?"

# ps + Claude para diagnostico de processo
$ ps aux --sort=-%mem | head -20 | claude -p "is any process consuming too much memory?"
Composition timeline

Anatomy of a Composite Pipeline

1

Data Source

cat, find, git, curl, docker, ps, etc.

The first command in the pipeline generates the raw data. It can be file content, API output, git diff, system logs, anything that produces text.

2

Filter/Transformation (optional)

grep, awk, sed, jq, sort, head, tail

Reduces data volume, extracts relevant fields, filters noise. Saves tokens and improves the quality of Claude's analysis. It's not mandatory, but it improves everything.

3

Intelligent Processing

claude -p "instruction"

Claude receives the filtered data via stdin, interprets it with the prompt instruction, and generates the output: analysis, summary, code, suggestions, diagnosis.

4

Departure/Destination (optional)

> file, >> append, | tee, | other-command

Claude's output can go to a file, be appended to a report, duplicated with tee, or even fed to another pipe. The cycle can continue.

Practical Tip

💡Practical Tip

The pipelinegit diff | claude -p "resuma estas mudancas"It is probably the most useful in a developer's day-to-day life. Run it before opening a PR to generate the description. Run it before merging to understand the impact. Run after a rebase to check that nothing broke. And a 10 second pipeline that saves 10 minutes.

🚀 Advanced Pipelines - Cutting-edge Automations

When you have mastered the basics of pipe, the next step iscomposite automations: loops that process multiple files, scripts that generate reports, integrations with cron for scheduled execution, and multi-stage pipelines where the output of one Claude feeds another. This is not theory. And how advanced operators use Claude Code in production.

Main Concept

🎯Main Concept

Advanced pipelines combineshell loops, multiple invocations of Claude e structured redirectto create automations that run without human intervention. The key is to understand that each invocation ofclaude -pIt is independent: it has no context from previous invocations, so you need to structure the pipeline so that each call is self-contained.

  • • Loop for:Iterate over files and process each one individually with Claude
  • • Shell scripts:Encapsulate complex pipelines into reusable scripts
  • • Cron:Schedule pipelines to run automatically (daily review, weekly audit)
  • • Multi-stage:The output of a Claude call feeds the next one for refinement
Code Examples

⌨️Real Scripts and Automations

# Loop: review de todos os arquivos Python do src/
$ for file in src/*.py; of
    echo "## Review: $file" >> review-report.md
    cat "$file" | claude -p "faca code review focused on security and performance" >> review-report.md
    echo "" >> review-report.md
  done

# Script de review automatico de PR
$ git diff main...HEAD | claude -p "review this PR: bugs, security, performance, suggestions" > pr-review.md

# Auditoria de seguranca do projeto inteiro
$ find . -name "*.py" -not -path "./venv/*" -exec grep -l "password\|secret\|key\|token" {} \; | \
    while read f; of
      echo "=== $f ===" >> security-audit.md
      cat "$f" | claude -p "does this file expose secrets? analyze line by line" >> security-audit.md
    done

# Documentacao automatica de API
$ for route_file in src/routes/*.py; of
    cat "$route_file" | claude -p "generate OpenAPI documentation for these routes" >> api-docs.yaml
  done

# Pipeline multi-estagio: analise + priorizacao
$ grep -rn "TODO\|FIXME" src/ | claude -p "list in JSON: {file, line, issue, severity}" | \
    claude -p "sort by severity and generate an action plan" > action-plan.md

# Cron: review diario dos commits do dia
# Add to crontab (crontab -e):
#0 18 * * 1-5 git log --since="8 hours ago" -p | claude -p "summary of the day" > ~/daily-summary.md
Box Data/Research

📊Use Cases in Production

  • CI/CD:Automatic PR review in the pipeline, blocks merge if vulnerabilities are found
  • Monitoring:Cron analyzes error logs every hour and generates intelligent alerts
  • Living documentation:Every time the code changes, the documentation is automatically regenerated
  • Security audit:Weekly scan of the entire codebase looking for insecure patterns
  • Onboarding:New dev? Run the script that generates an overview of the entire project automatically
Practical Tip

💡Practical Tip

Each invocation ofclaude -pand an independent API call. In loops with many files, you can spend a lot of tokens quickly. Monitor usage and consider grouping small files into a single call rather than one call per file.

For critical pipelines in production, always add error handling:claude -p "..." || echo "ERRO: Claude falhou para $file" >> errors.log. APIs can fail, and you need to know when this happens in an automated script.

Alert

⚠️Pay attention to automation costs

A loop that processes 100 files makes 100 API calls. If each call costs $0.05, that's $5 in a single script. With cron running daily, it's $150/month. Calculate the cost before automating. Consider only processing files that have changed (git diff --name-only) instead of the entire repository.

Do vs Avoid

✓ What TO DO

  • ✓ Add error handling (|| echo "ERROR") in scripts
  • ✓ Calculate estimated cost before running large loops
  • ✓ Process only changed files (git diff --name-only)

✗ What NOT to do

  • ✗ Run unlimited loops in cron (cost can explode)
  • ✗ Blindly trusting Claude's output in critical automations without validation
  • ✗ Create multi-stage pipelines without testing each stage in isolation first
