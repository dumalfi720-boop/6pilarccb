===== MODULE 2.1 =====
2.1 ~20 min

📄 What is CLAUDE.md

The project memory file that transforms Claude from a generic assistant into a partner who knows your stack, rules and conventions.

What is it:

CLAUDE.md is a Markdown file that lives in your repository and works as persistent memory for the project. Every time you start a Claude Code session, it reads this file first, gaining instant context about what the project is, how it works and what the rules are.

Why learn:

Without CLAUDE.md, you need to re-explain your project every session: which framework you use, how to run tests, which conventions to follow. With it, Claude already knows all this before you type the first word.

Key concepts:

Think of CLAUDE.md as an onboarding document for your AI code partner. Just like you onboard a new dev to the team, you onboard Claude with information about stack, structure and standards.

What is it:

CLAUDE.md solves the problem of amnesia between sessions. AI models have no memory between conversations - each session starts from scratch. CLAUDE.md automatically injects critical context, eliminating this gap.

Why learn:

The difference between a dev who uses Claude without CLAUDE.md and one who uses it with brutal e. The first spends 10-15 minutes per session explaining context. The second starts producing in 10 seconds.

Key concepts:

CLAUDE.md is not generic project documentation. And a document specifically written for Claude to understand. Focus on information it needs to generate correct code: conventions, commands, restrictions.

What is it:

When starting a session, Claude Code automatically searches and reads all relevant CLAUDE.md files: the one in the project's root directory, those in subdirectories where you are working, and the global one for your system.

Why learn:

Understanding when the file is read helps you write more effective content. CLAUDE.md is injected as system instructions, that is, it has a high priority in Claude's attention.

Key concepts:

Reading is automatic and silent. You don't need to ask. The content is treated as persistent instructions that survive even context compression with /compact.

What is it:

CLAUDE.md is not a README for humans, nor API documentation. And a file written specifically for Claude to interpret. The target audience is AI, not human developers.

Why learn:

Many people make the mistake of copying README.md to CLAUDE.md. This generates a file full of badges, links and irrelevant information for Claude, wasting precious tokens.

Key concepts:

README.md: presentation for humans. CLAUDE.md: instructions for the AI. CLAUDE.md must be concise, directive and focused on information that impacts code generation: stack, conventions, commands, restrictions.

What is it:

When you run /compact, the chat history is summarized. But CLAUDE.md instructions are reloaded in full, as they are treated as system instructions, not as history.

Why learn:

This means that your rules and conventions are never lost, even in very long sessions. And the guarantee of consistency: Claude may forget what you said in the chat, but he never forgets CLAUDE.md.

Key concepts:

Critical instructions must go on CLAUDE.md, not in the chat. If you say "never use any" in chat, this may be lost after compression. If it is in CLAUDE.md, it persists forever.

What is it:

With a well-written CLAUDE.md, Claude generates code that follows your standards automatically: correct imports, team naming conventions, preferred patterns, and avoids known project pitfalls.

Why learn:

The return on investment is absurd: 15 minutes writing a CLAUDE.md saves hours of corrections and re-explanations over weeks. And Claude Code's biggest productivity multiplier.

Key concepts:

Main benefits: consistent code from the first generation, fewer style reviews, instant onboarding in each session, rules that are never forgotten, and context sharing between team members.

===== MODULE 2.2 =====
2.2 ~25 min

🏗️ Anatomy of CLAUDE.md Perfect

The 6 essential sections that every CLAUDE.md must have: Project Overview, Tech Stack, Commands, Code Conventions, Things To Avoid and Additional Context.

What is it:

The first section of CLAUDE.md. In 2-3 lines, it says what the project does and what it was built with. Example: "SaaSify is a B2B subscription management platform. Built with Next.js 14, TypeScript and Supabase."

Why learn:

Claude needs high-level context to make correct architectural decisions. Knowing that it is a B2B SaaS vs a mobile app completely changes how it addresses authentication, permissions and structure issues.

Key concepts:

Keep it short and to the point. Project name, type (SaaS, API, CLI, etc.), and main technologies. Don't write a long paragraph - Claude needs facts, not narrative.

What is it:

Explicit list of all project technologies: language, framework, database, ORM, CSS, tests and package manager. Each item on a line with versions where relevant.

Why learn:

Without this section, Claude might suggest Express when you use Fastify, or npm when you use pnpm. The stack defines the vocabulary that Claude should use in his project.

Key concepts:

Be specific: don't say "React", say "Next.js 14 with App Router". Don't say "database", say "PostgreSQL via Supabase". The more accurate, the fewer corrections you will make.

What is it:

List of essential project commands: dev server, build, tests, lint, type check, migrations. Each command with its exact syntax. Example: "Dev: pnpm dev", "Test: pnpm test", "Lint: pnpm lint".

Why learn:

This is the most useful section of CLAUDE.md. When Claude needs to check that his code works, he uses these commands. Without them, it might try "npm test" when you use "uv run pytest".

Key concepts:

Include all the commands you use on a daily basis. Dev server, build, unit tests, e2e tests, lint, format, type check, database migrations. Claude uses exactly what you put here.

What is it:

Style rules and project standards: named exports vs default, file naming (kebab-case vs PascalCase), single vs double quotes, use of Server Components vs Client Components, validation with Zod, etc.

Why learn:

Without explicit conventions, Claude uses its default patterns that may conflict with those of your project. If you prefer named exports and Claude generates default exports, you need to correct it every time.

Key concepts:

Document what is unique about your project. Don't say "use TypeScript" (it already knows). Say "use Server Components by default, Client Components only with 'use client' when necessary". Focus on the specific.

What is it:

Explicit list of anti-patterns and prohibitions. Examples: "Never use any - use unknown and narrow", "Don't use Pages Router - only App Router", "Never commit .env files", "Don't install dependencies without discussing it first".

Why learn:

Preventing errors is as powerful as guiding behavior. A line "never use any" in CLAUDE.md prevents dozens of corrections throughout the project. Clear constraints generate more consistent code.

Key concepts:

Include mistakes that you have already made or that Claude has already made in your project. If he always suggests a library you don't want, add "Don't use X, prefer Y". Build this list iteratively.

What is it:

Section for information that does not fit into the previous ones: links to relevant docs, known quirks, deployment process, necessary environment variables, external dependencies.

Why learn:

Real projects have nuances that do not fit into fixed categories. A known bug in the ORM, an external API with rate limit, a deployment process that requires specific steps. This context avoids surprises.

Key concepts:

Use in moderation. CLAUDE.md is read every session and consumes tokens. Only put information that Claude really needs to generate better code. If it is information that only humans consult, leave it in the README.

===== MODULE 2.3 =====
2.3 ~20 min

📂 Scopes: Project, Subdirectory and Global

You can have multiple CLAUDE.md in different locations. Learn how the hierarchy works and when to use each scope.

What is it:

CLAUDE.md in the root (project-root/CLAUDE.md) is the main file. It applies to the entire project, regardless of which directory you are working in. And what most projects need.

Why learn:

This is the starting point. 90% of projects only need this single file. It defines stack, conventions and commands that apply to the entire codebase.

Key concepts:

Place it in the same folder as package.json, pyproject.toml or Cargo.toml. Claude detects it automatically. Include global project information: stack, commands, universal conventions.

What is it:

A CLAUDE.md within a subdirectory (e.g. src/CLAUDE.md or tests/CLAUDE.md) only applies when Claude is working in that directory. Allows specific rules per project area.

Why learn:

Monorepos and large projects have areas with different rules. The test directory may have different conventions than the production code. Subdirectories allow this granularity.

Key concepts:

Example: src/components/CLAUDE.md with "All components must use forwardRef and be exported with displayName". It only applies when Claude works in src/components/. The root CLAUDE.md is still valid as well.

What is it:

The file at ~/.claude/CLAUDE.md applies to ALL projects on your system. It's your global personal setup: language preferences, communication style, standards you use everywhere.

Why learn:

If you always prefer answers in Portuguese, or always use Conventional Commits, or always want type hints in Python, put them in the global instead of repeating them in each project.

Key concepts:

Best for: preferred language, commit style, editor preferences, personal defaults. Do not put specific project information here. The global is what is valid in any repository.

What is it:

Claude reads all relevant CLAUDE.md and combines their instructions. The order is: global first, then project root, then subdirectories. More specific instructions complement (not replace) more general instructions.

Why learn:

Understanding precedence avoids conflicts. If the global says "use double quotes" and the project says "use single quotes", Claude follows the more specific (project). Knowing this avoids confusion when rules seem to be ignored.

Key concepts:

All files are read and combined. There is no "substitution" - it is cumulative. If there is conflict, the most specific tends to prevail. Keep the scopes complementary, not conflicting.

What is it:

In large projects or monorepos, you can have a strategy with multiple CLAUDE.md: root with global conventions, packages/ with per-package rules, tests/ with specific test patterns.

Why learn:

Projects that grow need granular rules. The frontend may have different CSS conventions than the backend API rules. Multiple files allow this separation without polluting the root CLAUDE.md.

Key concepts:

Recommended strategy: CLAUDE.md root with stack and general commands. Subdirectories are where the rules really differ. Do not create CLAUDE.md in every directory - each file consumes tokens. Use with intention.

What is it:

Rule of thumb: Global for universal personal preferences. Root for everything project specific. Subdirectory only when an area has rules that significantly conflict or complement those of the root.

Why learn:

Many people make mistakes by exaggeration (CLAUDE.md in every directory) or by lack (no CLAUDE.md). The right balance is to have coverage without wasting tokens and without confusing rules.

Key concepts:

Just start with the root. Add global when you notice that you repeat instructions between projects. Add subdirectories when an area consistently needs different instructions. Evolve organically.

🚀 Real Example: Next.js SaaS App

A complete and commented CLAUDE.md for a SaaS application built with Next.js 14, TypeScript, Supabase and Tailwind CSS.

What is it:

Real example: "SaaSify is a B2B subscription management platform. Built with Next.js 14 (App Router), TypeScript and Supabase." Two lines that give Claude all the high-level context.

Why learn:

Upon learning that it is a B2B SaaS, Claude automatically thinks about multi-tenancy, permissions per company, subscription plans and billing. This context completely changes the architectural suggestions.

Key concepts:

Include the type of business (B2B, B2C, marketplace), the main proposal and the core stack. Don't detail features - Claude discovers this by reading the code. Focus on what he cannot infer on his own.

What is it:

Detailed Stack: "TypeScript (strict mode), Next.js 14 with App Router, Supabase (PostgreSQL), Tailwind CSS + shadcn/ui, Vitest + Playwright, pnpm". Each technology has a version and variant.

Why learn:

"strict mode" changes how Claude handles types. "App Router" vs "Pages Router" changes the entire file structure. "shadcn/ui" says that components already exist ready. Every detail matters.

Key concepts:

Specify the package manager (pnpm, not npm). Specify the test runner (Vitest, not Jest). Specify the CSS approach (Tailwind + shadcn, not styled-components). Claude adapts all generated code to these choices.

What is it:

Actual commands: "Dev: pnpm dev", "Build: pnpm build", "Test: pnpm test", "E2E: pnpm test:e2e", "Lint: pnpm lint". Exactly how you type in the terminal.

Why learn:

Claude uses these commands to verify that the code he generated works. If you say "pnpm test" and he tries "npm test", the command fails and he wastes time debugging a false problem.

Key concepts:

Include separate unit test AND e2e commands. Include separate lint and format. If you have a separate type check (npx tsc --noEmit), include it too. Each command you use on a daily basis should be listed.

What is it:

Real conventions: "Server Components by default, Client Components only when necessary", "Named exports for all components", "kebab-case for files, PascalCase for components", "Zod for validation".

Why learn:

These conventions eliminate 90% of style corrections. Without them, you will constantly say "no, use named export" or "no, this component must be a Server Component". CLAUDE.md automates this correction.

Key concepts:

"Place tests next to the code as *.test.ts" is a rule that changes where Claude creates test files. "Zod for input validation" makes it use Zod instead of manual validation. Each rule has a direct impact.

What is it:

Project restrictions: "Never use any - use unknown and narrow", "Don't use Pages Router - only App Router", "Never commit .env", "Don't install dependencies without discussing it first".

Why learn:

"Don't install dependencies without discussing it first" is powerful because Claude tends to suggest npm install to solve problems. This rule forces it to use what already exists before adding new packages.

Key concepts:

Build this list iteratively. Every time Claude does something you don't want, add it to CLAUDE.md. Over time, this section becomes your shield against recurring errors.

What is it:

After creating CLAUDE.md, test it by asking Claude to generate code and check if it follows the conventions. Ask for a component and see if it uses named export, Server Component by default and Zod.

Why learn:

An untested CLAUDE.md may have ambiguous or missing instructions. The test reveals gaps: "he used default export, I need to be clearer in the conventions section".

Key concepts:

Test with 3-4 different requests: create component, write test, create API route, refactor code. If in all cases it follows the conventions, your CLAUDE.md is working. Adjust iteratively.

===== MODULE 2.5 =====
2.5 ~20 min

🐍 Real Example: Python FastAPI Backend

A complete CLAUDE.md for a Python backend with FastAPI, SQLAlchemy, uv and async API best practices.

What is it:

Example: "FastAPI backend for a real-time analytics dashboard." A line that defines the domain (analytics), nature (real-time) and framework (FastAPI). Claude already knows that he will deal with WebSockets, queues and performance.

Why learn:

The domain context changes Claude's decisions. "Real-time analytics" implies optimized queries, aggressive caching and async endpoints. A "simple blog" would have a completely different approach.

Key concepts:

For backends, mention whether it is REST, GraphQL or gRPC. Mention whether you have async processing, queues or workers. Claude adapts patterns accordingly: async handlers, background tasks, connection pooling.

What is it:

Detailed Stack: "Python 3.12, FastAPI, PostgreSQL with SQLAlchemy, uv as package manager." Specifying uv (not pip) is essential because it changes all installation and execution commands.

Why learn:

Claude by default suggests pip and venv. If you use uv, it needs to know to generate "uv run pytest" instead of "python -m pytest". The difference between uv and pip changes how dependencies are managed.

Key concepts:

Specify the Python version (3.12 has different features than 3.9). Specify the ORM (SQLAlchemy vs Tortoise vs SQLModel). Specify the package manager (uv vs pip vs poetry). Each choice impacts the generated code.

What is it:

Commands with uv: "Dev: uv run uvicorn app.main:app --reload", "Test: uv run pytest", "Lint: uv run ruff check .", "Format: uv run ruff format .", "Migrate: uv run alembic upgrade head".

Why learn:

The "uv run" pattern before each command is what makes Claude run everything in the correct virtual environment. Without this, it may try to run pytest directly and fail due to lack of dependencies.

Key concepts:

Note that lint and format are separate (ruff check vs ruff format). Note that migrations use Alembic. Every detail you specify is one less mistake Claude will make.

What is it:

Conventions: "Type hints in all functions", "Pydantic models for request/response", "Dependency injection for database sessions", "Async handlers for all endpoints".

Why learn:

"Async handlers for all endpoints" forces Claude to use async def instead of def. "Pydantic models for request/response" forces automatic validation instead of loose dicts. Each rule makes the code more robust.

Key concepts:

Python conventions are different from JavaScript. Focus on: type hints (mandatory vs optional), async vs sync, ORM patterns (Unit of Work, Repository), validation (Pydantic vs manual), and tests (fixtures, factories).

What is it:

Python restrictions: "Never use raw SQL - use SQLAlchemy ORM", "Do not use DB sync calls in async handlers", "Never expose internal IDs - use UUIDs". Anti-patterns that destroy performance and security.

Why learn:

"Don't use DB sync calls in async handlers" is critical: a sync query blocks the entire FastAPI event loop, reducing performance. Claude needs to know this so as not to generate code that works but scales poorly.

Key concepts:

Common Python anti-patterns: raw SQL with string formatting (SQL injection), sync DB in async (blocking), expose sequential IDs (enumeration attack), import circular models. Each one can become a production incident.

What is it:

Comparison of the two approaches: Next.js focuses on Server/Client Components and UI patterns. FastAPI focuses on async handlers, dependency injection and ORM patterns. CLAUDE.md reflects these fundamental differences.

Why learn:

Comparing helps you abstract the principle: CLAUDE.md is not a fixed template. And a document that adapts to your stack. The sections are the same, but the content changes radically between frontend and backend.

Key concepts:

The principle is universal: tell Claude what you use (stack), how you use it (conventions), how to check (commands) and what to avoid (constraints). The specific content changes, but the structure is the same.

⚡ The /init Command

The shortcut to create a CLAUDE.md automatically. When to use it, what it generates, its limitations and how to combine it with manual editing.

What is it:

/init is a slash command that makes Claude explore its repository (folder structure, package.json/pyproject.toml, configs) and automatically generate a CLAUDE.md with overview, stack, commands and detected conventions.

Why learn:

It's the shortcut for those who don't want to write a CLAUDE.md from scratch. Instead of spending 15 minutes typing, you run /init and have a solid foundation in 30 seconds that just needs tweaking.

Key concepts:

Claude reads package.json, tsconfig.json, pyproject.toml, Cargo.toml and other configs. Detects language, framework, dependencies, build and test scripts. Generates a CLAUDE.md structured with this information.

What is it:

/init is ideal for: new projects that don't yet have CLAUDE.md, existing projects where you want a quick starting point, and when you are entering an unknown repo and want Claude to map the structure.

Why learn:

Knowing when to use it avoids waste. Running /init in a project that already has a mature CLAUDE.md can override fine-grained rules. Use at the right time: beginning of the project or first contact with a repo.

Key concepts:

Ideal moments: initial project setup, cloning an open source repo for the first time, migrating from another editor/AI to Claude Code. Do not use to update an existing CLAUDE.md - edit manually.

What is it:

/init generates a CLAUDE.md with the standard sections: Project Overview (based on repo), Tech Stack (detected from configs), Commands (extracted from scripts in package.json), Code Conventions (inferred from existing code).

Why learn:

Knowing what it generates helps you evaluate the quality of the output. If /init detected "npm" but you use "pnpm", you know exactly what to fix. If the “Things To Avoid” section was missing, you know you need to add it.

Key concepts:

Typical output covers: project name and description, language and framework, build/test/lint scripts, directory structure, and conventions detected in the code. Quality depends on how well structured the repo is.

What is it:

After running /init, you should always review and customize the generated CLAUDE.md. Add conventions that Claude did not detect, correct erroneous information, and add the "Things To Avoid" section that is not normally generated automatically.

Why learn:

/init generates a good starting point, but never a perfect CLAUDE.md. The most valuable rules are the ones you add manually: style preferences, anti-patterns, and architectural decisions that are not in the code.

Key concepts:

Post-/init checklist: check stack is correct, add "Things To Avoid", adjust commands (especially package manager), add naming and styling conventions, remove redundant information that consumes tokens.

What is it:

/init does not detect: implicit team conventions, undocumented architectural decisions, personal style preferences, project-specific anti-patterns, and business rules that impact the code.

Why learn:

Blindly trusting /init creates an incomplete CLAUDE.md. The most valuable information is precisely what it cannot infer: "never use any", "we prefer composition over inheritance", "deploy is done via Vercel CLI".

Key concepts:

Practical limitations: may confuse the package manager, does not consistently detect naming conventions, does not create a "Things To Avoid" section, may include redundant information, and quality varies depending on the repo structure.

What is it:

The recommended flow: run /init to get the base, review and correct the stack/commands, add your conventions manually, create the "Things To Avoid" section, test with real requests and iterate. The best of both worlds.

Why learn:

This flow combines speed (/init automation) with quality (manual refinement). You don't waste 15 minutes writing from scratch or end up with a generic CLAUDE.md. In 5 minutes you have a robust file.

Key concepts:

Pipeline: /init > review stack > correct commands > add conventions > add restrictions > test with 3 requests > adjust according to results. Repeat the "test and adjust" cycle in the first few sessions until CLAUDE.md is mature.
