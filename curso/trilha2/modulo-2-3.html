🎬 Scopes: Project, Subdirectory and Global

Understand the three levels of scope of CLAUDE.md and how to use them strategically. Learn when to use each and how they combine to form the complete context.

Topic 1: Project Root
1

📁 Project Scope (Root)

The most common scope andCLAUDE.md in the project root. This file contains general instructions that apply to the entire repository. It is the starting point for any Claude Code configuration.

💡Main Concept

The CLAUDE.md in the root is read whenever you start Claude Code anywhere within the project. It defines the base context that applies to all files and directories.

my-project/

├── CLAUDE.md ← Project Scope

├── src/

├── tests/

├── package.json

└──...

  • •Always loaded, regardless of current subdirectory
  • •Ideal for general conventions, tech stack and commands
  • •It must be versioned in Git along with the project

⚡Practical Tip

If you are only going to have one CLAUDE.md, put it in the root. It covers 90% of use cases. Only create other scopes when you have specific needs per project area.

Topic 2: Subdirectory
2

📂 Subdirectory Scope

You can place filesCLAUDE.md in subdirectoriesto add specific instructions for that area of ​​the project. This is useful in monorepos or projects with very different parts (e.g. frontend vs backend).

💡Main Concept

CLAUDE.md in subdirectories are loaded when Claude works with files in that directory. They complement (not replace) the root CLAUDE.md.

my-project/

├── CLAUDE.md ← General rules

├── frontend/

│ ├── CLAUDE.md ← Frontend rules

│ ├── src/

│ └── ...

├── backend/

│ ├── CLAUDE.md ← Backend rules

│ ├── src/

│ └── ...

  • •Complements CLAUDE.md from the root (does not replace)
  • •Ideal for monorepos with separate frontend/backend
  • •May have specific code conventions per area

⚡Practical Tip

In a monorepo, the frontend's CLAUDE.md might specify "use React Server Components" while the backend's says "use FastAPI with Pydantic v2". Each area has its rules.

Topic 3: Global
3

🌍 Global Scope

O CLAUDE.md globalstay in~/.claude/CLAUDE.mdand applies to ALL projects in which you use Claude Code. It is the place for personal preferences that apply to any project.

💡Main Concept

The global CLAUDE.md is personal and is not shared with the team. Use for preferences that are yours, regardless of the project.

~/.claude/

└── CLAUDE.md ← Global Scope

# Example content:

- Always respond in Brazilian Portuguese

- Use explanatory comments in the code

- Prefer functional approaches when possible

- Always include error handling

  • •Located in~/.claude/CLAUDE.md
  • •Applied to all projects automatically
  • •It is not versioned in Git (and personal)

⚡Practical Tip

Put in global CLAUDE.md: your preferred language for responses, your commit message style, and formatting preferences that you want in all projects.

Topic 4: Hierarchy
4

🏗️ Hierarchy and Combination

When there are multiple CLAUDE.md files, Claude Codecombines hierarchically. The global one is loaded first, then the one from the project root, and finally the one from the subdirectory. More specific instructions take priority over more general ones.

💡Main Concept

The loading and priority order follows the logic from the most general to the most specific:

# Load order:

1. ~/.claude/CLAUDE.md (Global - less specific)

2. /project/CLAUDE.md (Project)

3. /projeto/src/CLAUDE.md (Subdirectory - more specific)

# If there is a conflict, the most specific one wins

  • •All levels are combined, not replaced
  • •In case of conflict, the more specific one (subdirectory) prevails
  • •Claude receives all instructions as a single context

⚡Practical Tip

Avoid contradicting in the subdirectory what is in the root. If the root CLAUDE.md says "use Tailwind" and the subdirectory says "use CSS Modules", this could confuse Claude. Be intentional with exceptions.

Topic 5: Multiple Files
5

📑 Working with Multiple Files

On large projects, you may haveseveral CLAUDE.md scattered aroundthrough the repository. The key is to keep each one focused on their scope and avoid duplication of information between them.

💡Main Concept

The golden rule is: general information in the root, specific information in subdirectories. Never duplicate content.

monorepo/

├── CLAUDE.md # TypeScript, general conventions

├── apps/

│ ├── web/

│ │ └── CLAUDE.md # Next.js, React, Tailwind

│ └── mobile/

│ └── CLAUDE.md # React Native, Expo

├── packages/

│ └── shared/

│ └── CLAUDE.md # Rules for shared code

  • •Each CLAUDE.md must be self-contained in its scope
  • •Avoid duplicating information between levels
  • •Keep files short and to the point

⚡Practical Tip

Remember that each CLAUDE.md consumes context tokens. Keep them concise. If the total of all CLAUDE.md becomes too large, Claude may lose important information that has been "pushed" out of the context window.

Topic 6: When to Use Each
6

🎯 When to Use Each Scope

To knowWhen to use each scopeIt is essential to maintain the organization. Each level has its purpose and using the wrong scope can cause maintenance problems.

💡Main Concept

Use this quick guide to decide where to place each instruction:

  • •Global→ Personal preferences (language, response style, formatting)
  • •Project (Root)→ Tech stack, commands, general conventions, things to avoid
  • •Subdirectory→ Specific rules for an area (frontend, API, tests)

⚡Practical Tip

For most projects, a single CLAUDE.md in the root is sufficient. Only add subdirectories when you have areas with really different technologies or conventions. Start simple and expand as needed.
