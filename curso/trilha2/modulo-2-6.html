🎬 The /init Command

Learn how to use the /init command to automatically generate a CLAUDE.md from the analysis of your project. Understand what it generates, its limitations and how to combine it with manual editing.

Topic 1: What it does
1

🔍 What /init Does

The command/initand a Claude Code shortcut that automatically analyzes your project and generates an initial CLAUDE.md file. It examines the existing folder structure, configuration files, dependencies, and defaults to create a starting point.

💡Main Concept

/init works in three automatic steps:

  • •Analysis- Le package.json, pyproject.toml, Dockerfile, linting configs, etc.
  • •Inference- Identifies the stack, frameworks, code standards and structure
  • •Generation- Creates (or updates) CLAUDE.md with the detected information

⚡Practical Tip

To use, simply type/initat the Claude Code prompt. The command works in any directory and automatically detects the project type.

Topic 2: When to use
2

⏰ When to Use /init

/init is most useful inspecific situations. Knowing when to use it (and when not to use it) saves time and avoids unsatisfactory results.

💡Main Concept

The best scenarios to use /init:

  • •Existing project without CLAUDE.md- Quick starting point
  • •Onboarding in new project- Understand the stack quickly
  • •After big stack changes- Update existing CLAUDE.md

⚡Practical Tip

Use /init as a starting point, never as a final result. The generated CLAUDE.md is a draft that needs review and customization with the specific rules of your project.

Topic 3: What generates
3

📄 What /init Generates

/init generates a CLAUDE.md with the default sections, filled based on theautomatic project analysis. The result varies depending on the type of project and the information available in the configuration files.

💡Main Concept

/init typically outputs:

# Example of /init output in a Next.js project:

#CLAUDE.md

## Project Overview

Next.js application with TypeScript...

## Tech Stack

- Next.js 14, React 18, TypeScript

- Tailwind CSS, Prisma...

##Commands

- npm run dev, npm run build, npm test...

##CodeStyle

- ESLint config detected...

## Project Structure

- src/app/ - pages, src/components/ - components...

  • •Detect stack from package.json/pyproject.toml
  • •Extracts commands from scripts defined in the project
  • •Maps the main directory structure

⚡Practical Tip

Always review the generated CLAUDE.md. /init can detect the framework but it doesn't know whether you use App Router or Pages Router, whether you prefer Server Components, or what naming conventions the team adopts.

Topic 4: Customizing
4

✏️ Customizing the Result

After /init generates CLAUDE.md, the next step iscustomize and enrichthe content with information that only you (and your team) know about the project.

💡Main Concept

The customization process follows a checklist:

  • 1.Review the overview - add business context and scale
  • 2.Check versions in the tech stack - specify exact versions
  • 3.Add individual test command (almost never generated)
  • 4.Write team Code Conventions
  • 5.Add the Things To Avoid section (never generated by /init)
  • 6.Include additional context (migrations, sensitive areas, decisions)

⚡Practical Tip

The "Things To Avoid" section is almost never generated by /init, but it is one of the most valuable. Add it manually after /init with at least 5 rules of what Claude should not do in your project.

Topic 5: Limitations
5

⚠️ Limitations of /init

/init is a useful tool, but it hasclear limitationsthat you should know so as not to rely excessively on the automatically generated result.

💡Main Concept

Main limitations to consider:

  • •Does not know team conventions- It only detects what is in the configs, not verbal decisions
  • •Does not generate Things To Avoid- Don't know what errors you've already encountered
  • •May be too generic- Describes the framework, not YOUR use of it
  • •Does not detect architectural patterns- I don't know if you use DDD, Clean Architecture, etc.
  • •Can detect wrong- In complex monorepos, it can confuse the stacks

⚡Practical Tip

Never accept the output of /init without review. Treat it as a draft that needs at least 15 minutes of editing to become truly useful. The manual part is what makes the difference.

Topic 6: Combining with manual editing
6

🔄 Combining /init with Manual Editing

The best approach is to use /init asbootstrapand then refine manually. This combined flow creates a complete CLAUDE.md in minutes rather than hours of writing from scratch.

💡Main Concept

The recommended flow in 5 steps:

# Step 1: Generate the base

$ claude

> /init

# Step 2: Review the result

> Review CLAUDE.md and tell me what's missing

# Step 3: Add Things To Avoid

# (edit manually or ask Claude)

# Step 4: Add specific conventions

# (variable names, commit patterns, etc.)

# Step 5: Commit and share with the team

$ git add CLAUDE.md && git commit -m "add CLAUDE.md"

  • •Use /init for skeleton, add soul manually
  • •Ask Claude himself to suggest improvements to CLAUDE.md
  • •Update CLAUDE.md over time as the project evolves

⚡Practical Tip

After a few weeks using Claude Code with CLAUDE.md, run /init again and compare. Claude can detect new patterns that have emerged. Merge the new information with what you already had.
