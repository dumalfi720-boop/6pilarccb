Topic 1: Project Overview
1

🏗️ Project Overview

The first section of CLAUDE.md is theProject Overview- a concise description of what the project does, who it is for and what problem it solves. This section gives Claude the high-level context needed to make decisions aligned with the purpose of the project.

💡Main Concept

The Project Overview should be short (3-5 lines) but informative. Include the type of application, target audience and main objective.

# Project Overview

This is a task management SaaS for teams

remote. The app allows you to create projects, assign tasks and

track progress in real time via dashboard.

  • •Describe WHAT the project does in 1-2 sentences
  • •Mention the target audience or main users
  • •Include business context when relevant

⚡Practical Tip

Write the overview as if you were explaining the project to a senior dev on the first day of work. Be direct and avoid internal jargon that Claude would not understand.

Topic 2: Tech Stack
2

⚙️ Tech Stack

The sectionTech Stacklists all technologies, frameworks and tools used in the project. This is crucial so that Claude generates code that is compatible with the project's real dependencies, instead of suggesting alternatives that you don't use.

💡Main Concept

List the technologies in an organized way, separating frontend, backend, database and infrastructure tools. Include versions where relevant.

#TechStack

- Frontend: Next.js 14 (App Router), React 18, TypeScript

- Styling: Tailwind CSS v3

- Backend: Next.js API Routes + Server Actions

- Database: PostgreSQL via Prisma ORM

- Auth: NextAuth.js v5

- Deploy: Vercel

  • •Specify higher versions to avoid incompatible code
  • •Mention ORMs and authentication libraries explicitly
  • •Include package manager (npm, pnpm, yarn, uv)

⚡Practical Tip

If you use Next.js with App Router, specify this clearly. Claude can generate code for Pages Router if he doesn't know how. The same goes for React Server Components vs Client Components.

Topic 3: Commands
3

🖥️Commands

The sectionCommandsdocuments the project's essential commands: how to install dependencies, run the development server, run tests and build. This is one of the most useful sections because it allows Claude to execute commands correctly.

💡Main Concept

List each command with a brief description of what it does. Include commands to run a single test, as Claude often needs to validate his changes.

#Commands

- `npm install` - install dependencies

- `npm run dev` - development server

- `npm run build` - production build

- `npm test` - run all tests

- `npm test -- --testPathPattern="name"` - run a specific test

- `npm run lint` - check linting

  • •Include the command to run asingletest (very important!)
  • •Document environment variables required for each command
  • •Add database migration commands if applicable

⚡Practical Tip

The command to run a single test is the most important. Claude uses this to validate every change he makes. If this command is wrong, Claude will get stuck in error loops.

Topic 4: Code Conventions
4

📐 Code Conventions

The sectionCode Conventionsdefines the code standards that the project follows. This includes naming style, file organization, import standards, and any specific rules the team adopts.

💡Main Concept

Be specific in conventions. The clearer it is, the less Claude will deviate from his standards. Focus on conventions that are unique to your project.

#CodeConventions

- Use TypeScript strict mode in all new code

- React components: function declarations, not arrow functions

- File names: kebab-case (user-profile.tsx)

- Imports: absolutes with alias @ for src/

- Tests: colocate with the file (component.test.tsx)

- Error handling: always use try/catch in Server Actions

  • •Define naming standards (files, variables, components)
  • •Specify error handling standards
  • •Document the expected folder structure

⚡Practical Tip

Don't list obvious conventions (like "use camelCase in JavaScript"). Focus on decisions that are specific to your project and that Claude wouldn't be able to guess on his own.

Topic 5: Things To Avoid
5

🚫 Things To Avoid

The sectionThings To AvoidIt is as important as positive conventions. Here you clearly list what Claude should NOT do. This section prevents recurring errors and unwanted patterns.

💡Main Concept

Prohibitions are powerful because they eliminate entire categories of errors. Claude respects negative instructions very well when they are clear and direct.

# Things To Avoid

- NEVER use `any` in TypeScript - always define types

- DO NOT use inline CSS or styled-components (use Tailwind)

- DO NOT create files in the project root

- AVOID useEffect for data fetching (use Server Components)

- DOES NOT commit .env or secrets files

- NEVER change already applied migration files

  • •Use strong language: NEVER, NO, AVOID
  • •Explain the correct alternative when possible
  • •Add items as you discover recurring Claude errors

⚡Practical Tip

Every time you correct Claude by saying "don't do that", add that rule to the Things To Avoid section. Over time, this list becomes a powerful filter against errors.

Topic 6: Additional Context
6

📎Additional Context

The sectionAdditional ContextIt's space for any extra information that doesn't fit into the other sections. Here you can include architectural decisions, useful links, specific workflows or notes on sensitive areas of the code.

💡Main Concept

Use this section to communicate the “why” of decisions, not just the “what.” When Claude understands the reasons, he makes better decisions in new situations.

# Additional Context

- We are migrating from Pages Router to App Router

(new files must use App Router)

- The payment system uses Stripe and should not be changed

without explicit approval

- The lib/legacy.ts file is deprecated, do not add

new functions there

- We use feature flags via LaunchDarkly for gradual releases

  • •Document ongoing migrations and their status
  • •Mark critical areas that need extra care
  • •Include architectural decisions and their justifications

⚡Practical Tip

Keep CLAUDE.md concise and relevant. If an additional context section becomes too long, consider creating a separate CLAUDE.md in the relevant subdirectory.
